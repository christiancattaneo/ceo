Vector Databases for AI Agents: How Semantic Search Powers RAG and Memory

Most AI agent hallucinations aren't model failures — they're retrieval failures. The model is confidently summarizing context it was given. If the context is wrong, incomplete, or missing, the answer is wrong. Vector databases are the layer that determines what context the agent gets, which makes retrieval quality one of the highest-leverage engineering decisions in an agentic system.

What Vector Databases Actually Do

A vector database stores and searches data as high-dimensional numerical vectors — embeddings — rather than as text or structured fields. The key property is that semantically similar content has mathematically similar embeddings, which means you can search for "documents about recent software updates" and retrieve them even if they don't contain those exact words.

This is what makes vector search useful for AI agents: agents work with natural language queries, not SQL predicates. A user asking "what's our refund policy for enterprise customers" shouldn't require the agent to know the exact document title or the precise keywords — it should find the relevant policy document because the query and the document mean similar things.

The pipeline from raw content to retrievable vectors has four steps:

  1. Chunking — split source documents into segments of appropriate size
  2. Embedding — convert each chunk to a vector using an embedding model
  3. Indexing — store vectors in the database with metadata and a similarity index
  4. Retrieval — at query time, embed the query and find the most similar vectors using approximate nearest neighbor (ANN) search

Every step introduces quality tradeoffs. Retrieval quality at the end is bounded by the worst decision made in any earlier step.

Embeddings: The Foundation of Semantic Search

An embedding model maps text to a point in a high-dimensional space (typically 768–3,072 dimensions, depending on the model). The geometry of that space encodes semantic relationships: texts with similar meanings cluster nearby; texts about different topics are distant.

Embedding model choice matters more than most teams realize. Considerations:

Domain relevance

General-purpose embedding models (OpenAI text-embedding-3-large, Cohere embed-v3) perform well across diverse domains. For specialized domains — legal, medical, code — fine-tuned or domain-specific models can outperform general ones on retrieval tasks. If your agent primarily works with code, a code-specialized embedding model typically retrieves more relevant snippets than a general-purpose one.

Dimensionality and cost

Higher-dimensional embeddings capture more nuance but cost more to store and search. OpenAI's text-embedding-3-small uses 1,536 dimensions and costs ~10× less than text-embedding-3-large (3,072 dimensions). For most retrieval tasks, the smaller model performs within a few percentage points of the larger one on standard benchmarks — the right tradeoff depends on your accuracy requirements and scale.

Consistency between indexing and query

You must use the same embedding model to embed documents at indexing time and queries at retrieval time. Mixing models (embedding documents with model A, querying with model B) produces meaningless similarity scores. If you change embedding models, you must re-embed your entire corpus.

Chunking Strategy: Where Most RAG Systems Fail

Chunking is the most underappreciated variable in RAG quality. The chunk is the unit of retrieval — the agent gets back whichever chunks score highest on similarity to the query. Chunks that are too large return too much irrelevant context. Chunks that are too small lack the context needed to answer the question.

Fixed-size chunking

Split documents into chunks of N tokens with M tokens of overlap between adjacent chunks. Simple to implement; works reasonably well for homogeneous content. The overlap handles sentences that span chunk boundaries. Typical parameters: 512 tokens with 50–100 token overlap. The problem: fixed-size chunks cut through paragraphs, tables, and code blocks arbitrarily, sometimes retrieving half a thought.

Semantic chunking

Split on semantic boundaries — paragraph breaks, section headers, topic shifts — rather than fixed token counts. More expensive to implement but produces chunks that correspond to coherent units of meaning. For well-structured documents (documentation, legal contracts, research papers), semantic chunking retrieves more complete and coherent context.

Hierarchical chunking

Maintain two levels of chunks: large "parent" chunks (full sections) and small "child" chunks (paragraphs). Index the child chunks for high-precision retrieval; return the parent chunk as context. This gives the retrieval the precision of small chunks without losing the surrounding context. Often the highest-quality approach for document-heavy corpora, at the cost of increased complexity.

Document-type specific chunking

Different content types need different chunking strategies. Code should be chunked at function or class boundaries. Tables should be kept intact with their headers. Structured data (JSON, CSV) may not benefit from text chunking at all. A single chunking strategy applied uniformly to mixed content types will underperform a strategy that respects content structure.

The chunking diagnosis: If your RAG system returns relevant documents but the agent still produces wrong answers, the problem is often that the right chunk wasn't retrieved — a nearby chunk was. Inspect your retrieved chunks manually for a sample of failures. If you consistently see "almost right" retrievals, your chunk boundaries are in the wrong places.

Choosing a Vector Database

The vector database landscape has expanded rapidly. The right choice depends on your scale, infrastructure preferences, and query requirements:

Pinecone

Managed cloud

Fully managed, serverless option. Fast to get started, excellent performance at scale. No infrastructure to manage. Pricing by index size and query volume.

Weaviate

Open source / managed

Open source with managed cloud option. Supports hybrid search (vector + keyword) natively. Good for structured + unstructured mixed queries.

Qdrant

Open source / managed

Rust-based, high performance. Strong filtering on payload metadata. Self-hostable or managed. Good for workloads with complex metadata filters.

pgvector

PostgreSQL extension

Vector search in your existing Postgres database. No new infrastructure. Best for smaller corpora (<1M vectors) where SQL joins alongside vector search matter.

Chroma

Open source, local-first

Lightweight, easy to run locally. Good for development and small-scale production. Less operationally mature than dedicated vector DBs at large scale.

Milvus

Open source, high scale

Designed for billion-vector scale. More operational complexity than managed options. Right for very large corpora with strict latency requirements.

For most teams starting out: pgvector if you're already on Postgres and your corpus is under a few hundred thousand vectors; Pinecone or Qdrant Cloud if you want a managed service without Postgres constraints; Weaviate if you need hybrid search out of the box. Don't over-engineer the database choice early — chunking strategy and embedding model quality have a larger impact on retrieval quality than database selection within the same tier.

Retrieval Strategies Beyond Basic ANN Search

Basic nearest-neighbor search returns the top-K most similar vectors. For production RAG systems, several enhancements meaningfully improve relevance:

Hybrid search

Combine vector similarity search with traditional keyword (BM25/TF-IDF) search. Vector search excels at semantic matching; keyword search excels at exact matches for proper nouns, technical terms, and specific identifiers. Hybrid search with reciprocal rank fusion typically outperforms either alone on real-world corpora.

// Hybrid search combining vector + keyword results
vector_results = vector_search(query_embedding, top_k=20)
keyword_results = bm25_search(query_text, top_k=20)
final_results = reciprocal_rank_fusion(
  [vector_results, keyword_results],
  k=60  // RRF constant
)[:top_k]

Metadata filtering

Most vector databases support filtering on document metadata before or alongside vector search. If your agent knows the user is asking about a specific product version, filtering to only retrieve chunks tagged with that version dramatically improves precision. Always design your chunk metadata schema before indexing — retrofitting metadata after the fact usually requires re-indexing.

Re-ranking

Retrieve a larger candidate set (top-50 or top-100) with fast ANN search, then re-rank using a cross-encoder model that scores query-document pairs more accurately. Cross-encoders are much slower than ANN search but much more accurate — the two-stage pipeline gets the speed of ANN with the precision of cross-encoder ranking. Cohere Rerank and Jina Reranker are commonly used.

Query expansion and HyDE

Hypothetical Document Embeddings (HyDE): generate a hypothetical answer to the query, embed that answer, and use its embedding for retrieval. Counter-intuitive but effective — the hypothetical answer is closer in embedding space to relevant documents than the raw question is. Particularly useful for queries that are short and abstract.

Vector Databases and Content Discoverability

The connection to CEO (Claw Engine Optimization) is direct: AI agents that use RAG to answer questions about the web are running exactly this pipeline — chunking web pages, embedding them, and retrieving the most relevant chunks to answer user queries.

What determines whether your content gets retrieved? The same properties that make content useful to the retrieval pipeline: clear, self-contained sections that chunk cleanly; explicit headers that provide context for each chunk; specific, concrete language that embeds distinctly from generic content; structured metadata (JSON-LD schema) that enables metadata filtering.

Content that is organized for human reading — with each section making a complete, coherent point — tends to chunk well and retrieve well. Content that relies on surrounding context, uses highly abstract language, or mixes multiple unrelated topics in the same section tends to chunk poorly and retrieve inconsistently. The same editorial standards that make content readable for humans make it retrievable by AI agents.
