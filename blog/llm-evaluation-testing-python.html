LLM Evaluation and Testing in Python

A practical framework for measuring whether your Claude application actually works — golden datasets, LLM-as-judge scoring, semantic similarity metrics, faithfulness checks, and a pytest harness that catches regressions before they reach production.

Apr 14, 2026  ·  12 min read  ·  CEO Editorial

Unit tests verify that your code runs. They don't verify that your LLM application produces good outputs. Those are different problems — and most teams discover this difference in production, after a prompt change silently degraded answer quality across thousands of requests.

LLM evaluation fills this gap. It's not about perfection; it's about catching regressions and building confidence that changes improve rather than break things. Here's how to build an eval framework that's actually useful — not just theoretically sound.

1. The Eval Dataset

Everything starts with cases. An eval case pairs an input (prompt, context, or conversation) with an expected output — either a reference answer or a set of pass/fail criteria. Without cases, you're flying blind.

from dataclasses import dataclass, field
from typing import Any
import json, pathlib

@dataclass
class EvalCase:
    id: str
    input: str
    expected: str                    # reference answer (may be approximate)
    criteria: list[str]              # pass/fail rubric items
    tags: list[str] = field(default_factory=list)
    context: str = ""               # for RAG evals: retrieved docs
    min_score: float = 0.7          # threshold for this case to pass

@dataclass
class EvalDataset:
    name: str
    cases: list[EvalCase]
    version: str = "1.0"

    @classmethod
    def load(cls, path: str) -> "EvalDataset":
        data = json.loads(pathlib.Path(path).read_text())
        cases = [EvalCase(**c) for c in data["cases"]]
        return cls(name=data["name"], cases=cases, version=data.get("version", "1.0"))

    def filter_by_tag(self, tag: str) -> "EvalDataset":
        filtered = [c for c in self.cases if tag in c.tags]
        return EvalDataset(name=f"{self.name}[{tag}]", cases=filtered)

# evals/qa_dataset.json
EXAMPLE_DATASET = {
    "name": "qa_v1",
    "version": "1.0",
    "cases": [
        {
            "id": "qa_001",
            "input": "What is the capital of France?",
            "expected": "Paris",
            "criteria": [
                "Answer names Paris",
                "Answer is concise (under 50 words)"
            ],
            "tags": ["factual", "geography"],
            "min_score": 0.9
        }
    ]
}
How many cases do you need? 20–50 curated cases beats 500 auto-generated ones. Start with cases that represent your actual failure modes — pull from user feedback, edge cases found in prod, and known-hard inputs. Add cases when you fix a bug so the bug never silently returns.

2. LLM-as-Judge Scoring

For open-ended outputs, automated string matching fails. LLM-as-judge uses a second Claude call to evaluate the first one against a rubric. It's not perfect, but it correlates well with human judgment at scale.

from pydantic import BaseModel, Field
import anthropic

client = anthropic.Anthropic()

class CriterionResult(BaseModel):
    criterion: str
    passed: bool
    reasoning: str

class JudgeResult(BaseModel):
    score: float = Field(ge=0.0, le=1.0)
    criteria_results: list[CriterionResult]
    summary: str

def judge_response(
    question: str,
    expected: str,
    actual: str,
    criteria: list[str],
) -> JudgeResult:
    criteria_block = "\n".join(f"{i+1}. {c}" for i, c in enumerate(criteria))

    prompt = f"""Evaluate this AI response against the criteria below.

Question: {question}
Expected answer: {expected}
Actual response: {actual}

Criteria:
{criteria_block}

For each criterion, assess pass/fail with a one-sentence reasoning.
Then give an overall score from 0.0 to 1.0 (fraction of criteria passed, weighted by importance).
Output JSON matching this schema:
{{
  "score": float,
  "criteria_results": [{{"criterion": str, "passed": bool, "reasoning": str}}],
  "summary": str
}}"""

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="You are an objective evaluator. Be strict but fair. Output only valid JSON.",
        messages=[{"role": "user", "content": prompt}]
    )
    return JudgeResult.model_validate_json(response.content[0].text)
Judge model selection Use a model at least as capable as the one being judged — or one tier above. Judging Opus outputs with Haiku produces unreliable scores. Use Sonnet to judge Haiku; use Opus to judge Sonnet. The judge model's cost is amortized across your eval runs, not per request.

3. Semantic Similarity Scoring

For factual Q&A with deterministic expected answers, semantic similarity catches paraphrases that exact-match misses. Embed both expected and actual, then compute cosine similarity.

import numpy as np
import voyageai  # pip install voyageai

voyage = voyageai.Client()

def embed_texts(texts: list[str]) -> list[list[float]]:
    result = voyage.embed(texts, model="voyage-3", input_type="document")
    return result.embeddings

def cosine_similarity(a: list[float], b: list[float]) -> float:
    va, vb = np.array(a), np.array(b)
    return float(np.dot(va, vb) / (np.linalg.norm(va) * np.linalg.norm(vb)))

def semantic_score(expected: str, actual: str) -> float:
    """Returns 0.0–1.0; >0.85 is typically a good match."""
    embeddings = embed_texts([expected, actual])
    return cosine_similarity(embeddings[0], embeddings[1])

# For batch evals: embed once, compare many
def batch_semantic_scores(
    expected_list: list[str],
    actual_list: list[str],
) -> list[float]:
    all_texts = expected_list + actual_list
    embeddings = embed_texts(all_texts)
    n = len(expected_list)
    return [
        cosine_similarity(embeddings[i], embeddings[n + i])
        for i in range(n)
    ]

4. Faithfulness Check for RAG

For RAG pipelines, the critical failure mode is hallucination — claiming facts not in the retrieved context. A faithfulness scorer asks the judge: "Does every claim in the response appear in the context?"

class FaithfulnessResult(BaseModel):
    is_faithful: bool
    unsupported_claims: list[str]
    score: float  # fraction of claims that are supported

def check_faithfulness(
    context: str,
    response: str,
) -> FaithfulnessResult:
    prompt = f"""Given this context and AI response, identify any claims in the response
that are NOT supported by the context.

Context:
{context}

Response:
{response}

List each unsupported claim. If all claims are supported, return an empty list.
Output JSON: {{"is_faithful": bool, "unsupported_claims": [str], "score": float}}
score = (total_claims - unsupported_claims) / total_claims"""

    response_obj = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        system="You are a fact-checker. Be precise. Output only valid JSON.",
        messages=[{"role": "user", "content": prompt}]
    )
    return FaithfulnessResult.model_validate_json(response_obj.content[0].text)

5. pytest-Based Eval Harness

Wire everything into pytest so evals run in CI. Use pytest.mark.parametrize over the dataset and pytest.mark.eval to keep evals separate from unit tests — they're slower and call the real API.

# evals/test_qa_pipeline.py
import pytest
from your_app import answer_question  # the function under test
from evals.framework import EvalDataset, judge_response, semantic_score

dataset = EvalDataset.load("evals/qa_dataset.json")

@pytest.mark.eval
@pytest.mark.parametrize("case", dataset.cases, ids=[c.id for c in dataset.cases])
def test_qa_response_quality(case):
    actual = answer_question(case.input, context=case.context)

    # Fast check: semantic similarity against reference answer
    sim = semantic_score(case.expected, actual)

    # Slow check: LLM judge evaluates rubric criteria
    judge = judge_response(
        question=case.input,
        expected=case.expected,
        actual=actual,
        criteria=case.criteria,
    )

    # Combined score: weight similarity 30%, judge 70%
    combined = 0.3 * sim + 0.7 * judge.score

    assert combined >= case.min_score, (
        f"[{case.id}] Score {combined:.2f} < threshold {case.min_score}\n"
        f"Judge summary: {judge.summary}\n"
        f"Semantic similarity: {sim:.2f}\n"
        f"Failed criteria: {[r.criterion for r in judge.criteria_results if not r.passed]}"
    )

# Run: pytest -m eval -v --tb=short
# Skip in normal CI: pytest -m "not eval"

6. Regression Tracking

A single eval run tells you today's score. Regression tracking tells you whether it's getting better or worse. Store results in a JSON file per run and compare against baseline.

import json, time
from dataclasses import dataclass, asdict
from pathlib import Path

@dataclass
class EvalRunResult:
    run_id: str
    timestamp: float
    dataset: str
    model: str
    pass_rate: float
    mean_score: float
    case_results: list[dict]

def save_run(result: EvalRunResult, path: str = "evals/results/") -> None:
    p = Path(path)
    p.mkdir(parents=True, exist_ok=True)
    filename = p / f"{result.run_id}.json"
    filename.write_text(json.dumps(asdict(result), indent=2))

def load_baseline(dataset: str, path: str = "evals/results/") -> EvalRunResult | None:
    """Load the most recent run for this dataset as baseline."""
    results_dir = Path(path)
    runs = sorted(results_dir.glob("*.json"), key=lambda p: p.stat().st_mtime)
    for run_file in reversed(runs):
        data = json.loads(run_file.read_text())
        if data["dataset"] == dataset:
            return EvalRunResult(**data)
    return None

def check_regression(
    current: EvalRunResult,
    baseline: EvalRunResult,
    max_regression: float = 0.05,
) -> None:
    """Raise if mean_score dropped more than max_regression vs baseline."""
    delta = current.mean_score - baseline.mean_score
    if delta < -max_regression:
        raise AssertionError(
            f"Regression detected: score dropped {abs(delta):.3f} "
            f"({baseline.mean_score:.3f} → {current.mean_score:.3f}). "
            f"Max allowed regression: {max_regression}"
        )
Gate on regressions, not on absolute score Your baseline score will never be 1.0 — some cases are genuinely hard. What matters is that changes don't make things worse. Set max_regression=0.05 (5 percentage points) as your CI gate. Alert on drops; celebrate improvements; don't block on a fixed absolute threshold that was always aspirational.

Metric Selection Guide

Task type Primary metric When to also use
Factual Q&A Semantic similarity (>0.85) Exact match for short deterministic answers
Open-ended generation LLM-as-judge (rubric criteria) Human review for 10% sample
RAG / retrieval Faithfulness score + answer relevance Context precision/recall for retrieval quality
Classification / routing Exact match accuracy Confusion matrix for error pattern analysis
Code generation Test pass rate (execute the code) LLM judge for style + correctness explanation
Summarization LLM-as-judge (coverage + concision) ROUGE-L for recall of key facts

Production Checklist

  • Eval dataset is version-controlled alongside the code it tests
  • Cases added for every bug fixed in production — golden dataset grows over time
  • Judge model is one tier above or equal to the model being evaluated
  • Eval tests tagged with pytest.mark.eval — skipped in fast CI, run on merge to main
  • Results saved per run with timestamp and model version
  • Regression check gates on delta vs. baseline, not absolute score
  • Faithfulness check runs on all RAG pipeline evals — hallucination is the primary failure mode
  • Semantic similarity threshold calibrated per task — 0.85 for factual, 0.70 for creative
  • LLM judge criteria are specific and binary ("mentions X", "under N words") not vague ("is good")
  • Human review sample (10–20 cases) run before any major prompt change ships
