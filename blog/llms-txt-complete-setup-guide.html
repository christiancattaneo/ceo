llms.txt: The Robots.txt for AI — Complete Setup Guide

In 1994, robots.txt told web crawlers which pages to index. In 2026, llms.txt tells AI agents what your product is, what it does, and how to call it. Here's everything you need to know — and a template you can deploy today.

What Is llms.txt?

llms.txt is a plain-text (or Markdown) file served at the root of your domain — yourdomain.com/llms.txt — designed to give large language models and autonomous AI agents a structured overview of your product.

Think of it as a self-describing API manifest for the agentic web. When an AI agent (running in Claude, Cursor, OpenClaw, or any MCP-capable runtime) encounters your domain, it can fetch /llms.txt to instantly understand:

  • What your product/service does in plain English
  • What tools or capabilities it exposes
  • What endpoints or protocols it supports (MCP, A2A, REST)
  • What context the agent needs to use it effectively
  • What it should not try to do

Key distinction: robots.txt is about crawl access. llms.txt is about capability discovery. A human reads your landing page to understand what you do. An AI agent reads your llms.txt.

robots.txt vs. llms.txt — Side by Side

robots.txt llms.txt
Purpose Tell crawlers what to index Tell agents what you can do
Consumer Google, Bing, web spiders LLMs, AI agents, MCP runtimes
Format Key-value directives Markdown (structured prose)
Content Allow/disallow URL paths Product description, tools, endpoints, usage rules
Standard RFC 9309 (formal) Emerging (de facto, not yet RFC)
Required? Strongly recommended Not yet — but adoption is accelerating

Why It Matters Right Now

The agentic web is still young, but the pattern is clear: AI agents will increasingly be the first consumer of your product's capabilities. Not a human clicking around your marketing site — an agent calling your MCP server, reading your schema, deciding whether to route a task to you.

If you don't have an llms.txt, agents have to infer your capabilities from scraped HTML. That's slow, unreliable, and often wrong. A well-written llms.txt is the difference between an agent confidently calling your API and an agent ignoring you entirely.

Early movers are already seeing results. OpenClaw's Claw Engine Registry (part of the CEO framework) prioritizes products with valid llms.txt files in its agent routing decisions. As more runtimes adopt similar patterns, the file becomes table stakes.

The llms.txt Format

There's no finalized RFC yet, but a clear community standard has emerged — pioneered by Answer.AI and adopted by a growing ecosystem. The format is Markdown with a few conventional sections: 

# Product Name

> One-sentence description of what this product is and does.

## What I Do
Plain English description of your product's purpose, target user,
and core value proposition. 2-4 sentences.

## Core Capabilities
- Capability 1: brief description
- Capability 2: brief description
- Capability 3: brief description

## Tools / MCP Endpoints
- `tool_name` — what it does, what arguments it takes
- `tool_name_2` — what it does

## Protocols
- MCP: [endpoint URL]
- A2A: [endpoint URL if applicable]
- REST: [base URL] — link to OpenAPI spec

## Pricing & Access
Free tier: [what's free]
Paid tier: [what's paid, price if public]
Auth: [API key / OAuth / none]

## Usage Guidelines
- What agents SHOULD do
- What agents SHOULD NOT do
- Rate limits, quotas, usage policies

## Quick Start
Minimal example an agent can follow to make its first successful call.

## Links
- [Full docs](https://yourdomain.com/docs)
- [OpenAPI spec](https://yourdomain.com/openapi.json)
- [GitHub](https://github.com/yourorg/yourrepo)

A Real-World Example: merlyn-mcp

Here's what a well-formed llms.txt looks like for a real product — merlyn-mcp, a local AI development coach: 

# merlyn-mcp

> Local development coach with MCP server, interactive TUI, and AI-powered
> analysis. Tracks project phase (PLAN/BUILD/SHIP/GROW) and tells you what
> to do next.

## Core capabilities

- Phase tracking: PLAN → BUILD → SHIP → GROW state machine
- 37 MCP tools, 20 prompts, 5 resources — all local
- Interactive TUI coach (`npx merlyn-mcp`)
- AI-powered project analysis via Anthropic API (optional)
- 12-ingredient production readiness audit

## Quick start

```
npx merlyn-mcp       # launch TUI coach
npx merlyn-mcp server # start MCP server for Cursor
```

## Links
- [Full docs](https://merlyn.tech/docs)
- [npm](https://www.npmjs.com/package/merlyn-mcp)
- [GitHub](https://github.com/christiancattaneo/merlyn-mcp)

Notice what it does well: unambiguous description in the first line, concrete capability list, exact commands an agent can copy-paste, and links to deeper sources. An LLM reading this in 2 seconds knows exactly what to do with it.

Step-by-Step: Writing Your llms.txt

Step 1 — Write the headline

The first line (after the # heading) is the most important. It's the one-sentence answer to "what does this product do?" Write it for an agent that has zero prior context. No buzzwords. No "AI-powered platform." Just the function.

Bad: "The next-generation developer productivity suite."
Good: "MCP server that analyzes your codebase, detects your development phase, and generates the next coding prompt."

Step 2 — List capabilities, not features

Features are what you built. Capabilities are what an agent can do with you. "37 MCP tools" is a feature. "Analyze a codebase and return a phase classification with confidence score" is a capability. Lead with capabilities.

Step 3 — Document your tools precisely

If you have MCP tools, list each one with its name, what it does, and what arguments it accepts. An agent uses this to decide whether to call you — and how: 

## Tools

- `midas_analyze(path: string)` — Analyzes codebase at path, returns phase
  classification (PLAN|BUILD|SHIP|GROW) + confidence + next_action

- `midas_verify(path: string)` — Runs build/test/lint gates, returns pass/fail
  per gate + overall verdict

- `midas_vuln_scan(path: string)` — Security audit: checks for hardcoded secrets,
  CORS misconfig, missing input validation. Returns findings[]

Step 4 — State what agents should NOT do

This is underrated. Tell agents your limits. Rate limits, what data you don't store, what calls will fail. Agents that respect your constraints are more likely to route tasks to you successfully: 

## Constraints
- Do not call analyze() on paths outside the user's project root
- No authentication required for local MCP server
- Rate limit: 60 tool calls/minute
- All analysis is local — no data leaves the machine

Step 5 — Add a quick start

One working example. An agent should be able to copy-paste this and get a successful result on the first try. If it can't, rewrite the quick start.

Step 6 — Deploy it

Serve it as a plain text file at /llms.txt. Set the content type to text/plain. No authentication — it should be publicly readable. If you want Markdown rendering for humans, also serve it at /llms as HTML.

Bonus: Some products serve both /llms.txt (minimal, agent-optimized) and /llms-full.txt (comprehensive, with full tool docs, examples, and edge cases). Start with the minimal version and expand as you learn what agents actually need.

Common Mistakes

  • Marketing copy: "Industry-leading AI-powered developer tools" tells an agent nothing. Be literal.
  • Missing tool signatures: Listing tool names without arguments forces agents to guess. Don't make them guess.
  • No quick start: An agent that can't make a successful first call in <60 seconds will route to a competitor.
  • Stale docs: An outdated llms.txt is worse than none — it causes hallucinations. Treat it like your API changelog.
  • Requiring auth just to read it: llms.txt should be public. Auth goes in the capabilities doc, not in the discovery file.

The llms.txt Ecosystem in 2026

Adoption is accelerating fast. As of early 2026, notable products with public llms.txt files include Anthropic's documentation, several major MCP server registries, and a growing list of developer tools in the OpenClaw ecosystem. The pattern is spreading to enterprise APIs — particularly those targeting agentic workflows.

The OpenClaw Claw Engine Registry (part of the CEO framework) indexes llms.txt files from registered products and uses them for agent routing decisions. Products without a valid llms.txt are harder to route to — they require the agent to do more inference work, which increases latency and error rates.

In the same way that a missing robots.txt doesn't break Google indexing but a well-crafted one improves it, a missing llms.txt doesn't prevent AI agents from using your product — but a well-crafted one dramatically increases the probability they will.

The 5-Minute Checklist

  • File is at /llms.txt, publicly accessible, no auth
  • First heading describes your product in one literal sentence
  • Capabilities are listed as what agents can DO, not feature names
  • Every MCP/API tool has name + what it does + key arguments
  • At least one copy-pasteable quick start example
  • Constraints and rate limits are documented
  • Links to full docs, OpenAPI spec, and GitHub
  • File is versioned alongside your codebase (not an afterthought)
