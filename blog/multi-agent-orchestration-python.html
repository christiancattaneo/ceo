Multi-Agent Orchestration Patterns in Python

Practical patterns for coordinating fleets of Claude agents — orchestrator/worker decomposition, fan-out/gather, routing by capability, stateful handoffs, and failure recovery that actually holds up in production.

Apr 14, 2026  ·  13 min read  ·  CEO Editorial

A single Claude call handles a lot. But some tasks are inherently parallel, exceed a single context window, or need specialized handling that one generalist prompt can't deliver reliably. That's where multi-agent systems earn their keep.

The flip side: multi-agent architectures introduce coordination overhead, partial failure modes, and state management problems that don't exist in single-agent flows. The patterns below address each of these without adding unnecessary complexity.

1. The Orchestrator / Worker Pattern

The most common structure: one orchestrator agent that plans and delegates, plus N worker agents that execute specific subtasks. The orchestrator never does domain work — its only job is decomposition and synthesis.

import asyncio
from dataclasses import dataclass, field
from typing import Any
import anthropic

client = anthropic.Anthropic()

@dataclass
class SubTask:
    id: str
    description: str
    worker_role: str
    result: str | None = None

async def orchestrate(goal: str) -> str:
    # Step 1: orchestrator decomposes the goal into subtasks
    plan_response = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1024,
        system="""You are an orchestrator. Given a goal, output a JSON array of subtasks.
Each subtask: {"id": "t1", "description": "...", "worker_role": "researcher|writer|analyst"}
Output only valid JSON.""",
        messages=[{"role": "user", "content": goal}]
    )

    import json
    subtasks_raw = json.loads(plan_response.content[0].text)
    subtasks = [SubTask(**t) for t in subtasks_raw]

    # Step 2: run workers in parallel
    await asyncio.gather(*[run_worker(task) for task in subtasks])

    # Step 3: orchestrator synthesizes results
    results_block = "\n\n".join(
        f"[{t.id}] {t.worker_role}: {t.result}"
        for t in subtasks
    )
    synthesis = client.messages.create(
        model="claude-opus-4-6",
        max_tokens=2048,
        system="Synthesize the subtask results into a coherent final answer.",
        messages=[{
            "role": "user",
            "content": f"Goal: {goal}\n\nSubtask results:\n{results_block}"
        }]
    )
    return synthesis.content[0].text

async def run_worker(task: SubTask) -> None:
    role_prompts = {
        "researcher": "You are a thorough researcher. Gather facts and cite key points.",
        "writer": "You are a precise technical writer. Produce clear, structured prose.",
        "analyst": "You are a data analyst. Identify patterns and draw conclusions.",
    }
    system = role_prompts.get(task.worker_role, "Complete the task thoroughly.")

    response = client.messages.create(
        model="claude-sonnet-4-6",  # workers use cheaper model
        max_tokens=1024,
        system=system,
        messages=[{"role": "user", "content": task.description}]
    )
    task.result = response.content[0].text
Model tier strategy Use Opus for the orchestrator (planning and synthesis require deeper reasoning) and Sonnet for workers (faster, cheaper, sufficient for bounded tasks). This split typically cuts cost by 60–70% versus running everything on Opus.

2. Fan-Out / Gather with asyncio.TaskGroup

When tasks are fully independent and you need structured cancellation on any failure, asyncio.TaskGroup (Python 3.11+) beats gather(). If one worker raises, all siblings are cancelled automatically.

from typing import TypeVar
import asyncio

T = TypeVar("T")

async def fan_out(items: list[str], process_fn) -> list[Any]:
    """Run process_fn over all items in parallel; cancel all on first failure."""
    results: list[Any] = [None] * len(items)

    async with asyncio.TaskGroup() as tg:
        for i, item in enumerate(items):
            async def _task(idx=i, val=item):
                results[idx] = await process_fn(val)
            tg.create_task(_task())

    return results

# Usage: analyze 5 documents in parallel
async def analyze_document(doc: str) -> dict:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        system="Extract: {topics: [], sentiment: '', key_claim: ''} as JSON.",
        messages=[{"role": "user", "content": doc}]
    )
    import json
    return json.loads(response.content[0].text)

async def analyze_corpus(docs: list[str]) -> list[dict]:
    return await fan_out(docs, analyze_document)
TaskGroup vs gather asyncio.gather(return_exceptions=True) lets all tasks finish even when some fail. TaskGroup cancels siblings on first exception. Use TaskGroup when partial results are useless (batch analysis), gather when you want best-effort (parallel enrichment where some can fail gracefully).

3. Routing Agent — Dynamic Dispatch

A routing agent classifies incoming requests and dispatches them to the right specialist. The router itself is a Claude call that returns a routing decision — simpler and more flexible than hard-coded rules.

from pydantic import BaseModel
from enum import Enum
import json

class AgentType(str, Enum):
    CODE_REVIEW = "code_review"
    DATA_ANALYSIS = "data_analysis"
    DOCUMENT_QA = "document_qa"
    GENERAL = "general"

class RoutingDecision(BaseModel):
    agent: AgentType
    confidence: float
    reason: str

AGENT_SYSTEMS = {
    AgentType.CODE_REVIEW: """You are a senior engineer. Review code for correctness,
security, and style. Be specific about line numbers and suggest fixes.""",

    AgentType.DATA_ANALYSIS: """You are a data analyst. Identify patterns, outliers,
and actionable insights. Use precise numerical language.""",

    AgentType.DOCUMENT_QA: """You are a document analyst. Answer questions strictly
from the provided context. Quote relevant passages.""",

    AgentType.GENERAL: """You are a helpful assistant. Answer clearly and concisely.""",
}

def route_request(user_input: str) -> AgentType:
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",  # Haiku is fast + cheap for routing
        max_tokens=128,
        system="""Classify the request. Output JSON: {"agent": "code_review|data_analysis|document_qa|general", "confidence": 0.0-1.0, "reason": "one line"}.
Output only JSON.""",
        messages=[{"role": "user", "content": user_input}]
    )
    decision = RoutingDecision.model_validate_json(response.content[0].text)
    return decision.agent

def dispatch(user_input: str, context: str = "") -> str:
    agent_type = route_request(user_input)
    system = AGENT_SYSTEMS[agent_type]

    content = user_input
    if context:
        content = f"Context:\n{context}\n\nQuestion: {user_input}"

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=2048,
        system=system,
        messages=[{"role": "user", "content": content}]
    )
    return response.content[0].text

4. Stateful Handoffs — Shared AgentContext

When agents need to hand off work mid-stream — passing partial results and accumulated context — you need a shared state object that travels with the task. Avoid global state; pass context explicitly.

from dataclasses import dataclass, field
from typing import Any
import time, uuid

@dataclass
class AgentContext:
    task_id: str = field(default_factory=lambda: str(uuid.uuid4())[:8])
    original_goal: str = ""
    history: list[dict] = field(default_factory=list)
    artifacts: dict[str, Any] = field(default_factory=dict)
    current_agent: str = ""
    created_at: float = field(default_factory=time.time)

    def add_message(self, role: str, content: str) -> None:
        self.history.append({
            "role": role,
            "content": content,
            "agent": self.current_agent,
            "ts": time.time()
        })

    def handoff(self, next_agent: str, note: str = "") -> None:
        if note:
            self.artifacts[f"handoff_{self.current_agent}_to_{next_agent}"] = note
        self.current_agent = next_agent

def research_agent(ctx: AgentContext) -> AgentContext:
    ctx.current_agent = "researcher"
    ctx.add_message("user", ctx.original_goal)

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="Research the topic thoroughly. Gather key facts and data points.",
        messages=[{"role": "user", "content": ctx.original_goal}]
    )
    research = response.content[0].text
    ctx.add_message("assistant", research)
    ctx.artifacts["research"] = research
    ctx.handoff("writer", note="Research complete, ready for drafting.")
    return ctx

def writer_agent(ctx: AgentContext) -> AgentContext:
    ctx.current_agent = "writer"
    research = ctx.artifacts.get("research", "")

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=2048,
        system="Turn research into a well-structured draft. Use headers and clear paragraphs.",
        messages=[{
            "role": "user",
            "content": f"Research:\n{research}\n\nWrite a draft about: {ctx.original_goal}"
        }]
    )
    draft = response.content[0].text
    ctx.artifacts["draft"] = draft
    ctx.handoff("reviewer", note="Draft complete, needs review.")
    return ctx

def run_pipeline(goal: str) -> str:
    ctx = AgentContext(original_goal=goal)
    ctx = research_agent(ctx)
    ctx = writer_agent(ctx)
    return ctx.artifacts.get("draft", "")
Why explicit context over global state Passing AgentContext through each function makes every agent's inputs and outputs explicit. You can serialize the context to Redis for resumability, replay it in tests, and audit the full decision chain per task — none of which is possible with shared globals.

5. Retry and Partial Failure Recovery

In a fan-out of 20 worker calls, 1–2 will fail. Stopping the whole pipeline for a transient API error is wrong. The right pattern: retry workers independently with exponential backoff, then decide at the orchestrator level whether partial results are acceptable.

import asyncio, logging
from anthropic import APIStatusError, APITimeoutError

logger = logging.getLogger(__name__)

async def worker_with_retry(
    task: str,
    *,
    max_attempts: int = 3,
    base_delay: float = 1.0,
) -> str | None:
    for attempt in range(max_attempts):
        try:
            response = client.messages.create(
                model="claude-sonnet-4-6",
                max_tokens=512,
                messages=[{"role": "user", "content": task}]
            )
            return response.content[0].text

        except (APIStatusError, APITimeoutError) as e:
            if attempt == max_attempts - 1:
                logger.error(f"Worker failed after {max_attempts} attempts: {e}")
                return None
            delay = base_delay * (2 ** attempt)
            logger.warning(f"Attempt {attempt + 1} failed, retrying in {delay:.1f}s")
            await asyncio.sleep(delay)

async def tolerant_fan_out(
    tasks: list[str],
    *,
    min_success_ratio: float = 0.8,
) -> list[str]:
    """Run all tasks; raise if fewer than min_success_ratio complete."""
    results = await asyncio.gather(
        *[worker_with_retry(t) for t in tasks],
        return_exceptions=True
    )

    successes = [r for r in results if isinstance(r, str)]
    ratio = len(successes) / len(tasks)

    if ratio < min_success_ratio:
        failed = len(tasks) - len(successes)
        raise RuntimeError(
            f"Too many worker failures: {failed}/{len(tasks)} failed (threshold: {min_success_ratio:.0%})"
        )

    return successes

6. Agentic Loop — Self-Directed Tool Use

Some tasks require an agent to decide dynamically which tools to call and in what order. Claude's native tool use supports this via a while loop that continues until the model returns a non-tool-use stop reason.

from anthropic.types import ToolUseBlock, TextBlock

TOOLS = [
    {
        "name": "search_web",
        "description": "Search the web for current information.",
        "input_schema": {
            "type": "object",
            "properties": {"query": {"type": "string"}},
            "required": ["query"]
        }
    },
    {
        "name": "run_python",
        "description": "Execute Python code and return stdout.",
        "input_schema": {
            "type": "object",
            "properties": {"code": {"type": "string"}},
            "required": ["code"]
        }
    }
]

def execute_tool(name: str, inputs: dict) -> str:
    # Replace with real implementations
    if name == "search_web":
        return f"[search results for: {inputs['query']}]"
    if name == "run_python":
        import io, contextlib
        buf = io.StringIO()
        with contextlib.redirect_stdout(buf):
            exec(inputs["code"], {})  # sandbox in production!
        return buf.getvalue()
    return "unknown tool"

def agentic_loop(goal: str, max_turns: int = 10) -> str:
    messages = [{"role": "user", "content": goal}]

    for _ in range(max_turns):
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=4096,
            tools=TOOLS,
            messages=messages
        )

        if response.stop_reason == "end_turn":
            return next(
                (b.text for b in response.content if isinstance(b, TextBlock)),
                ""
            )

        # Append assistant's tool_use blocks
        messages.append({"role": "assistant", "content": response.content})

        # Execute each tool and collect results
        tool_results = []
        for block in response.content:
            if isinstance(block, ToolUseBlock):
                result = execute_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result
                })

        messages.append({"role": "user", "content": tool_results})

    raise RuntimeError(f"Agent did not complete within {max_turns} turns")
Always set max_turns An agentic loop without a turn limit is a liability. Set max_turns conservatively (8–12 for most tasks) and log when it fires. If you're hitting the limit regularly, the task is too large for one agent — break it into subtasks handled by the orchestrator/worker pattern instead.

Pattern Comparison

Pattern When to use Complexity Cost profile
Orchestrator / Worker Task decomposes into independent subtasks with a final synthesis step Medium 1 Opus + N Sonnet calls
Fan-Out / Gather Same operation over many items in parallel (batch analysis, enrichment) Low N Sonnet calls
Routing Agent Request type is unknown at design time; dispatch to specialists Low 1 Haiku + 1 Sonnet/Opus
Stateful Handoff Sequential pipeline where each stage depends on prior output Medium N sequential Sonnet calls
Agentic Loop Task requires dynamic tool selection across an unknown number of steps High Variable; set turn limit

Production Checklist

  • Orchestrator uses Opus; workers use Sonnet or Haiku depending on task difficulty
  • All fan-outs have a min_success_ratio threshold — decide upfront what partial completion means
  • Every agentic loop has a max_turns guard and logs when it fires
  • State is passed explicitly as a dataclass or dict — no module-level globals
  • Worker failures retry with exponential backoff before propagating
  • Routing decisions are logged for audit — store agent_type and confidence with the request
  • Use asyncio.TaskGroup when partial results are worthless; gather(return_exceptions=True) when best-effort is fine
  • Context objects are serializable — enables resumability and replay in tests
  • Token budgets per worker are explicit — max_tokens set conservatively per role
  • Integration tests run the full pipeline end-to-end against the real API, not mocks
Start simple Most production use cases start with fan-out/gather or a single stateful handoff pipeline. Reach for the full orchestrator/worker pattern only when task decomposition is genuinely dynamic. Premature orchestration adds latency and cost without adding reliability.
