Context Window Management for Production AI Agents

Context limits don't announce themselves gracefully. An agent that has been working reliably for 15 turns suddenly truncates earlier context, loses track of earlier instructions, or hits a hard limit mid-task. Here's how to design for context from the start — and what to do when you didn't.

The Problem With "Just Use a Bigger Context Window"

When 128K-token and 200K-token context windows became available, a lot of teams concluded that context management was a solved problem. Just stuff everything in and let the model sort it out. That conclusion turned out to be premature for two reasons.

First, long contexts are expensive. A 200K-token context costs roughly 40× more per call than a 5K-token context. For an agent that makes 20 tool calls in a session, each carrying the full conversation history, your cost-per-task calculation changes dramatically. At scale, what seemed like an acceptable per-call cost becomes unsustainable.

Second, model attention degrades over very long contexts. Research has consistently shown the "lost in the middle" effect: models recall information near the beginning and end of a context well, but attend poorly to content buried in the middle. A 200K-token context doesn't give you 200K tokens of equal retrieval quality — it gives you a gradient where the most important information can get lost if it's not near the edges.

Context management, then, is not just a cost optimization. It's a quality concern.

Anatomy of an Agent's Context Budget

Before you can manage a context, you need to understand what's consuming it. A typical agent turn draws from several distinct sources:

System prompt

The instructions that define the agent's persona, capabilities, constraints, and tool schemas. System prompts for capable agents tend to grow over time as you add guardrails, examples, and edge case handling. It's common to see system prompts balloon to 4K–10K tokens. Review them periodically — many contain deprecated instructions, redundant examples, or verbose descriptions that could be tightened without affecting behavior.

Tool definitions

Every tool you register with the model consumes tokens, whether the tool is used that turn or not. A set of 15 tools with descriptive schemas can consume 3K–6K tokens per turn. For agents that have access to many tools, consider dynamic tool loading — injecting only the tools relevant to the current task rather than the full registry.

Conversation history

The accumulating turns of the conversation. This is the primary growth vector for long-running agents. A conversation that starts at 2K tokens can reach 50K after 30 turns if history is naive — appending every message in full without any compression.

Retrieved context

Documents, search results, or other content retrieved from external sources mid-task. RAG-retrieved chunks that aren't carefully sized can consume large blocks of context even when most of the retrieved content isn't relevant to the current step.

Tool call results

The responses from tool executions. A web search that returns three full-page documents, a database query that returns 500 rows, a code execution result with verbose stdout — these can consume context rapidly if you pass results through unfiltered.

Instrument before optimizing. Add token counting to every agent invocation. Log the breakdown: system prompt tokens, tool schema tokens, history tokens, retrieved context tokens, tool result tokens. You cannot optimize what you haven't measured, and the distribution is almost always surprising — usually a different component than you assumed is the primary driver.

Context Management Strategies

There's no universal answer — the right strategy depends on your agent's task type, session length, and quality requirements. Most production agents use a combination.

Strategy Best for Trade-off
Sliding window Conversational agents, customer support Simple to implement; loses early context permanently
Summarization Long research or analysis tasks Preserves semantic content; adds latency + cost for summarization call
Hierarchical memory Multi-session agents, personal assistants Highest fidelity; complex to implement correctly
Dynamic tool loading Agents with large tool registries Reduces per-turn cost; requires tool routing logic
Result compression Agents that call data-heavy tools Targeted; only helps with tool result bloat

Sliding Window

The simplest approach: keep only the N most recent turns in context. When the conversation exceeds the window size, drop the oldest turns. Implementation is trivial — slice the message array before constructing the API call.

The weakness is obvious: if the user referenced something from 30 turns ago, the agent has no access to it. For task-focused agents where each turn is relatively self-contained, this is acceptable. For agents that accumulate context across a session — building on earlier research, tracking decisions made earlier — it breaks down.

def build_context(messages: list[dict], max_turns: int = 20) -> list[dict]:
    """Keep system message + last N turns."""
    system = [m for m in messages if m["role"] == "system"]
    conversation = [m for m in messages if m["role"] != "system"]

    # Keep last max_turns messages (preserves most recent context)
    trimmed = conversation[-max_turns:] if len(conversation) > max_turns else conversation
    return system + trimmed

Progressive Summarization

When the context approaches a threshold, summarize the oldest portion of the conversation into a compact representation, then replace those turns with the summary. The agent continues with the summary in context rather than the full history.

This preserves more semantic content than a sliding window, but introduces a new risk: summarization compresses information, and compressed information loses nuance. Critical details — a specific number mentioned in turn 3, an explicit constraint the user stated early in the conversation — can be lost in a summary that captures the gist but not the specifics.

Mitigate this with structured summarization prompts that explicitly ask the model to preserve key facts, decisions, and constraints, not just themes:

SUMMARIZE_PROMPT = """
Summarize this conversation segment. Preserve:
- All specific numbers, dates, and identifiers mentioned
- Explicit constraints or requirements stated by the user
- Decisions that were made and their rationale
- Current state of any in-progress tasks

Format as a structured summary under these headers.
Keep it concise — the goal is dense information, not narrative.

Conversation:
{conversation_segment}
"""

Hierarchical Memory

The most sophisticated approach, and the one used by the best long-running agents. Maintain memory at multiple levels of granularity simultaneously:

  • Working memory — the current context window, with full fidelity for recent turns
  • Episodic memory — summaries of earlier turns in the current session, retrievable when relevant
  • Semantic memory — durable facts extracted from conversations (user preferences, confirmed information, established context) stored in a vector database and retrieved as needed

When constructing a context window, the agent retrieves relevant episodic and semantic memories based on the current query and injects them alongside the recent working memory. This gives the agent access to a much larger effective context than the window allows, at the cost of retrieval latency and the imprecision introduced by embedding-based retrieval.

Hierarchical memory is the right long-term architecture for persistent agents, but it's a significant engineering investment. Sliding window or summarization is almost always the right starting point — implement it, instrument it, and only graduate to hierarchical memory when you can demonstrate that the simpler approaches are failing for your specific use case.

Dynamic Tool Loading

Instead of registering all tools in every system prompt, maintain a tool registry and inject only the tools relevant to the current task. This requires a routing step — a lightweight classifier or another model call that maps the current user message to the relevant tool subset — but it can reduce tool schema token consumption by 60–80% for agents with large registries.

def get_relevant_tools(user_message: str, all_tools: dict) -> list:
    """Use a fast classifier to select relevant tools."""
    tool_names = list(all_tools.keys())

    # Fast heuristic: keyword matching for common cases
    if any(w in user_message.lower() for w in ["search", "find", "look up"]):
        priority = ["search_web", "search_docs", "search_database"]
    elif any(w in user_message.lower() for w in ["email", "send", "message"]):
        priority = ["send_email", "draft_message"]
    else:
        priority = tool_names  # fall back to all tools

    # Return schemas for priority tools only
    return [all_tools[t] for t in priority if t in all_tools]

Tool Result Compression

Raw tool results are often far larger than what the agent actually needs. A web search returning three full articles when only a paragraph is relevant, a database query returning a full result set when only the count matters, a file read returning a 500-line file when only the function signature is needed.

Add a compression layer between tool execution and context injection. For each tool type, define a maximum token budget for results and a compression strategy for when results exceed it:

def compress_tool_result(tool_name: str, result: str, max_tokens: int = 2000) -> str:
    token_count = count_tokens(result)

    if token_count <= max_tokens:
        return result  # fits, pass through

    if tool_name == "search_web":
        # Extract just titles, URLs, and first sentence of each result
        return extract_search_snippets(result, max_tokens)
    elif tool_name == "read_file":
        # Return first N lines with a note that content was truncated
        return truncate_with_notice(result, max_tokens)
    else:
        # Generic: ask a fast model to summarize to budget
        return summarize_to_budget(result, max_tokens)

Handling the Context Limit Gracefully

Despite your best management, agents occasionally approach or hit context limits mid-task. The difference between a graceful degradation and a hard failure is whether you designed for this case.

Monitor context usage proactively

Calculate the token budget consumed before each API call. If you're within 20% of the limit, trigger compression proactively — don't wait for the API to return a context length error.

MAX_CONTEXT = 128_000
COMPRESSION_THRESHOLD = 0.80  # compress at 80% full

def should_compress(messages: list[dict]) -> bool:
    current_tokens = count_tokens(messages)
    return current_tokens > (MAX_CONTEXT * COMPRESSION_THRESHOLD)

Checkpoint long tasks

For multi-step tasks that will definitely span many turns, design the agent to checkpoint its state at defined intervals. A checkpoint captures: what has been accomplished, what remains, and any key facts accumulated so far. If the context needs to be reset, the agent can resume from the last checkpoint rather than starting over.

Graceful degradation over hard failure

When compression is unavoidable and you might lose important context, tell the user. "I've been working on this for a while and need to summarize our earlier conversation to continue. Let me know if I've missed anything important." This is far better than silently losing context and producing wrong answers.

Cost Modeling

Context management is ultimately a cost-quality trade-off. The math is worth doing explicitly for your use case. Suppose your agent makes 20 API calls per session with an average context size of 50K tokens (input) and 500 tokens (output), using a model at $3/M input + $15/M output:

  • Input cost per session: 20 × 50K × ($3 / 1M) = $3.00
  • Output cost per session: 20 × 500 × ($15 / 1M) = $0.15
  • Total: ~$3.15 per session

Implement summarization that reduces average context to 15K tokens:

  • Input cost per session: 20 × 15K × ($3 / 1M) = $0.90
  • Plus summarization calls: 5 × 5K × ($3 / 1M) = $0.075
  • Total: ~$0.975 per session — a 69% reduction

The quality cost of that compression depends on your task. For many conversational agents, a 70% cost reduction with no perceptible quality loss is available — it just requires instrumentation and intentional design to find it.

Prompt caching changes the calculus. Anthropic and OpenAI both offer prompt caching that reduces the cost of repeated context prefixes by 80–90%. If your system prompt and tool schemas are stable across turns, caching makes large contexts more affordable. Design your context so cacheable content (system prompt, tools) stays at the front and changes (conversation history) append at the end.
