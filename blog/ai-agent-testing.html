AI Agent Testing and Evaluation: How to Validate Agents Before Production

Traditional unit tests break when the function under test is a language model. Here's a practical framework — from tool-level unit tests to behavioral evaluation suites to CI/CD integration — for building confidence in agents before they touch real users.

Why Testing Agents Is Fundamentally Different

When you test a function that parses a date string, you have a ground truth. Given input "2026-04-11", the correct output is a Date object representing April 11th. Deterministic. Binary. Easy to assert.

LLM-based agents don't work that way. The same prompt might produce slightly different phrasing each run. "Correct" often means something about intent, not exact string equality. An agent that calls search_web("best restaurant NYC") instead of search_web("top-rated restaurants New York City") might still accomplish the task perfectly — or might not, depending on the retrieval system.

The implication: you can't just assert output === expected. You need a different vocabulary for what "passing" means, and a different infrastructure to enforce it.

The trap teams fall into: They skip evaluation entirely ("we'll know when it's wrong") or use simple string matching ("if 'I cannot help' appears, flag it"). Both approaches give you false confidence. The first misses regressions. The second produces flaky tests that break on trivially different phrasing.

The Three Layers of Agent Testing

Think of agent testing as a pyramid. Each layer builds on the one below it and is progressively more expensive to run.

Layer 1 — Unit Tests for Deterministic Components

The non-LLM parts of your agent are fully testable with conventional unit tests. Don't skip these because you're distracted by the harder parts.

Deterministic components in a typical agent include:

  • Tool implementations — the Python/JS functions that execute when the model calls a tool. get_weather("Austin, TX") should return a specific shape. Mock the HTTP call and assert the shape.
  • Tool schema validation — if your model calls a tool with invalid parameters, does the error handling work? Can the agent recover?
  • Retrieval pipelines — if your agent uses RAG, test that the retrieval returns the right chunks for known queries against a fixed corpus. Embed once, serialize, run assertions on retrieved doc IDs.
  • Context assembly logic — if you build the system prompt programmatically from user data, test that assembly function directly. It's just string construction.
  • Output parsers — if your agent emits structured JSON and you parse it, test the parser against known outputs including malformed cases.
# Example: unit test for a tool implementation
def test_search_tool_returns_correct_shape():
    results = search_web_tool("climate change 2026", max_results=3)
    assert isinstance(results, list)
    assert len(results) <= 3
    for r in results:
        assert "title" in r
        assert "url" in r
        assert "snippet" in r

def test_search_tool_handles_empty_query():
    with pytest.raises(ValueError, match="query cannot be empty"):
        search_web_tool("", max_results=5)

These tests are fast, cheap, and should run on every commit. Get them green before worrying about the harder layers.

Layer 2 — Behavioral Tests with Golden Datasets

Behavioral tests check that your agent does the right thing, not that it produces the exact same output. You define a set of input scenarios with expected behaviors, run the agent against them, and evaluate whether the behavior was correct.

Building a golden dataset is the most important investment you'll make in agent quality. Start with 20-50 cases that cover:

  • Happy path — typical inputs that should work
  • Edge cases — ambiguous queries, missing information, conflicting constraints
  • Adversarial inputs — things that should fail gracefully (rude requests, off-topic questions, prompt injection attempts)
  • Regression cases — specific failures you've seen in production, captured as test cases

Each entry in the dataset includes an input, the expected behavior (not the exact output), and optionally a rubric the evaluator should use.

[
  {
    "id": "booking-001",
    "input": "I want to book a flight from Austin to New York next Friday",
    "expected_behavior": "Agent asks clarifying questions about departure time, class preference, or passenger count before searching",
    "tags": ["booking", "clarification"],
    "should_use_tools": ["search_flights"],
    "should_not_say": ["I cannot help", "as an AI"]
  },
  {
    "id": "booking-002",
    "input": "Book me a flight",
    "expected_behavior": "Agent asks for origin, destination, and date — does not attempt to search without this information",
    "tags": ["booking", "incomplete-input"]
  }
]

Keep the dataset in version control. The golden dataset is code. It documents your expected agent behavior as precisely as any spec. When behavior intentionally changes, update the dataset in the same PR. When a production incident surfaces a new failure mode, add a test case before the fix.

Layer 3 — End-to-End Scenario Tests

E2E tests run complete multi-turn conversations and evaluate outcomes at the task level. They're the most expensive layer — each run costs real API tokens and takes real time — but they're the only way to catch emergent failures that only appear across turn sequences.

An E2E test for a research agent might run a 6-turn conversation simulating a user drilling into a topic, then check: Did the agent maintain coherent context across turns? Did it cite sources? Did it avoid hallucinating facts that contradict documents it retrieved?

Keep your E2E suite small (10-20 scenarios), run it on a schedule rather than every commit, and invest in good logging so failures are diagnosable.

Evaluation Methods: How to Score Non-Deterministic Outputs

Once you have test cases, you need evaluation methods — ways of turning "agent output" into a pass/fail or score. There's a spectrum from cheap-and-brittle to expensive-and-robust.

Method When to use Cost Reliability
Exact match Structured outputs, IDs, enum values Free Brittle for prose
Contains / regex Required phrases, forbidden phrases, format checks Free Moderate — misses semantic equivalents
Semantic similarity Comparing meaning of short answers against reference Low (embedding call) Good for factual answers, weaker for instructions
LLM-as-judge Behavioral rubrics, open-ended quality, nuanced criteria Medium (~$0.001–0.01/eval) High when prompt is well-designed
Human eval High-stakes releases, new capability validation High Ground truth

Designing Good LLM-as-Judge Prompts

LLM-as-judge is the workhorse of modern agent evaluation. The idea: use a separate model call to score your agent's output against a rubric. It scales better than human review and handles nuance better than string matching.

The quality of your evaluator depends entirely on how well you specify the rubric. Vague criteria like "was the response helpful?" produce noisy, inconsistent scores. Specific, anchored criteria produce reliable ones.

EVAL_PROMPT = """
You are evaluating an AI assistant's response to a user message.

User message: {user_message}
Assistant response: {assistant_response}

Evaluate the response on the following criteria. For each criterion,
output a score of 0, 1, or 2 and a one-sentence justification.

Criteria:
1. TASK_ADDRESSED (0=ignored the task, 1=partially addressed, 2=fully addressed)
2. FACTUAL_ACCURACY (0=contains clear errors, 1=uncertain/unverifiable, 2=accurate)
3. APPROPRIATE_SCOPE (0=out of scope or refuses unnecessarily, 1=borderline, 2=appropriate)
4. NO_HARMFUL_CONTENT (0=contains harmful content, 2=clean)

Return JSON: {{"scores": {{"TASK_ADDRESSED": N, "FACTUAL_ACCURACY": N, ...}},
               "justifications": {{"TASK_ADDRESSED": "...", ...}},
               "overall_pass": true/false}}

overall_pass is true only if all scores are 2 except FACTUAL_ACCURACY
which may be 1 if the topic is inherently uncertain.
"""

Calibration tip: Before trusting your LLM judge, calibrate it against 20-30 human-labeled examples. If the judge's pass/fail agrees with humans less than 85% of the time, rework the rubric before relying on it for CI.

Measuring What Matters: Key Evaluation Metrics

Beyond individual test cases, track these metrics across your evaluation suite over time:

Task Completion Rate

What percentage of end-to-end scenarios did the agent complete successfully? This is your headline number. Define "completed" per scenario type — for a booking agent, it might mean "reached a confirmed reservation state." Track it weekly, alarm if it drops more than 5 points.

Tool Call Accuracy

When the agent calls a tool, does it call the right one with the right parameters? You can assert on tool call logs deterministically: given scenario X, the agent must call search_flights before book_flight, and the destination parameter must match what the user said.

Hallucination Rate

For agents that cite or retrieve information, track what percentage of factual claims in responses are supported by retrieved documents. LLM-as-judge works well here: "Given these retrieved documents and this response, which factual claims in the response are not supported by the documents?"

Refusal Rate (with segmentation)

Refusals aren't inherently bad — an agent should refuse clearly harmful requests. The metric to watch is the rate of inappropriate refusals: cases where the user's request was legitimate but the agent declined anyway. Segment refusals by scenario type to find the policy miscalibrations.

Cost per Successful Task

Track total token spend divided by successful task completions. Evaluation surfaces the expensive paths — agents that spin in loops, contexts that balloon unnecessarily, tool calls that should be cached. Cost per task often reveals architectural problems that pure quality metrics miss.

Integrating Evaluation into CI/CD

Running evaluations manually doesn't scale. The goal is a pipeline where every significant change to your agent — prompt edits, new tools, model version bumps — triggers an automatic evaluation run before merging.

A practical CI integration runs in three stages:

Stage 1 — Fast gate (every PR, every commit)

Run unit tests and a small behavioral slice — 10-15 cases from your golden dataset, selected to cover the critical paths. Should complete in under 2 minutes. Fails the PR if any deterministic assertion breaks or if task completion rate drops below threshold.

# .github/workflows/agent-eval.yml (partial)
jobs:
  fast-eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: pip install -r requirements.txt
      - run: pytest tests/unit/ -x -q
      - run: python eval/run_eval.py
            --dataset eval/golden_critical.json
            --model $AGENT_MODEL
            --threshold-pass-rate 0.85
            --output eval-results.json
      - uses: actions/upload-artifact@v4
        with:
          name: eval-results
          path: eval-results.json

Stage 2 — Full suite (nightly or pre-release)

Run the complete golden dataset plus E2E scenarios. Catches regressions across the full distribution of inputs, not just critical paths. Results feed a dashboard — track pass rate and per-category scores over time to spot slow degradations that don't cross any single threshold.

Stage 3 — Shadow evaluation (production)

Log a sample of production conversations (with appropriate privacy handling) and run them through your judge asynchronously. This is your ground truth signal. When production quality diverges from eval suite quality, it tells you your golden dataset is missing coverage for real-world inputs.

The shadow eval feedback loop: When production surfaces a new failure type, add 3-5 examples to the golden dataset immediately. Don't wait for a postmortem. The dataset grows toward the actual distribution of inputs your agent faces, which is the whole point.

Common Testing Anti-Patterns

Testing the prompt, not the behavior

If your eval checks that the system prompt contains certain phrases, you're testing authoring, not execution. The model might follow the prompt perfectly and still fail the task. Always test outputs, not inputs.

Eval suite that only covers happy paths

If every case in your dataset is a well-formed request that the agent handles correctly, your suite will always pass — and tell you nothing. Deliberately include edge cases, adversarial inputs, and known failure modes. A 95% pass rate on a challenging dataset is more meaningful than 100% on a trivial one.

Judge prompts that are easier than the agent task

If your judge is a smaller/cheaper model than your agent, and the evaluation task requires the same reasoning as the agent task, the judge will make errors that corrupt your metrics. Use a model at least as capable as your agent for quality evaluation. Use the faster/cheaper model only for format and constraint checks.

No version tracking for the eval suite

If you update the golden dataset without recording when and why cases changed, you can't compare eval results across time. Tag each case with a version, record the version used for each eval run, and diff results against the same dataset version — not whatever's currently checked in.

Starting From Scratch: A Practical Sequence

If you have a working agent with no test coverage, here's the sequence that gives you the fastest return on investment:

  1. Write unit tests for all tools this week. Mock external calls. Assert on output shapes and error handling. This immediately prevents a class of regressions that are otherwise invisible.
  2. Capture 20 real conversations from production or manual testing. Label them: did the agent succeed or fail? This becomes the seed of your golden dataset.
  3. Write a basic LLM-as-judge evaluator against those 20 cases. Calibrate against your labels until agreement exceeds 85%.
  4. Add the fast eval slice to CI. Even 10 cases with a 0.8 pass threshold will catch the regressions that matter most.
  5. Grow the dataset over time, especially after production incidents. By the time you have 100 cases, you'll have a reliable signal for every release.

The goal isn't perfection, it's signal. An imperfect eval suite that runs automatically is worth more than a perfect one that requires manual effort to run. Start with something that catches 60% of regressions automatically. Improve from there.
