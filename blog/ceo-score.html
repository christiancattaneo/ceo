The CEO Score: How to Measure Your Site's Agent-Readiness

You can't optimize what you can't measure. PageRank gave SEO a number. Core Web Vitals gave performance a number. The CEO Score gives agent-readiness a number — 100 points across five pillars that determine whether AI agents discover, understand, trust, and transact with your product. Here's the complete framework.

Why a Score, Not a Checklist

Checklists tell you what to do. Scores tell you where you stand, how you compare, and what to prioritize next. That distinction matters because agent optimization isn't binary — you don't flip a switch and become "agent-ready." You move along a spectrum, and different positions on that spectrum produce different outcomes.

Consider what happened with PageRank. Before PageRank, webmasters had checklists: add meta keywords, submit to directories, use descriptive titles. Those were useful. But PageRank changed the game by creating a single number that captured the aggregate effect of hundreds of signals. It let you benchmark, track progress, and prioritize. It turned SEO from a craft into a discipline.

Core Web Vitals did the same thing for performance. Before CWV, developers knew they should "make things fast." After CWV, they had LCP, FID, and CLS — three numbers they could measure, monitor, and optimize with precision. Google even tied them to search ranking, making the numbers impossible to ignore.

The CEO Score applies the same logic to agent optimization. It distills the question "how agent-ready is my site?" into a single 0–100 number, decomposed into five pillars of 20 points each. You can measure it today, benchmark against competitors, track it over time, and know exactly which pillar is dragging your score down.

The CEO Score is not a vanity metric. Each point maps to a specific, measurable improvement in how AI agents interact with your product. A site at 35 is functionally invisible to agents. A site at 85 gets preferred routing. The difference is quantifiable — and so is the path between them.

The 5 Pillars

The CEO Score is built on five pillars, each worth 20 points. The pillars are ordered by the agent interaction lifecycle: an agent must first discover you, then understand your schema, then call your endpoints, then parse your content, then trust you enough to transact. Each pillar has specific, auditable criteria with defined point values.

Pillar 1: Discovery (20 points)

Discovery is whether agents can find you at all. This is the foundation — zero discovery means zero everything else. Discovery covers the artifacts and protocols that make your existence known to AI agents probing your domain or querying capability registries.

Check Points What agents look for
/llms.txt present and valid 6 Product description, capabilities, API base URL, protocol links — all in one fetch
sitemap.xml includes capability pages 3 Not just blog posts — pages describing what your product can do
robots.txt allows agent user-agents 3 Not blocking AI crawlers (ChatGPT-User, Anthropic, PerplexityBot, etc.)
Canonical URLs are consistent 3 No duplicate content, no conflicting canonical tags, one URL per resource
Listed in at least one agent registry 5 Capability registries, MCP server directories, A2A agent indexes

The single highest-impact item here is llms.txt. A well-structured llms.txt file gives an agent complete orientation in a single HTTP request — product identity, capabilities, API location, and protocol entry points. It's the equivalent of a business card that also happens to contain your full resume.

Pillar 2: Schema (20 points)

Schema is whether agents can understand what you do in machine-readable terms. Discovery tells agents you exist; schema tells them what you're capable of — with typed inputs, outputs, and constraints that require zero human interpretation.

Check Points What agents look for
JSON-LD structured data on product pages 4 SoftwareApplication, WebAPI, or Organization schema with capability metadata
Capability schema with typed I/O 6 Each capability has input/output JSON schemas — not just text descriptions
OpenAPI spec hosted and complete 6 Every endpoint documented with summaries, parameter descriptions, response schemas
Constraints declared (rate limits, latency, file sizes) 4 Machine-readable operational bounds so agents can make routing decisions

The capability schema and OpenAPI spec are the heavy hitters. An agent evaluating whether to route a task to your product needs typed schemas to make that decision programmatically. A natural-language description of "we parse invoices" is useful for language models; a JSON schema with input types, output types, and constraints is useful for agents that need to actually call you.

Pillar 3: Endpoints (20 points)

Endpoints measure whether agents can actually interact with your product — not just read about it, but call it. This pillar covers the protocols and interfaces that turn your product from a description into a callable service.

Check Points What agents look for
Machine-readable API (REST or GraphQL) 5 Programmatic access — not just a web UI
MCP server or /mcp route 6 The Model Context Protocol is how agents discover and call tools natively
Health endpoint (/health or /status) 3 Agents check availability before routing — no health endpoint means no trust
Agent-compatible auth (bearer tokens, API keys) 6 Not session cookies, not OAuth redirects — agents need programmatic auth

MCP support is increasingly the dividing line between "agent-usable" and "agent-preferred." An agent can theoretically call any REST API if it has the OpenAPI spec, but an MCP server wraps your capabilities in the native protocol agents already speak. It's the difference between an agent translating your API docs and an agent calling your tools directly.

Pillar 4: Content (20 points)

Content measures the quality of your human-readable descriptions — not for humans, but for the language models that power agents. Agents don't read your marketing copy the way a customer does. They parse it for capability signals, intent matching, and disambiguation. Clear, jargon-free content that describes what your product does (not how cool it is) directly improves agent routing accuracy.

Check Points What agents look for
Plain-language capability descriptions 6 "Extracts line items from invoices" beats "AI-powered document intelligence platform"
No jargon walls or buzzword stacking 4 Marketing copy that sounds impressive to humans is noise to agents parsing for capability match
Clear value proposition for agents specifically 5 What can an agent do with your product? Not what a human experiences — what a machine calls
Documentation covers agent use cases 5 API docs, integration guides, and examples written for agent consumption, not just developer onboarding

This pillar is where most SaaS products lose points they don't expect to lose. Their content is optimized for human marketing — benefit-driven, emotionally resonant, designed to convert visitors. That's great for your website. It's terrible for agents, which need to parse "what this product actually does" from your homepage in under 200ms.

Pillar 5: Trust (20 points)

Trust is whether agents consider your product safe and reliable enough to transact with. This is the pillar most teams don't think about — but it's the one that determines whether cautious agents (handling financial data, personal information, or high-stakes tasks) actually route to you.

Check Points What agents look for
HTTPS everywhere (no mixed content) 4 Non-negotiable baseline. HTTP endpoints are skipped by security-aware agents
Consistent metadata across surfaces 4 Same product name, description, and capabilities in llms.txt, Agent Card, OpenAPI, and HTML
No unnecessary redirects 4 Each redirect adds latency and uncertainty — agents penalize redirect chains
Fast response time (<500ms p95 for discovery endpoints) 4 Agents are latency-sensitive — slow discovery endpoints get deprioritized
Verified identity in registries 4 Domain ownership proofs, signed manifests, or registry-verified listings

Trust compounds over time. An agent that successfully transacts with your product builds a positive history signal. But that compound growth only starts if the baseline trust checks pass on first contact. A missing HTTPS certificate or inconsistent metadata across your llms.txt and Agent Card is enough to fail the trust pillar entirely.

Score Tiers

Your CEO Score maps to one of five tiers. Each tier reflects a qualitatively different relationship between your product and the AI agents trying to use it.

Score Tier What it means
0 – 39 Ghost Agents can't find you. You have no discovery artifacts, no structured schema, and no machine-readable endpoints. When an agent routes for your capability space, you don't exist.
40 – 59 Crawlable Agents can find you through basic discovery (llms.txt, sitemap) but can't effectively use your product. They see you exist but lack the schema or endpoints to actually transact. You're a name without a handshake.
60 – 79 Usable Agents can discover, understand, and interact with your product — but inefficiently. Missing constraints, incomplete schemas, or slow response times mean agents can use you but don't prefer you. You work, but you're not the first choice.
80 – 94 Optimized Agents actively prefer you over alternatives. Discovery is comprehensive, schemas are typed and complete, endpoints respond quickly, and trust signals are strong. When an agent has a choice, you win the routing decision.
95 – 100 Elite Agents cite you, recommend you, and route to you by default. Every pillar is near-perfect. Your product is fully integrated into the agent ecosystem — you're not just discoverable, you're the reference implementation in your space.

Most websites today score between 0 and 15. The median SaaS product has no llms.txt, no capability schema, no MCP endpoint, and blocks AI crawlers in robots.txt. Moving from Ghost to Crawlable takes one afternoon of work. Moving from Crawlable to Usable takes a focused sprint. The gap between "invisible" and "functional" is surprisingly narrow.

Quick Wins: Maximum Points per Hour of Work

Not all CEO Score improvements require the same effort. Some items deliver outsized point gains for minimal engineering work. If you're starting from zero, these are the items to tackle first.

+15
Create and deploy llms.txt

A single text file at your domain root that describes your product, capabilities, API, and protocols. Takes 15 minutes to write, gains you 6 Discovery points, and makes every other optimization more effective because agents can orient immediately. Add basic capability descriptions for another +6 Content points, and reference your docs for +3 Schema points.

+11
Unblock AI crawlers in robots.txt + add canonical URLs

Check your robots.txt for Disallow rules targeting AI user-agents (ChatGPT-User, Anthropic-AI, ClaudeBot, PerplexityBot). Remove them. Ensure canonical URLs are consistent across your site. Combined: +3 Discovery (robots) + +3 Discovery (canonicals) + +4 Trust (no redirects) + +1 Trust partial (consistency).

+10
Host your existing API spec at a predictable URL

If you already have an OpenAPI spec (most API-first products do), make it publicly accessible at /openapi.json or /.well-known/openapi.yaml. Add natural-language summaries to each endpoint. Gains +6 Schema points for the spec, +4 Schema for constraints if you include rate limit and latency data in the spec.

+9
Add a health endpoint and HTTPS audit

A /health endpoint that returns {"status":"ok"} with a 200 response code takes 5 lines of code in any framework. Combined with an HTTPS audit (no mixed content, no HTTP-only resources), you gain +3 Endpoints + +4 Trust + partial Trust points for response time.

+6
Add JSON-LD structured data to your product page

A single <script type="application/ld+json"> block with SoftwareApplication or WebAPI schema. Include applicationCategory, featureList, and offers. Takes 20 minutes, gains +4 Schema points, and improves citation likelihood in language model outputs as a bonus.

The 80/20 of CEO optimization: llms.txt + robots.txt fixes + hosting your OpenAPI spec = roughly 36 points from three tasks. That's the difference between Ghost tier and nearly Crawlable — achievable in a single afternoon.

How to Run a Self-Audit

Walk through each pillar systematically. For every check, you either pass or fail — there's no partial credit. Here's the step-by-step:

Step 1: Discovery audit

  1. Fetch https://yourdomain.com/llms.txt — does it return 200 with valid content? Does it include product description, capabilities, API URL, and protocol links?
  2. Fetch https://yourdomain.com/sitemap.xml — does it include your product/capability pages, not just blog posts?
  3. Fetch https://yourdomain.com/robots.txt — are any AI agent user-agents blocked?
  4. Check 5 random pages — do canonical tags point to a single, consistent URL for each?
  5. Search for your product in MCP server directories, agent registries, and capability indexes — are you listed in at least one?

Step 2: Schema audit

  1. View source on your main product page — is there a <script type="application/ld+json"> block with valid SoftwareApplication, WebAPI, or Organization schema?
  2. Fetch your capability schema (if it exists) — does every capability have typed input and output definitions, not just text descriptions?
  3. Fetch your OpenAPI spec — does it exist at a predictable URL? Does every endpoint have a summary, parameter descriptions, and typed response schemas?
  4. Check for declared constraints — are rate limits, latency bounds, and file size limits in your spec or capability schema?

Step 3: Endpoints audit

  1. Does your product expose a programmatic API? Can you call it without a browser?
  2. Do you have an MCP server or /mcp route? Does it respond to tool discovery requests?
  3. Fetch /health or /status — does it return 200?
  4. Test API auth — can you authenticate with a bearer token or API key (not just browser cookies)?

Step 4: Content audit

  1. Read your homepage's first three paragraphs — can you extract exactly what your product does in one sentence, without jargon?
  2. Count the buzzwords on your landing page ("AI-powered," "revolutionary," "enterprise-grade"). If there are more buzzwords than concrete capability descriptions, you fail.
  3. Does your value proposition make sense from an agent's perspective? "An agent can use this product to [verb] [object]" — can you fill in that sentence from your content?
  4. Check your docs — are there agent-specific integration guides or just human developer onboarding?

Step 5: Trust audit

  1. Check every page and API endpoint for HTTPS. Any HTTP-only resources or mixed content is a fail.
  2. Compare your product name, description, and capability list across llms.txt, Agent Card, OpenAPI spec, and HTML meta tags — are they consistent?
  3. Run curl -sI on your main endpoints — count the redirects. More than one hop on any path is a fail.
  4. Measure response time on your discovery endpoints (/llms.txt, /openapi.json, /.well-known/agent.json) — are they under 500ms p95?
  5. Check your registry listing (if any) — is your identity verified, or just self-declared?

What's Coming

The CEO Score as presented here is a manual framework — you walk through the checklist, check each item, and calculate your score. That's intentional for v1: a manual audit forces you to understand each criterion and see exactly where your gaps are. But the trajectory is clear.

Automated scoring

The CEO Score is designed to be fully automatable. Every criterion maps to a testable check — an HTTP fetch, a schema validation, a response time measurement. Automated CEO scoring tools will probe your domain, test every artifact, and return a scored breakdown in seconds. This is the Core Web Vitals model: start manual, then build tooling that runs continuously.

CEO Score as an industry benchmark

As the agent economy matures, CEO Scores will become a competitive benchmark — the way Lighthouse scores and PageSpeed Insights became benchmarks for web performance. Registries will surface CEO Scores alongside product listings. Agents will use them as a signal in routing decisions: given two products with similar capabilities, route to the one with the higher CEO Score.

Agent analytics

Today, you can see your web traffic in Google Analytics and your API calls in your logs. What's missing is agent-specific analytics: which agents are discovering you, which pillar they enter through, how many complete a transaction, and where they drop off. CEO Score analytics will close that loop — giving you the same visibility into agent interactions that GA gives you for human interactions.

The window is now. Products that establish high CEO Scores early will have compounding advantages: discovery history in registries, trust scores built over time, and agent routing preferences that are hard for late entrants to displace. The best time to start measuring was yesterday. The second best time is right now.
