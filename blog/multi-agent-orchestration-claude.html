Multi-Agent Orchestration with Claude: Patterns for Parallel Subagents

Orchestrator-subagent architecture, parallel fan-out with asyncio TaskGroup, result synthesis, agent routing, handoff protocols, and error isolation across agent boundaries.

Apr 14, 2026  ·  13 min read  ·  CEO Editorial

Single-agent systems hit a wall on complex tasks. Long context windows help, but they don't solve the fundamental problem: one agent doing everything sequentially is slow, and a single large context costs more than several focused smaller ones. Multi-agent architectures solve both problems — parallelism cuts latency, and specialized subagents use focused contexts that are cheaper per token.

This guide covers the patterns that work in production: the orchestrator-subagent model, parallel execution with asyncio.TaskGroup, result synthesis, routing, agent handoffs, and error isolation so one failed subagent doesn't take down the entire pipeline.

1. The Orchestrator-Subagent Pattern

The orchestrator receives a high-level task, decomposes it, dispatches subagents, and synthesizes results. Subagents are narrow: they receive a well-defined subtask and return a structured result. Neither knows about the other's existence.

from dataclasses import dataclass, field
from typing import Any
import anthropic
import asyncio
import json

client = anthropic.AsyncAnthropic()

@dataclass
class SubtaskResult:
    agent_id: str
    task: str
    result: str
    success: bool
    tokens_used: int = 0

@dataclass
class OrchestrationPlan:
    subtasks: list[dict] = field(default_factory=list)
    synthesis_prompt: str = ""

async def orchestrate(task: str) -> str:
    # Step 1: plan decomposition
    plan = await decompose_task(task)

    # Step 2: parallel subagent execution
    results = await run_subagents_parallel(plan.subtasks)

    # Step 3: synthesis
    return await synthesize_results(task, results, plan.synthesis_prompt)

Decomposition via structured output

The orchestrator uses a JSON schema tool to produce a machine-readable plan. This is more reliable than parsing free text — Claude fills in a defined structure rather than improvising a format.

DECOMPOSE_TOOL = {
    "name": "create_plan",
    "description": "Decompose a complex task into parallel subtasks",
    "input_schema": {
        "type": "object",
        "properties": {
            "subtasks": {
                "type": "array",
                "items": {
                    "type": "object",
                    "properties": {
                        "agent_id": {"type": "string"},
                        "task": {"type": "string"},
                        "context": {"type": "string"},
                        "can_run_parallel": {"type": "boolean"}
                    },
                    "required": ["agent_id", "task", "context", "can_run_parallel"]
                }
            },
            "synthesis_prompt": {"type": "string"}
        },
        "required": ["subtasks", "synthesis_prompt"]
    }
}

async def decompose_task(task: str) -> OrchestrationPlan:
    response = await client.messages.create(
        model="claude-opus-4-6",
        max_tokens=1024,
        system="You are a task planner. Break complex requests into focused parallel subtasks.",
        messages=[{"role": "user", "content": task}],
        tools=[DECOMPOSE_TOOL],
        tool_choice={"type": "tool", "name": "create_plan"}
    )
    tool_use = next(b for b in response.content if b.type == "tool_use")
    plan_data = tool_use.input
    return OrchestrationPlan(
        subtasks=plan_data["subtasks"],
        synthesis_prompt=plan_data["synthesis_prompt"]
    )

2. Parallel Fan-Out with asyncio.TaskGroup

asyncio.TaskGroup (Python 3.11+) is the right primitive for parallel subagents. It runs all tasks concurrently, surfaces the first exception if any task fails, and cancels remaining tasks automatically — cleaner than managing gather() with return_exceptions=True.

async def run_subagents_parallel(
    subtasks: list[dict],
    max_concurrent: int = 5
) -> list[SubtaskResult]:
    semaphore = asyncio.Semaphore(max_concurrent)
    results: list[SubtaskResult] = []

    async def run_one(subtask: dict) -> SubtaskResult:
        async with semaphore:
            try:
                result = await run_subagent(
                    agent_id=subtask["agent_id"],
                    task=subtask["task"],
                    context=subtask["context"]
                )
                results.append(result)
                return result
            except Exception as e:
                # Isolate: record failure but don't crash others
                failed = SubtaskResult(
                    agent_id=subtask["agent_id"],
                    task=subtask["task"],
                    result=f"ERROR: {e}",
                    success=False
                )
                results.append(failed)
                return failed

    # Run parallel and sequential tasks separately
    parallel = [t for t in subtasks if t["can_run_parallel"]]
    sequential = [t for t in subtasks if not t["can_run_parallel"]]

    async with asyncio.TaskGroup() as tg:
        for subtask in parallel:
            tg.create_task(run_one(subtask))

    # Sequential tasks run after parallel batch completes
    for subtask in sequential:
        await run_one(subtask)

    return results
Semaphore vs TaskGroup The semaphore caps concurrent Claude API calls to avoid rate limit errors. TaskGroup handles the concurrency primitives. Set max_concurrent based on your tier's requests-per-minute limit — at Tier 2, 5 concurrent Sonnet calls is safe; at Tier 4, you can push 20+.

Individual subagent execution

async def run_subagent(
    agent_id: str,
    task: str,
    context: str,
    model: str = "claude-sonnet-4-6"
) -> SubtaskResult:
    # Subagents use focused system prompts, not the orchestrator's full context
    system = f"""You are a specialized agent with ID '{agent_id}'.
Complete the assigned subtask and return ONLY the result.
Be concise. The orchestrator will handle synthesis.
Context: {context}"""

    response = await client.messages.create(
        model=model,
        max_tokens=2048,
        system=[{
            "type": "text",
            "text": system,
            "cache_control": {"type": "ephemeral"}  # cache repeated system prompts
        }],
        messages=[{"role": "user", "content": task}]
    )

    return SubtaskResult(
        agent_id=agent_id,
        task=task,
        result=response.content[0].text,
        success=True,
        tokens_used=response.usage.input_tokens + response.usage.output_tokens
    )

3. Result Synthesis

Synthesis is where the orchestrator earns its place. It receives all subagent outputs and produces a unified response. The orchestrator model (Opus) is appropriate here — it needs to reason across multiple perspectives and catch contradictions.

async def synthesize_results(
    original_task: str,
    results: list[SubtaskResult],
    synthesis_prompt: str
) -> str:
    # Build a structured summary of what succeeded and what failed
    success_results = [r for r in results if r.success]
    failed_results = [r for r in results if not r.success]

    agent_outputs = "\n\n".join([
        f"=== Agent {r.agent_id} ===\nTask: {r.task}\nResult: {r.result}"
        for r in success_results
    ])

    failure_note = ""
    if failed_results:
        failed_ids = ", ".join(r.agent_id for r in failed_results)
        failure_note = f"\n\nNote: agents [{failed_ids}] failed. Synthesize from available results."

    response = await client.messages.create(
        model="claude-opus-4-6",
        max_tokens=4096,
        system="You are a synthesis engine. Combine agent outputs into a coherent, unified response.",
        messages=[{
            "role": "user",
            "content": f"""Original task: {original_task}

Agent outputs:
{agent_outputs}{failure_note}

Synthesis instructions: {synthesis_prompt}"""
        }]
    )
    return response.content[0].text

4. Agent Routing

Not every task needs decomposition. A router inspects the incoming request and selects the appropriate specialist agent — no orchestration overhead for simple cases.

from enum import StrEnum

class AgentType(StrEnum):
    CODE_REVIEW = "code_review"
    DATA_ANALYSIS = "data_analysis"
    DOCUMENT_SUMMARY = "document_summary"
    ORCHESTRATOR = "orchestrator"

AGENT_CONFIGS: dict[AgentType, dict] = {
    AgentType.CODE_REVIEW: {
        "model": "claude-sonnet-4-6",
        "system": "You are a senior code reviewer. Focus on correctness, security, and maintainability.",
        "max_tokens": 2048
    },
    AgentType.DATA_ANALYSIS: {
        "model": "claude-sonnet-4-6",
        "system": "You are a data analyst. Identify patterns, anomalies, and actionable insights.",
        "max_tokens": 3000
    },
    AgentType.DOCUMENT_SUMMARY: {
        "model": "claude-haiku-4-5-20251001",  # cheaper for simple summarization
        "system": "You are a document summarizer. Be concise and structured.",
        "max_tokens": 1024
    },
    AgentType.ORCHESTRATOR: {
        "model": "claude-opus-4-6",
        "system": "You handle complex multi-step tasks requiring decomposition.",
        "max_tokens": 4096
    }
}

ROUTE_TOOL = {
    "name": "route_request",
    "description": "Route the request to the appropriate specialized agent",
    "input_schema": {
        "type": "object",
        "properties": {
            "agent_type": {
                "type": "string",
                "enum": [e.value for e in AgentType]
            },
            "reason": {"type": "string"}
        },
        "required": ["agent_type", "reason"]
    }
}

async def route_request(user_input: str) -> tuple[AgentType, str]:
    # Use Haiku for routing — fast and cheap
    response = await client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=256,
        system="Classify requests and route them to the correct specialist agent.",
        messages=[{"role": "user", "content": user_input}],
        tools=[ROUTE_TOOL],
        tool_choice={"type": "tool", "name": "route_request"}
    )
    tool_use = next(b for b in response.content if b.type == "tool_use")
    agent_type = AgentType(tool_use.input["agent_type"])
    reason = tool_use.input["reason"]
    return agent_type, reason

async def handle_request(user_input: str) -> str:
    agent_type, _ = await route_request(user_input)
    config = AGENT_CONFIGS[agent_type]

    if agent_type == AgentType.ORCHESTRATOR:
        return await orchestrate(user_input)

    response = await client.messages.create(
        model=config["model"],
        max_tokens=config["max_tokens"],
        system=config["system"],
        messages=[{"role": "user", "content": user_input}]
    )
    return response.content[0].text
Model selection by role Use Haiku for routing (fast, cheap, needs low reasoning). Use Sonnet for subagents (capable, cost-effective). Use Opus for orchestration and synthesis (complex reasoning across many inputs). A typical orchestrated request costs ~3–5x a single Sonnet call but does 4–8x the work in parallel.

5. Agent Handoff Protocol

Some tasks require sequential handoffs where agent B needs agent A's output before it can start. This differs from fan-out — the agents form a pipeline, and each step's output becomes the next step's input.

@dataclass
class HandoffContext:
    """Passed between agents in a pipeline."""
    original_task: str
    pipeline_stage: int
    prior_outputs: list[SubtaskResult]
    metadata: dict[str, Any] = field(default_factory=dict)

    def as_context_string(self) -> str:
        prior = "\n".join([
            f"Stage {i+1} ({r.agent_id}): {r.result[:500]}..."
            if len(r.result) > 500
            else f"Stage {i+1} ({r.agent_id}): {r.result}"
            for i, r in enumerate(self.prior_outputs)
        ])
        return f"Original task: {self.original_task}\n\nPrior stages:\n{prior}"

async def run_pipeline(
    task: str,
    stages: list[dict]  # each: {agent_id, role, system_prompt}
) -> HandoffContext:
    ctx = HandoffContext(original_task=task, pipeline_stage=0, prior_outputs=[])

    for stage in stages:
        handoff_message = f"""{ctx.as_context_string()}

Your role: {stage['role']}
Your task: Based on the above context, perform your specialized analysis."""

        response = await client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=2048,
            system=stage["system_prompt"],
            messages=[{"role": "user", "content": handoff_message}]
        )

        ctx.prior_outputs.append(SubtaskResult(
            agent_id=stage["agent_id"],
            task=stage["role"],
            result=response.content[0].text,
            success=True,
            tokens_used=response.usage.input_tokens + response.usage.output_tokens
        ))
        ctx.pipeline_stage += 1

    return ctx

]

6. Error Isolation and Retry Logic

Production multi-agent systems need two layers of resilience: per-subagent retry for transient API errors, and graceful degradation when retries exhaust. The orchestrator should always complete — either with all results or with a partial synthesis that acknowledges the gaps.

import asyncio
from anthropic import APIStatusError, APITimeoutError, RateLimitError

async def run_subagent_with_retry(
    agent_id: str,
    task: str,
    context: str,
    max_retries: int = 3,
    base_delay: float = 1.0
) -> SubtaskResult:
    last_error: Exception | None = None

    for attempt in range(max_retries):
        try:
            return await run_subagent(agent_id, task, context)

        except RateLimitError as e:
            # Exponential backoff for rate limits
            delay = base_delay * (2 ** attempt)
            await asyncio.sleep(delay)
            last_error = e

        except APITimeoutError as e:
            # Shorter retry for timeouts
            await asyncio.sleep(base_delay)
            last_error = e

        except APIStatusError as e:
            if e.status_code >= 500:
                # Server error — retry
                await asyncio.sleep(base_delay * (attempt + 1))
                last_error = e
            else:
                # Client error (4xx) — don't retry
                return SubtaskResult(
                    agent_id=agent_id, task=task,
                    result=f"Client error {e.status_code}: {e.message}",
                    success=False
                )

    return SubtaskResult(
        agent_id=agent_id, task=task,
        result=f"Failed after {max_retries} attempts: {last_error}",
        success=False
    )
Minimum viable result threshold Before calling the synthesis step, check if enough subagents succeeded. If fewer than 50% succeeded, abort synthesis and return an error rather than producing an unreliable partial result. Define this threshold per task type — a legal document analysis needs all agents; a content summary can tolerate 1–2 failures.

7. Prompt Caching Across Subagents

When multiple subagents share a large common context (a document, a codebase, a knowledge base), prompt caching saves significant cost. Mark the shared context with cache_control: {"type": "ephemeral"} — the cache persists for 5 minutes, long enough to cover all parallel subagent calls.

async def analyze_document_parallel(document: str, analysis_tasks: list[str]) -> list[SubtaskResult]:
    # Shared document cached once, reused by all subagents
    shared_context = [{
        "type": "text",
        "text": f"Document to analyze:\n\n{document}",
        "cache_control": {"type": "ephemeral"}
    }]

    async def analyze_one(task: str, i: int) -> SubtaskResult:
        response = await client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            system="You are a document analyst. Answer the specific question asked.",
            messages=[{
                "role": "user",
                "content": shared_context + [{
                    "type": "text",
                    "text": task  # not cached — unique per subagent
                }]
            }]
        )
        cache_read = response.usage.cache_read_input_tokens
        print(f"Agent {i}: {cache_read} tokens from cache")
        return SubtaskResult(
            agent_id=f"analyst_{i}",
            task=task,
            result=response.content[0].text,
            success=True,
            tokens_used=response.usage.input_tokens + response.usage.output_tokens
        )

    async with asyncio.TaskGroup() as tg:
        tasks = [tg.create_task(analyze_one(t, i)) for i, t in enumerate(analysis_tasks)]

    return [t.result() for t in tasks]

8. Monitoring and Cost Tracking

Multi-agent systems can silently accumulate high token costs. Track spend per orchestration run, not per API call — this makes it easy to set cost ceilings per task type.

@dataclass
class OrchestrationMetrics:
    run_id: str
    total_tokens: int = 0
    total_cost_usd: float = 0.0
    agent_count: int = 0
    failed_agents: int = 0
    wall_time_ms: float = 0.0

# Pricing per million tokens (input/output)
MODEL_PRICING = {
    "claude-opus-4-6":       {"input": 15.00, "output": 75.00},
    "claude-sonnet-4-6":     {"input": 3.00,  "output": 15.00},
    "claude-haiku-4-5-20251001": {"input": 0.80,  "output": 4.00},
}

def calculate_cost(model: str, input_tokens: int, output_tokens: int) -> float:
    pricing = MODEL_PRICING.get(model, MODEL_PRICING["claude-sonnet-4-6"])
    return (
        (input_tokens / 1_000_000) * pricing["input"] +
        (output_tokens / 1_000_000) * pricing["output"]
    )

async def orchestrate_with_metrics(task: str) -> tuple[str, OrchestrationMetrics]:
    import time, uuid
    metrics = OrchestrationMetrics(run_id=str(uuid.uuid4())[:8])
    start = time.monotonic()

    plan = await decompose_task(task)
    results = await run_subagents_parallel(plan.subtasks)
    final = await synthesize_results(task, results, plan.synthesis_prompt)

    metrics.wall_time_ms = (time.monotonic() - start) * 1000
    metrics.agent_count = len(results)
    metrics.failed_agents = sum(1 for r in results if not r.success)
    metrics.total_tokens = sum(r.tokens_used for r in results)

    return final, metrics

Pattern Summary

PatternWhen to useKey primitive
Fan-out / Fan-inIndependent subtasks — parallel research, multi-aspect analysisasyncio.TaskGroup + semaphore
RouterMixed request types, single-agent sufficient for mostHaiku + tool_choice
Pipeline / handoffSequential steps where each depends on prior outputHandoffContext passed forward
Shared-context fan-outMany agents analyzing the same large documentcache_control: ephemeral
Retry with degradationProduction systems that must always return a resultExponential backoff + partial synthesis
Cost rule of thumb A 4-subagent Sonnet fan-out with Opus synthesis costs roughly $0.01–0.05 per orchestration run for 2K–10K total tokens. Use Haiku subagents for high-volume workflows. Cache shared context aggressively — on a 50K-token document, cache hits save ~90% of input cost on the second and subsequent subagent calls.

Multi-agent systems pay off when tasks are genuinely parallelizable or when specialist agents outperform a generalist on subtasks. For simple requests, routing to a single Sonnet call is always cheaper than spinning up an orchestrator. Build the router first — it gives you clean data on which request types actually benefit from decomposition before you invest in the full orchestration stack.
