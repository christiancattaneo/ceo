Structured Outputs in AI Agents: JSON Mode, Schema Enforcement, and Reliable Data Extraction

Asking an LLM to "respond in JSON" and actually getting valid, schema-conformant JSON every time are two different problems. Here's how to use structured outputs properly — and how to handle the cases where they still fail.

The Problem With Unstructured LLM Output

When an agent's output needs to flow into a downstream system — a database write, a tool call, a UI component — you need that output to have a predictable shape. Prose responses are fine for displaying to users. They're not fine when you need to parse a status field, extract a list of action items, or trigger a workflow based on a classification.

The naive approach is to ask for JSON in the prompt: "Respond with a JSON object containing...". This works most of the time. But "most of the time" is not production-grade reliability. Models will occasionally:

  • Wrap the JSON in markdown code fences (```json ... ```)
  • Include a preamble before the JSON ("Here is the JSON: {...}")
  • Produce keys with slightly different names than specified
  • Omit optional fields instead of including them with null values
  • Add extra explanation after the closing brace
  • Generate valid JSON that doesn't match the expected schema

Any of these edge cases breaks a parser that expects clean JSON input. In production, these failures cluster around the edge cases in your prompt — the inputs the model finds ambiguous or unusual — which means they're hardest to reproduce in development and most likely to surface at scale.

The Three Approaches to Structured Output

Approach Reliability Flexibility Works with
Prompt-only JSON ~95% (varies by model) High — any schema All models
JSON mode ~99% (valid JSON guaranteed) Medium — valid JSON, not schema-validated OpenAI, Anthropic (tool use), Gemini
Structured outputs / constrained decoding ~99.9% (schema conformance guaranteed) Lower — schema must be JSON Schema OpenAI gpt-4o+, some open-source models

Prompt-Only JSON

Describe the desired output format in the prompt with examples. Works well for simple schemas and capable models. The failure rate is low enough for many use cases, and it doesn't lock you into any particular API feature.

The key to reliable prompt-only JSON:

  • Show a concrete example, not just a schema description
  • End the prompt with the opening brace of the JSON to prime the model to continue in JSON format
  • Specify what to do with null/missing values explicitly
  • Keep the schema simple — nested objects with many optional fields are harder
EXTRACT_PROMPT = """
Extract the following information from the user message.
Return ONLY a JSON object with no preamble or explanation.

Schema:
{
  "intent": "one of: question, complaint, purchase_request, other",
  "product_mentioned": "product name or null if none",
  "urgency": "one of: high, medium, low",
  "summary": "one sentence summary of the request"
}

Example output:
{"intent": "complaint", "product_mentioned": "Pro Plan", "urgency": "high", "summary": "User reports billing error on Pro Plan subscription."}

User message: {user_message}

JSON:"""

Ending with JSON: followed by nothing strongly biases the model to start with { on the next token. It's a simple trick that meaningfully reduces preamble generation.

JSON Mode

JSON mode (available in OpenAI, Anthropic via tool use, Google Gemini) constrains the model's output to be syntactically valid JSON. It doesn't guarantee the JSON matches your schema — it just guarantees it parses. You still need to validate the parsed object against your expected structure.

# OpenAI JSON mode
response = client.chat.completions.create(
    model="gpt-4o",
    response_format={"type": "json_object"},
    messages=[
        {"role": "system", "content": "You are a data extraction assistant. Always respond with valid JSON."},
        {"role": "user", "content": f"Extract intent and urgency from: {user_message}"}
    ]
)
data = json.loads(response.choices[0].message.content)
# data is guaranteed to parse — but may not have the fields you expect

JSON mode eliminates parse errors but not schema mismatch errors. A model in JSON mode might return {"urgency_level": "high"} when you expected {"urgency": "high"}. Always validate the parsed object.

Structured Outputs / Constrained Decoding

The strongest approach: provide a JSON Schema and the model's output is guaranteed to conform to it at the token level. The decoding process is constrained to only produce tokens that keep the output schema-valid. This is available in OpenAI's response_format with a schema, and in open-source models via libraries like Outlines or Guidance.

# OpenAI structured outputs
from pydantic import BaseModel
from typing import Literal

class ExtractedIntent(BaseModel):
    intent: Literal["question", "complaint", "purchase_request", "other"]
    product_mentioned: str | None
    urgency: Literal["high", "medium", "low"]
    summary: str

response = client.beta.chat.completions.parse(
    model="gpt-4o-2024-08-06",
    messages=[...],
    response_format=ExtractedIntent,
)
result: ExtractedIntent = response.choices[0].message.parsed
# result is a validated Pydantic object — no json.loads, no KeyError, no validation needed

The returned object is already a validated Pydantic instance. You can't get a parse error or a missing field. This is the most reliable approach for production pipelines where the output feeds directly into typed code.

Constrained decoding has a small quality cost. Forcing the model to conform to a strict schema can reduce the quality of generated text slightly, because the model sometimes can't produce the most natural phrasing if it would violate the schema. For extraction tasks (where there's a ground truth to extract) this tradeoff is almost always worth it. For generation tasks (where the model is creating content), consider whether strict schema enforcement is actually necessary.

Validation with Pydantic

Whether you use JSON mode or prompt-only JSON, adding a Pydantic validation layer between the raw LLM output and your application code is the most impactful reliability improvement you can make for minimal effort.

from pydantic import BaseModel, validator
from typing import Optional

class ClassificationResult(BaseModel):
    intent: str
    confidence: float
    entities: list[str]
    requires_human_review: bool

    @validator('confidence')
    def confidence_in_range(cls, v):
        if not 0.0 <= v <= 1.0:
            raise ValueError(f'confidence must be 0-1, got {v}')
        return v

    @validator('intent')
    def valid_intent(cls, v):
        valid = {"question", "complaint", "purchase_request", "other"}
        if v not in valid:
            raise ValueError(f'intent must be one of {valid}, got {v!r}')
        return v

def parse_classification(raw_json: str) -> ClassificationResult:
    try:
        data = json.loads(raw_json)
        return ClassificationResult(**data)
    except (json.JSONDecodeError, ValidationError) as e:
        raise OutputParseError(f"LLM output failed validation: {e}") from e

Pydantic validators catch problems that JSON mode can't: values outside expected ranges, strings that don't match enum constraints, missing required fields. The validation error message is also useful for retry strategies — you can feed the error back to the model and ask for a correction.

Retry Strategies for Parse Failures

Even with structured outputs, failures happen. Your retry strategy should be smarter than "call the API again with the same prompt."

Self-correction with the validation error

When parsing fails, feed the error message back to the model as a follow-up message and ask it to fix the output:

def extract_with_retry(prompt: str, max_retries: int = 2) -> ClassificationResult:
    messages = [{"role": "user", "content": prompt}]

    for attempt in range(max_retries + 1):
        response = call_llm(messages)
        raw = response.choices[0].message.content

        try:
            return parse_classification(raw)
        except OutputParseError as e:
            if attempt == max_retries:
                raise

            # Ask model to fix its output
            messages.append({"role": "assistant", "content": raw})
            messages.append({
                "role": "user",
                "content": f"Your response failed validation: {e}\n\nPlease provide a corrected JSON response that fixes this error."
            })

Self-correction works surprisingly well because the model can often reason about why its output was wrong when shown the specific error. This approach catches ~80% of initial failures in one retry, at the cost of one additional API call for the failures.

Fallback to a more capable model

If your primary extraction uses a fast/cheap model (haiku, mini, flash), consider falling back to a more capable model on validation failure. The speed-cost optimization holds for the 95%+ of cases that succeed on first try; the expensive model only runs on failures.

async def extract_robust(prompt: str) -> ClassificationResult:
    # Try fast model first
    try:
        return await extract_with_model(prompt, model="claude-haiku-4-5")
    except OutputParseError:
        pass

    # Fall back to capable model
    return await extract_with_model(prompt, model="claude-sonnet-4-6")

When Structured Outputs Break Down

Structured outputs are not a silver bullet. Several scenarios cause even schema-constrained models to produce unusable results:

The schema is too narrow for the input

If a user's input doesn't fit any of your enumerated values, the model is forced to pick the closest one — which may be wrong. An intent classifier with four options will mis-classify inputs that don't fit any of them, not report "none of the above." Design escape hatches: "other" categories, nullable fields, or a confidence field that signals uncertain classifications.

Deeply nested schemas with many optional fields

Complex schemas — objects nested 3+ levels deep, many optional fields, arrays of objects — are harder for models to fill correctly. A constrained decoder can guarantee the JSON is structurally valid, but it can't guarantee the semantics are right. A model forced to populate a deeply nested schema may hallucinate plausible-looking values for fields it doesn't have enough information to fill.

Flatten schemas where possible. Extract in stages (extract the top-level structure first, then drill into sub-objects in separate calls) rather than asking for everything at once.

Long outputs with structured requirements

For very long outputs (articles, reports, extensive analysis) where you also need structured metadata, don't put the long text inside the structured object. Extract the metadata separately from the content generation. Trying to constrain a 2,000-word generation to a JSON schema adds latency, increases cost, and often degrades content quality.

Structured outputs aren't a substitute for output validation in your business logic. A model can produce a perfectly valid ClassificationResult with confidence: 0.95 that is semantically wrong — the classification is incorrect but schema-valid. Structured outputs handle format reliability. Semantic correctness still requires evaluation, human review thresholds, or LLM-as-judge techniques.

Practical Extraction Patterns

Chain of thought before structured output

For complex extractions, asking the model to reason before outputting the JSON often improves accuracy. Have the model think through the problem in a reasoning field that you discard, followed by the actual structured fields you need:

class AnalysisResult(BaseModel):
    reasoning: str  # model's scratchpad — you won't use this downstream
    classification: Literal["positive", "negative", "neutral"]
    key_topics: list[str]
    action_required: bool

The reasoning field "uses up" the model's tendency to produce explanatory text before the answer — channeling it into a field you can ignore rather than letting it pollute the structured fields.

Extraction + generation separation

Use structured outputs for the extraction pass; use free-form generation for the content pass. Don't mix them. If you need both a structured summary and a prose explanation of a document, make two calls: one that extracts the structured metadata, one that generates the prose.
