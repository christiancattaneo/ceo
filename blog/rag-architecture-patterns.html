RAG Architecture Patterns: From Naive Retrieval to Production

Naive RAG — embed your documents, retrieve the top-k, stuff them in the prompt — works in demos. It fails in production because chunking destroys context, vector search misses lexical matches, and retrieved passages aren't always what the question actually needs. This is the upgrade path.

What Naive RAG Gets Wrong

The basic retrieval-augmented generation pipeline has three steps: index your corpus (chunk → embed → store), retrieve at query time (embed query → ANN search → return top-k chunks), and generate (inject chunks into prompt → LLM response). This works well enough to build a demo in a day and badly enough to frustrate users in week two.

The failure modes are predictable:

  • Chunks lose context. A 512-token chunk from the middle of a document has no knowledge of the document's title, the section it belongs to, or the argument it's part of. Semantically it looks like an isolated paragraph. Retrieval accuracy drops because the embedding doesn't capture the broader meaning.
  • Vector search misses exact matches. "What is the ISO 27001 certification status of vendor X?" is a keyword query. Dense retrieval will find semantically similar text about certifications but miss the exact passage that contains "ISO 27001" if it wasn't in the training distribution of your embedding model.
  • Top-k isn't top-relevant. ANN returns the k nearest vectors, not the k most relevant chunks for answering the question. Position in the ranking matters — irrelevant chunks consume context window and dilute the signal for the LLM.
  • No query understanding. Users don't phrase queries the way documents are written. "How do I fix the login bug?" won't match a chunk titled "Authentication Error Resolution Procedures."

Each of these has a specific fix. Together they define what production RAG looks like.

Chunking Strategies

Chunking is where most production RAG systems win or lose. The goal is to produce chunks that are semantically self-contained — a chunk should be understandable and useful without requiring knowledge of surrounding chunks.

Fixed-size chunking (naive baseline)

Split every N tokens with M tokens of overlap. Fast to implement, terrible for anything structural. A 512-token chunk will routinely split mid-sentence, mid-table, or mid-code-block. Use only if your documents have no meaningful structure.

def fixed_chunk(text: str, size: int = 512, overlap: int = 50) -> list[str]:
    tokens = tokenizer.encode(text)
    chunks = []
    for i in range(0, len(tokens), size - overlap):
        chunk_tokens = tokens[i:i + size]
        chunks.append(tokenizer.decode(chunk_tokens))
    return chunks

Recursive character splitting

Split on a hierarchy of separators: try paragraph breaks first, then sentences, then words. LangChain's RecursiveCharacterTextSplitter popularized this. Better than fixed-size — it respects paragraph and sentence boundaries — but still blind to document structure.

Semantic chunking

Embed every sentence, compute cosine similarity between adjacent sentences, and split where similarity drops below a threshold. Chunks track topic changes rather than arbitrary size limits. Slower to index but produces significantly more coherent chunks for narrative documents.

def semantic_chunk(sentences: list[str], threshold: float = 0.75) -> list[str]:
    embeddings = embed_batch(sentences)
    chunks, current = [], [sentences[0]]

    for i in range(1, len(sentences)):
        sim = cosine_similarity(embeddings[i-1], embeddings[i])
        if sim < threshold:
            chunks.append(" ".join(current))
            current = [sentences[i]]
        else:
            current.append(sentences[i])

    if current:
        chunks.append(" ".join(current))
    return chunks

Document-structure-aware chunking

For structured documents (Markdown, HTML, PDFs with headings), parse the document tree and chunk by section. A heading-level-2 section becomes a single chunk with its subsections. This preserves the natural semantic units of the document and prevents splitting across related content.

Contextual chunk enrichment

Prepend each chunk with a brief contextual header generated by an LLM at index time. Anthropic's contextual retrieval paper showed this can reduce retrieval failures by 49% compared to vanilla chunking:

def enrich_chunk(document_title: str, document_summary: str,
                 chunk_text: str, client) -> str:
    context = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=200,
        messages=[{
            "role": "user",
            "content": f"""Document: {document_title}
Summary: {document_summary}
Chunk: {chunk_text}

Write a 1-2 sentence context header for this chunk that describes what
document it comes from and what role this passage plays in the document.
Be specific and concise."""
        }]
    )
    return context.content[0].text + "\n\n" + chunk_text

The overhead is one cheap LLM call per chunk at index time — a one-time cost that pays off at every query.

Hybrid Search: Dense + Sparse

Dense retrieval (vector similarity) excels at semantic matching. Sparse retrieval (BM25, TF-IDF) excels at exact term matching. Production systems use both and merge results with Reciprocal Rank Fusion (RRF).

from rank_bm25 import BM25Okapi

def hybrid_search(
    query: str,
    vector_store,
    bm25_index: BM25Okapi,
    corpus: list[str],
    k: int = 10,
    alpha: float = 0.5   # weight: 0=pure sparse, 1=pure dense
) -> list[str]:

    # Dense retrieval
    dense_results = vector_store.similarity_search(query, k=k)
    dense_ids = {doc.id: rank for rank, doc in enumerate(dense_results)}

    # Sparse retrieval
    tokenized_query = query.lower().split()
    bm25_scores = bm25_index.get_scores(tokenized_query)
    sparse_ids = {i: rank for rank, i in
                  enumerate(bm25_scores.argsort()[::-1][:k])}

    # Reciprocal Rank Fusion
    rrf_scores: dict[str, float] = {}
    for doc_id, rank in dense_ids.items():
        rrf_scores[doc_id] = rrf_scores.get(doc_id, 0) + alpha / (60 + rank)
    for doc_id, rank in sparse_ids.items():
        rrf_scores[str(doc_id)] = rrf_scores.get(str(doc_id), 0) + \
                                  (1 - alpha) / (60 + rank)

    sorted_ids = sorted(rrf_scores, key=rrf_scores.get, reverse=True)
    return [corpus[int(i)] if i.isdigit() else i for i in sorted_ids[:k]]

The 60 constant in RRF is not arbitrary. It was empirically derived in the original RRF paper to prevent high-ranking documents from dominating too aggressively. Smaller values amplify rank differences; larger values flatten them. 60 is the standard starting point — adjust based on your retrieval characteristics.

Reranking

Vector similarity is a proxy for relevance. A cross-encoder reranker is the real thing: it takes (query, chunk) pairs and scores them jointly, with full attention across both. It's too slow to run on the entire corpus but fast enough to rerank the top-50 candidates from your initial retrieval.

from sentence_transformers import CrossEncoder

reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")

def rerank(query: str, candidates: list[str], top_n: int = 5) -> list[str]:
    pairs = [(query, chunk) for chunk in candidates]
    scores = reranker.predict(pairs)

    ranked = sorted(zip(scores, candidates), reverse=True)
    return [chunk for _, chunk in ranked[:top_n]]

# Pipeline: retrieve 50 → rerank to 5 → generate
candidates = hybrid_search(query, vector_store, bm25_index, corpus, k=50)
top_chunks = rerank(query, candidates, top_n=5)

Reranking consistently improves RAG quality at modest cost. A reranker inference call is milliseconds for 50 candidates on a standard GPU. If you're not reranking, you're passing mediocre context to your LLM and wondering why the answers are mediocre.

Query Transformation

Users write queries in natural language. Documents are written in structured prose. The vocabulary gap between them causes retrieval failures that better chunking or search won't fix. Query transformation rewrites or expands the query before retrieval.

HyDE (Hypothetical Document Embeddings)

Generate a hypothetical answer to the query, embed that answer, and use it as the retrieval query. The hypothesis is often closer to the vocabulary of real documents than the original question.

def hyde_query(query: str, client) -> str:
    """Generate a hypothetical document that would answer the query."""
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=300,
        messages=[{
            "role": "user",
            "content": f"""Write a short, factual paragraph that would directly answer
this question. Write it as if you're an expert document that contains the answer.

Question: {query}

Hypothetical answer paragraph:"""
        }]
    )
    return response.content[0].text

Multi-query expansion

Generate N rephrased versions of the query, retrieve for each, then deduplicate and merge results. Increases recall at the cost of N retrieval calls — worth it for complex queries where a single formulation may miss important documents.

def multi_query_retrieve(query: str, client, vector_store, n: int = 3) -> list[str]:
    # Generate query variants
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=400,
        messages=[{
            "role": "user",
            "content": f"""Generate {n} different phrasings of this question
that would retrieve relevant documents. Return as JSON array of strings.

Question: {query}"""
        }]
    )
    variants = json.loads(response.content[0].text)
    variants.append(query)  # include original

    # Retrieve for each, deduplicate
    seen, results = set(), []
    for q in variants:
        for doc in vector_store.similarity_search(q, k=5):
            if doc.id not in seen:
                seen.add(doc.id)
                results.append(doc)
    return results

Contextual Compression

Retrieved chunks often contain more than the answer. A 500-token chunk retrieved for a specific fact about a product's price still contains 480 tokens about other things. Contextual compression strips the chunk down to just the portion relevant to the query before injecting it into the prompt.

def compress_chunk(query: str, chunk: str, client) -> str | None:
    """Extract only the parts of the chunk relevant to the query."""
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=500,
        messages=[{
            "role": "user",
            "content": f"""Extract only the sentences from the passage that are
directly relevant to answering the question. If nothing is relevant,
return the string "IRRELEVANT".

Question: {query}

Passage: {chunk}

Relevant sentences only:"""
        }]
    )
    result = response.content[0].text.strip()
    return None if result == "IRRELEVANT" else result

Compression reduces context window consumption and focuses the LLM on signal rather than noise. The tradeoff is additional LLM calls — use a fast, cheap model and run compressions in parallel.

RAG Evaluation

RAG systems have two quality surfaces: retrieval quality (did we get the right chunks?) and generation quality (given the chunks, did the LLM answer correctly?). Evaluating them separately is essential for knowing where to invest improvement effort.

Metric What it measures How to compute
Context Recall Did retrieval return the chunks needed to answer the question? For a gold-labeled QA set, check what fraction of "ground truth" chunks appear in the retrieved set
Context Precision What fraction of retrieved chunks were actually useful? LLM judge rates each retrieved chunk as relevant/irrelevant to the question; precision = relevant / total retrieved
Faithfulness Is the generated answer grounded in the retrieved context, or is the LLM confabulating? LLM judge decomposes answer into claims, checks each claim against retrieved context; faithfulness = supported claims / total claims
Answer Relevance Does the answer actually address the question asked? Embed the answer and the question; cosine similarity of their embeddings (higher = more on-topic)
Answer Correctness Is the answer factually correct? Requires ground-truth answers in your eval set; LLM judge compares generated vs. reference answer

RAGAS is a widely used open-source framework that implements these metrics. Run your eval suite on every change to chunking strategy, embedding model, or retrieval configuration — these changes can silently regress quality while appearing to improve latency or cost.

Low faithfulness is a retrieval problem, not a generation problem. When the LLM produces confabulated answers, the instinct is to improve the prompt or switch models. Usually the real cause is poor retrieval — the correct information isn't in the context, so the model fills the gap with plausible-sounding fabrication. Check context recall first before tuning generation.

The Production RAG Stack

Putting it together, a production RAG pipeline looks like this:

class ProductionRAG:
    def __init__(self, vector_store, bm25_index, corpus, reranker, client):
        self.vector_store = vector_store
        self.bm25_index = bm25_index
        self.corpus = corpus
        self.reranker = reranker
        self.client = client

    def query(self, user_query: str) -> str:
        # 1. Query transformation
        hyde_doc = hyde_query(user_query, self.client)

        # 2. Hybrid retrieval on transformed query
        candidates = hybrid_search(
            hyde_doc, self.vector_store,
            self.bm25_index, self.corpus, k=50
        )

        # 3. Rerank candidates
        top_chunks = rerank(user_query, candidates, top_n=10)

        # 4. Contextual compression (parallel)
        with ThreadPoolExecutor() as executor:
            compressed = list(executor.map(
                lambda c: compress_chunk(user_query, c, self.client),
                top_chunks
            ))
        relevant_chunks = [c for c in compressed if c is not None]

        # 5. Generate with focused context
        context = "\n\n---\n\n".join(relevant_chunks[:5])
        response = self.client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            system="""Answer the question using only the provided context.
If the context doesn't contain enough information to answer, say so.
Never fabricate information not present in the context.""",
            messages=[{
                "role": "user",
                "content": f"Context:\n{context}\n\nQuestion: {user_query}"
            }]
        )
        return response.content[0].text

When to Add Each Layer

Not every RAG system needs all of this. The upgrade path should follow the failure mode:

  • Retrieval misses relevant documents: Add hybrid search (BM25 + dense)
  • Retrieved chunks are semantically complete but ranked poorly: Add reranking
  • Chunks lose meaning without surrounding context: Add contextual enrichment at index time
  • Query vocabulary doesn't match document vocabulary: Add HyDE or multi-query expansion
  • Context window is full of partially-relevant text: Add contextual compression
  • Low faithfulness scores: Improve retrieval before touching generation

Start with the simplest system that passes your eval suite. Add complexity only when measurements show a specific failure mode. A production RAG system with well-tuned chunking and hybrid search will outperform a naive RAG system with every advanced technique layered on top.
