Pydantic Validation for LLM Outputs

Schema definition, retry-on-validation-failure, partial parse recovery, discriminated unions for multi-intent routing, and Claude's native structured output mode — with full Python examples.

Apr 13, 2026  ·  11 min read  ·  CEO Editorial

LLMs don't always return exactly what you asked for. Even with explicit instructions, you'll get missing fields, wrong types, extra prose around the JSON, or values that are technically valid JSON but semantically wrong. Pydantic gives you a typed contract between your prompt and your application code — and when the model fails to meet it, you have a clear recovery path.

This post covers the full validation stack: defining schemas, extracting JSON from messy responses, retry loops with validation error feedback, discriminated unions for intent routing, and how to use Claude's native structured output mode to reduce parse failures in the first place.

Why Pydantic for LLM Outputs

Pydantic v2 gives you three things that matter for LLM output handling:

  • Automatic coercion — "42" becomes 42 when the field is int; "true" becomes True; ISO strings become datetime objects
  • Detailed validation errors — ValidationError tells you exactly which field failed and why, which you can feed back to the model in a retry
  • JSON Schema generation — model.model_json_schema() produces the schema you can include in your system prompt, reducing hallucinated structure

Basic Schema Definition and Parse

from pydantic import BaseModel, Field, field_validator
from typing import Literal
from enum import Enum
import json


class Sentiment(Enum):
    POSITIVE = "positive"
    NEGATIVE = "negative"
    NEUTRAL = "neutral"


class ReviewAnalysis(BaseModel):
    sentiment: Sentiment
    score: float = Field(ge=0.0, le=1.0, description="Confidence score 0-1")
    key_topics: list[str] = Field(min_length=1, max_length=5)
    summary: str = Field(max_length=200)
    requires_followup: bool

    @field_validator("key_topics")
    @classmethod
    def lowercase_topics(cls, v: list[str]) -> list[str]:
        return [t.lower().strip() for t in v]


def extract_json(text: str) -> str:
    """Extract JSON from text that may contain prose before/after."""
    # Strategy 1: the whole response is JSON
    stripped = text.strip()
    if stripped.startswith("{") or stripped.startswith("["):
        return stripped

    # Strategy 2: JSON fenced block
    import re
    fence = re.search(r'```(?:json)?\s*(\{.*?\})\s*```', text, re.DOTALL)
    if fence:
        return fence.group(1)

    # Strategy 3: find the last {...} block
    brace = re.search(r'\{[^{}]*(?:\{[^{}]*\}[^{}]*)?\}', text, re.DOTALL)
    if brace:
        return brace.group(0)

    raise ValueError(f"No JSON found in response: {text[:200]}")


def parse_review(llm_response: str) -> ReviewAnalysis:
    raw_json = extract_json(llm_response)
    data = json.loads(raw_json)
    return ReviewAnalysis.model_validate(data)

Retry-on-Validation-Failure

The real power comes when you feed validation errors back to the model and ask it to correct them. A ValidationError from Pydantic gives you machine-readable field-level feedback — more useful than a generic "try again."

from pydantic import ValidationError
from anthropic import Anthropic
import json


client = Anthropic()

SYSTEM_PROMPT = """You are a review analysis engine. Always respond with valid JSON matching this schema:

{schema}

Rules:
- sentiment must be exactly one of: "positive", "negative", "neutral"
- score must be a float between 0.0 and 1.0
- key_topics must be 1-5 lowercase strings
- summary must be under 200 characters
- requires_followup must be a boolean (true/false)

Respond with JSON only — no prose, no markdown fences."""


def analyze_with_retry(
    review_text: str,
    max_retries: int = 3,
) -> ReviewAnalysis:
    schema = json.dumps(ReviewAnalysis.model_json_schema(), indent=2)
    system = SYSTEM_PROMPT.format(schema=schema)
    messages = [{"role": "user", "content": f"Analyze this review:\n\n{review_text}"}]

    last_error = None
    for attempt in range(max_retries):
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=512,
            system=system,
            messages=messages,
        )
        raw = response.content[0].text

        try:
            return parse_review(raw)

        except (json.JSONDecodeError, ValueError) as e:
            last_error = e
            # Add model response + correction request to conversation
            messages.append({"role": "assistant", "content": raw})
            messages.append({
                "role": "user",
                "content": f"Your response could not be parsed as JSON. Error: {e}\n\nPlease respond with valid JSON only.",
            })

        except ValidationError as e:
            last_error = e
            # Pydantic gives field-level errors — pass them back verbatim
            error_summary = "\n".join(
                f"  - {'.'.join(str(x) for x in err['loc'])}: {err['msg']}"
                for err in e.errors()
            )
            messages.append({"role": "assistant", "content": raw})
            messages.append({
                "role": "user",
                "content": f"Validation failed. Fix these specific issues and return corrected JSON:\n{error_summary}",
            })

    raise RuntimeError(f"Failed after {max_retries} attempts. Last error: {last_error}")
Field-Level Error Feedback Passing err['loc'] and err['msg'] back to the model dramatically improves correction rates over generic "try again" prompts. In production testing, targeted field-level feedback reduces retry loops by ~60% compared to generic error messages.

Discriminated Unions for Intent Routing

When a single endpoint handles multiple intent types — search, create, update, delete — discriminated unions let you route cleanly without a tangle of if-else checks:

from pydantic import BaseModel, Field
from typing import Annotated, Literal, Union
import json


class SearchIntent(BaseModel):
    intent: Literal["search"]
    query: str
    filters: dict[str, str] = Field(default_factory=dict)
    max_results: int = Field(default=10, ge=1, le=100)


class CreateIntent(BaseModel):
    intent: Literal["create"]
    resource_type: str
    fields: dict[str, str]


class UpdateIntent(BaseModel):
    intent: Literal["update"]
    resource_id: str
    changes: dict[str, str]


class ClarifyIntent(BaseModel):
    intent: Literal["clarify"]
    question: str  # what the model needs clarified before acting


# Pydantic discriminated union — routes on the "intent" field
AgentAction = Annotated[
    Union[SearchIntent, CreateIntent, UpdateIntent, ClarifyIntent],
    Field(discriminator="intent"),
]


def dispatch(action: AgentAction) -> str:
    match action:
        case SearchIntent():
            return f"Searching for '{action.query}' with filters {action.filters}"
        case CreateIntent():
            return f"Creating {action.resource_type} with {action.fields}"
        case UpdateIntent():
            return f"Updating {action.resource_id} with {action.changes}"
        case ClarifyIntent():
            return f"Model needs clarification: {action.question}"


from pydantic import TypeAdapter

adapter = TypeAdapter(AgentAction)

def parse_action(llm_response: str) -> AgentAction:
    raw = extract_json(llm_response)
    data = json.loads(raw)
    return adapter.validate_python(data)  # routes by "intent" field automatically

Claude's Native Structured Output Mode

Claude supports a tools-based structured output pattern that forces the model to produce JSON conforming to your schema — no extraction step needed:

from anthropic import Anthropic
from pydantic import BaseModel
import json


client = Anthropic()


def structured_call[T: BaseModel](
    prompt: str,
    schema_model: type[T],
    system: str = "",
) -> T:
    """Call Claude with forced structured output using tool use."""
    tool_def = {
        "name": "structured_output",
        "description": "Return the structured analysis result.",
        "input_schema": schema_model.model_json_schema(),
    }

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system=system,
        tools=[tool_def],
        tool_choice={"type": "tool", "name": "structured_output"},
        messages=[{"role": "user", "content": prompt}],
    )

    # Tool use response contains the structured data directly
    tool_use = next(b for b in response.content if b.type == "tool_use")
    return schema_model.model_validate(tool_use.input)


# Usage — no JSON extraction needed, no markdown fences to strip
result = structured_call(
    prompt="Analyze this customer review: 'The product arrived late but works great!'",
    schema_model=ReviewAnalysis,
    system="You are a customer review analyst.",
)
print(result.sentiment)        # Sentiment.POSITIVE
print(result.requires_followup) # False
Tool Use vs Prompt-Level JSON Forcing output via tool_choice reduces JSON parse failures to near zero — the model outputs structured data natively rather than serializing it as text. Use this when parse reliability matters more than latency (tool use adds ~50–100ms).

Partial Parse with Fallbacks

Sometimes you want to extract whatever valid fields are present even if the full schema fails — useful for long-running extractions where partial data is better than nothing:

from pydantic import BaseModel, ValidationError
from typing import Optional
import json


class ExtractedEntity(BaseModel):
    name: str
    entity_type: str
    confidence: float = Field(ge=0.0, le=1.0)
    context: Optional[str] = None
    aliases: list[str] = Field(default_factory=list)


def partial_parse(data: dict, model: type[BaseModel]) -> tuple[BaseModel | None, list[str]]:
    """Try to parse; if validation fails, collect errors and return None."""
    try:
        return model.model_validate(data), []
    except ValidationError as e:
        errors = [f"{'.'.join(str(x) for x in err['loc'])}: {err['msg']}" for err in e.errors()]
        return None, errors


def extract_entities_resilient(llm_response: str) -> list[ExtractedEntity]:
    """Parse a list of entities, skipping any that fail validation."""
    try:
        raw = extract_json(llm_response)
        items = json.loads(raw)
        if not isinstance(items, list):
            items = [items]
    except Exception:
        return []

    valid_entities = []
    for item in items:
        entity, errors = partial_parse(item, ExtractedEntity)
        if entity:
            valid_entities.append(entity)
        else:
            # Log but don't raise — keep processing remaining items
            print(f"Skipping invalid entity {item.get('name', '?')}: {errors}")

    return valid_entities

Schema Design Patterns for Reliability

PatternBeforeAfterWhy It Helps
Narrow enum vs free string category: str category: Literal["bug", "feature", "question"] Model can't hallucinate outside the enum; Pydantic validates it
Constrained numeric ranges score: float score: float = Field(ge=0, le=1) Catches model outputs like 0.95/1.0 or 85 (percent instead of fraction)
Optional with default notes: str notes: Optional[str] = None Prevents ValidationError when model omits an optional field
List length bounds tags: list[str] tags: list[str] = Field(min_length=1, max_length=5) Prevents empty lists and runaway tag generation
String length cap summary: str summary: str = Field(max_length=300) Enforces brevity; model will trim rather than exceed if instructed
Field descriptions confidence: float confidence: float = Field(description="0=uncertain, 1=certain") Descriptions appear in model_json_schema() — model reads them as instructions

Summary

The Pydantic-LLM validation stack in order of preference:

  1. Use tool_choice forced output when reliability is paramount — near-zero parse failures
  2. Inject model_json_schema() into the system prompt when tool use isn't available — gives the model the exact contract
  3. Use extract_json() + model_validate() for standard parse
  4. Retry with field-level ValidationError feedback on failure — 3 retries covers 99%+ of cases
  5. Discriminated unions for multi-intent endpoints — TypeAdapter.validate_python() routes on a discriminator field
  6. Partial parse when you need best-effort extraction over strict correctness

Design your schemas with constraints from the start — narrow enums, bounded numbers, capped string lengths. The tighter the schema, the fewer retries you'll need.
