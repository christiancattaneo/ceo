Tool Calling Design for AI Agents: Patterns That Work in Production

The model's ability to call tools is what transforms a chat completion into an agent. But the quality of your tool design determines whether that agent is reliable or constantly surprising you in production. Here's what separates tools that work from tools that fail silently.

Why Tool Design Is an Underrated Skill

Most teams spend the majority of their agent development time on prompts and orchestration. Tool design — the schemas, error handling, return formats, and contracts that govern how the model interacts with your systems — often gets treated as boilerplate. Register the function, write a one-line description, ship it.

This is a mistake. The model makes tool-calling decisions based almost entirely on the information you provide in the schema: the function name, parameter descriptions, and return value. A poorly designed schema leads to wrong tool selection, incorrect parameter construction, and misinterpretation of results. These failures are hard to debug because they look like model errors when they're actually interface errors.

Good tool design is a form of API design, with one important difference: your user is a language model, not a human developer. The conventions that make APIs intuitive for humans — RESTful conventions, HTTP status codes, pagination patterns — don't necessarily translate. You're optimizing for a different kind of comprehension.

Schema Design Principles

Name functions after their effect, not their mechanism

The function name is the first signal the model uses to decide whether to call it. Names that describe the mechanism (db_query, http_post, vector_search) force the model to reason about implementation details it shouldn't need to know. Names that describe the effect (search_knowledge_base, send_notification, get_customer_orders) let the model match the tool to the task semantically.

This matters most when you have multiple tools with overlapping capabilities. If your agent has both search_web and search_docs, the names alone communicate when to use each. If you named them serp_api_call and embedding_retrieval, the model has to infer purpose from mechanics.

Write parameter descriptions as instructions, not labels

A parameter named query with description "The search query" gives the model almost no useful information. A description that reads "The search query to send. Use specific, keyword-rich terms. Avoid full sentences — 3-7 words works best for this search engine." is actually useful.

Think of parameter descriptions as the closest thing you have to inline documentation for a model that will interpret them at inference time. Be specific about:

  • Format constraints ("ISO 8601 date string, e.g. '2026-04-12'")
  • Value constraints ("One of: 'low', 'medium', 'high'")
  • When to use optional parameters ("Only include if filtering by date range is needed")
  • What counts as a good value vs. a bad value
# Bad schema
{
  "name": "search",
  "parameters": {
    "q": {"type": "string", "description": "Query string"},
    "n": {"type": "integer", "description": "Number of results"}
  }
}

# Good schema
{
  "name": "search_knowledge_base",
  "description": "Search the internal knowledge base for relevant documents. Use this when answering questions about company policies, product documentation, or internal procedures. Do NOT use for web search.",
  "parameters": {
    "query": {
      "type": "string",
      "description": "Search terms. Use specific keywords from the user's question. Avoid stop words. Example: 'refund policy international orders' not 'what is the refund policy for orders shipped internationally'"
    },
    "max_results": {
      "type": "integer",
      "description": "Maximum number of documents to return (1-10). Default 5. Use 10 only when comprehensive coverage is important.",
      "default": 5
    }
  },
  "required": ["query"]
}

Use the description field to disambiguate between tools

When multiple tools could plausibly apply to a situation, the model needs something to differentiate them. Put that disambiguation explicitly in the tool description — not just what the tool does, but when to use it versus the alternatives.

"Search the company knowledge base for internal documentation. Use this for questions about policies, procedures, and product specs. Use search_web instead for current events, prices, or any information that might have changed recently."

This explicit disambiguation reduces wrong-tool selection, which is one of the most common agent failure modes in production.

Error Handling That Agents Can Recover From

Tools fail. APIs time out, queries return no results, permissions are denied, upstream services are down. How your tools communicate failure determines whether the agent can recover gracefully or gets stuck in a loop.

Return structured errors, not exceptions

When a tool call fails, don't raise an exception that gets converted to a generic error message. Return a structured result that describes what went wrong and what the agent should do about it:

def search_web(query: str, max_results: int = 5) -> dict:
    try:
        results = serp_api.search(query, num=max_results)
        return {
            "success": True,
            "results": results,
            "count": len(results)
        }
    except RateLimitError:
        return {
            "success": False,
            "error_type": "rate_limit",
            "message": "Search API rate limit reached. Wait 60 seconds before retrying, or try a different search approach.",
            "retry_after_seconds": 60
        }
    except NoResultsError:
        return {
            "success": False,
            "error_type": "no_results",
            "message": "No results found for this query. Try broader search terms or a different phrasing.",
            "suggestion": "broaden_query"
        }
    except Exception as e:
        return {
            "success": False,
            "error_type": "unknown",
            "message": f"Search failed: {str(e)}. Consider trying an alternative approach."
        }

Structured errors give the model actionable information. "Wait 60 seconds before retrying" is something the agent can act on. "Error: 429" is not.

Never return empty results silently

A tool that returns an empty list or null when it finds nothing is worse than a tool that returns an error. The model may interpret an empty result as "I searched and there is nothing," when the actual meaning might be "the query was too narrow" or "the category doesn't exist." Explicitly annotate empty results:

if not results:
    return {
        "success": True,
        "results": [],
        "count": 0,
        "note": "No records matched this query. Consider: (1) checking the spelling of the customer ID, (2) using a broader date range, or (3) searching by email instead of name."
    }

The silent empty result trap: An agent searching for a customer record that doesn't exist might conclude "the customer doesn't exist" when in fact the query was wrong. If your data model has a distinction between "not found" and "query error," surface it explicitly in the return value.

Idempotency and Side Effect Management

This is where tool design intersects with system safety. Agents can and do call tools multiple times — due to retries, re-planning loops, or multi-step reasoning that returns to an earlier step. If your tools have unguarded side effects, multiple calls mean multiple side effects.

Mark destructive tools clearly in their description

Tools that write, delete, or trigger external actions should have their side effects called out in the description: "IRREVERSIBLE: Sends an email to the customer. Only call this after confirming the email content with the user. Cannot be undone."

This isn't just good practice — models that have been trained to be careful about irreversible actions will attend to these signals and apply additional caution before calling.

Implement idempotency keys for write operations

Any tool that creates a record, sends a message, or initiates a transaction should accept an idempotency key. The tool handler uses the key to deduplicate calls — if the same key arrives twice, the second call returns the result of the first without executing again.

def send_email(
    to: str,
    subject: str,
    body: str,
    idempotency_key: str  # agent generates this; same key = no duplicate send
) -> dict:
    if cache.get(idempotency_key):
        return {"success": True, "status": "already_sent", "cached": True}

    result = email_client.send(to=to, subject=subject, body=body)
    cache.set(idempotency_key, result, ttl=3600)
    return {"success": True, "status": "sent", "message_id": result.id}

Separate read and write operations

Resist the temptation to build "do everything" tools. A single manage_customer tool that can read, update, and delete based on an action parameter is harder for the model to use safely than three separate tools: get_customer, update_customer, delete_customer. The separation makes the intent of each call explicit and makes it easier to add authorization checks at the tool boundary.

Output Formatting for Model Consumption

Tool results go back into the model's context. The format of that result affects how well the model can reason about it. A few patterns that consistently improve result utilization:

Return semantic labels, not raw codes

A tool that returns {"status": 2} forces the model to know your status encoding. A tool that returns {"status": "pending_review"} is self-documenting. Use human-readable values wherever possible, even if your database uses integer codes internally — translate at the tool boundary.

Include context that helps the model interpret the result

Don't just return data — return data plus the context the model needs to use it correctly:

# Instead of:
{"balance": 47.23}

# Return:
{
  "balance_usd": 47.23,
  "currency": "USD",
  "as_of": "2026-04-12T10:00:00Z",
  "note": "This is the available balance. The total balance including pending transactions is 52.18."
}

Summarize large results rather than truncating

If a tool returns more data than you want in the context (see context window management), summarize rather than truncate. A truncated result looks complete and may mislead the model. A summarized result explicitly represents the compression: "Showing 5 of 247 matching records. To see more, call again with offset=5."

The Tool Registry Pattern

As your agent's capabilities grow, managing tools becomes an engineering challenge of its own. The tool registry pattern centralizes tool definitions, makes them composable, and keeps schema and implementation co-located:

class ToolRegistry:
    def __init__(self):
        self._tools: dict[str, Tool] = {}

    def register(self, fn, schema: dict, requires_auth: bool = False):
        name = schema["name"]
        self._tools[name] = Tool(fn=fn, schema=schema, requires_auth=requires_auth)
        return fn  # allows use as decorator

    def get_schemas(self, names: list[str] | None = None) -> list[dict]:
        """Return tool schemas for injection into system prompt."""
        tools = self._tools.values() if names is None else [self._tools[n] for n in names]
        return [t.schema for t in tools]

    def execute(self, name: str, params: dict, context: AgentContext) -> dict:
        tool = self._tools.get(name)
        if not tool:
            return {"success": False, "error": f"Unknown tool: {name}"}
        if tool.requires_auth and not context.is_authenticated:
            return {"success": False, "error": "Authentication required for this tool"}
        return tool.fn(**params)

registry = ToolRegistry()

@registry.register(schema={
    "name": "get_order_status",
    "description": "...",
    "parameters": {...}
}, requires_auth=True)
def get_order_status(order_id: str) -> dict:
    ...

The registry also enables dynamic tool loading — inject only the tools relevant to the current task rather than the full set. See Context Window Management for the cost implications of large tool registries.

Common Tool Design Anti-Patterns

The God Tool

One tool that does everything via an action parameter. Hard to use safely, impossible to test well, and forces the model to reason about your internal API design.

The Silent Failure

Returns null or empty array on failure. The model interprets this as "no results" rather than "something went wrong." Always distinguish failure from emptiness.

The Ambiguous Schema

Parameter descriptions that don't specify format, valid values, or when to omit optional params. Forces the model to guess — and it will guess wrong inconsistently.

The Undifferentiated Toolkit

Multiple tools with overlapping capabilities and no disambiguation in their descriptions. Leads to random tool selection when multiple tools could apply.

The Non-Idempotent Side Effect

Write tools with no deduplication. In multi-turn sessions or on retry, the agent calls the tool again and the side effect fires twice. Customer gets two emails, order is placed twice.

The Raw Code Return

Returns HTTP status codes, integer enums, or database codes that require external context to interpret. The model has to guess what status: 3 means.

Testing Your Tool Interfaces

Tools are the most testable part of your agent stack. Unlike prompts and model outputs, tool inputs and outputs are deterministic. Build a test suite that covers:

  • Happy path — valid inputs produce the expected output shape
  • Empty results — queries that match nothing return useful guidance, not bare empty arrays
  • Error cases — each error type returns a structured response the model can act on
  • Idempotency — calling a write tool twice with the same idempotency key produces one side effect
  • Schema validity — your JSON schema is syntactically valid and passes the API's validation

These tests catch a class of agent bugs that are otherwise invisible until they hit production — the tool works fine in isolation but fails under specific conditions the agent encounters. See AI Agent Testing and Evaluation for how tool unit tests fit into the broader evaluation stack.
