Deploying LLM Applications to Production

Docker + FastAPI + Claude, API key management, Railway vs. Fly.io vs. Cloud Run trade-offs, health checks, zero-downtime deploys, and production monitoring — a complete deployment playbook.

Apr 13, 2026  ·  12 min read  ·  CEO Editorial

Getting a Claude-powered app working locally is the easy part. Deploying it reliably — with secrets managed properly, health checks that actually reflect LLM availability, zero-downtime rollouts, and observability on response latency — takes more thought. This guide covers the full path from Dockerfile to production.

The Production-Ready FastAPI App

Start with a well-structured app before packaging it. Key production requirements: async client, startup validation, graceful shutdown, and health endpoint that checks the real dependency (the Anthropic API).

# main.py
from contextlib import asynccontextmanager
from fastapi import FastAPI, HTTPException
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
from anthropic import AsyncAnthropic
import os, logging

logger = logging.getLogger("app")

# Module-level client — one instance, reused across requests
_client: AsyncAnthropic | None = None


@asynccontextmanager
async def lifespan(app: FastAPI):
    global _client
    api_key = os.environ.get("ANTHROPIC_API_KEY")
    if not api_key:
        raise RuntimeError("ANTHROPIC_API_KEY not set — refusing to start")
    _client = AsyncAnthropic(api_key=api_key)
    logger.info("Anthropic client initialized")
    yield
    # Graceful shutdown — close any open connections
    await _client.close()
    logger.info("Anthropic client closed")


app = FastAPI(title="LLM API", lifespan=lifespan)


@app.get("/health")
async def health():
    """Deep health check — verifies API key is valid."""
    try:
        # Cheapest possible real call to verify key works
        await _client.messages.create(
            model="claude-haiku-4-5-20251001",
            max_tokens=1,
            messages=[{"role": "user", "content": "hi"}],
        )
        return {"status": "ok", "anthropic": "reachable"}
    except Exception as e:
        raise HTTPException(status_code=503, detail=f"Anthropic unreachable: {e}")


class ChatRequest(BaseModel):
    message: str
    stream: bool = False


@app.post("/chat")
async def chat(req: ChatRequest):
    if req.stream:
        async def generate():
            async with _client.messages.stream(
                model="claude-sonnet-4-6", max_tokens=2048,
                messages=[{"role": "user", "content": req.message}],
            ) as s:
                async for text in s.text_stream:
                    yield f"data: {text}\n\n"
            yield "data: [DONE]\n\n"
        return StreamingResponse(generate(), media_type="text/event-stream")

    msg = await _client.messages.create(
        model="claude-sonnet-4-6", max_tokens=2048,
        messages=[{"role": "user", "content": req.message}],
    )
    return {"response": msg.content[0].text}

Dockerfile for LLM Apps

A production Dockerfile for Python LLM services — multi-stage for minimal image size, non-root user for security, explicit health check baked in:

# Dockerfile
FROM python:3.12-slim AS builder
WORKDIR /app

# Install dependencies in isolation
COPY requirements.txt .
RUN pip install --no-cache-dir --prefix=/install -r requirements.txt

# ---- Runtime stage ----
FROM python:3.12-slim
WORKDIR /app

# Non-root user — required by many cloud platforms
RUN useradd -m -u 1000 appuser

COPY --from=builder /install /usr/local
COPY . .

RUN chown -R appuser:appuser /app
USER appuser

# Health check — platform uses this for readiness
HEALTHCHECK --interval=30s --timeout=10s --start-period=15s --retries=3 \
  CMD curl -f http://localhost:8000/health || exit 1

EXPOSE 8000

# Use uvicorn with 2 workers; tune to your instance size
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000", \
     "--workers", "2", "--timeout-keep-alive", "75"]

# requirements.txt
anthropic>=0.40.0
fastapi>=0.115.0

API Key in Docker Never bake ANTHROPIC_API_KEY into your image with ENV or ARG. The key ends up in the image layer history and any registry it's pushed to. Pass it at runtime via environment variable injection from your platform's secrets manager.

LLM-Specific Recommendation For Claude-backed APIs: use Railway for MVPs (zero config), Fly.io for global latency requirements, Cloud Run for GCP-native stacks. All three support the 75-second keepalive needed for long LLM responses. Avoid platforms with hard 30-second response timeouts (Vercel Hobby, Netlify Functions) — Claude can easily exceed those.

Deploying to Railway

# railway.toml — place in project root
[build]
builder = "DOCKERFILE"
dockerfilePath = "Dockerfile"

[deploy]
startCommand = "uvicorn main:app --host 0.0.0.0 --port $PORT --workers 2"
healthcheckPath = "/health"
healthcheckTimeout = 30
restartPolicyType = "ON_FAILURE"
restartPolicyMaxRetries = 3

# CLI deploy
railway login
railway init          # link to project
railway secrets set ANTHROPIC_API_KEY=sk-ant-...
railway up            # build + deploy
railway logs          # tail logs

Deploying to Fly.io

# fly.toml
app = "my-llm-api"
primary_region = "dfw"

[build]
  dockerfile = "Dockerfile"

[http_service]
  internal_port = 8000
  force_https = true
  auto_stop_machines = false   # keep warm — avoids cold start on LLM calls
  auto_start_machines = true
  min_machines_running = 1

  [http_service.concurrency]
    type = "requests"
    hard_limit = 50
    soft_limit = 25

[[vm]]
  size = "shared-cpu-1x"
  memory = "512mb"

# CLI deploy
fly launch --no-deploy     # creates fly.toml without deploying
fly secrets set ANTHROPIC_API_KEY=sk-ant-...
fly deploy                 # build image, push to Fly registry, deploy
fly logs                   # tail logs
fly status                 # health + machine status

Deploying to Cloud Run

# Build and push to Artifact Registry
PROJECT=my-gcp-project
REGION=us-central1
IMAGE=$REGION-docker.pkg.dev/$PROJECT/llm/api:latest

docker build -t $IMAGE .
docker push $IMAGE

# Store API key in Secret Manager
echo -n "sk-ant-..." | gcloud secrets create anthropic-key --data-file=-

# Deploy with secret injected as env var
gcloud run deploy llm-api \
  --image $IMAGE \
  --region $REGION \
  --platform managed \
  --allow-unauthenticated \
  --min-instances 1 \        # keep warm for LLM latency
  --max-instances 10 \
  --timeout 300 \            # 5 min — enough for long generations
  --memory 512Mi \
  --set-secrets ANTHROPIC_API_KEY=anthropic-key:latest

Zero-Downtime Deploys

All three platforms support rolling deploys — new containers receive traffic only after passing the health check. Key settings to verify:

  • Health check path: use /health (your deep check), not /
  • Start period: give the container 15–30s before the first health probe — FastAPI + Anthropic client initialization takes a few seconds
  • Drain timeout: allow 30–60s for in-flight requests to complete before terminating the old container
  • Min instances: at least 1 always-on instance — eliminates cold start latency for the first request after a deploy

Production Monitoring Patterns

# middleware.py — structured logging + latency tracking
import time, json, logging
from fastapi import Request
from starlette.middleware.base import BaseHTTPMiddleware

logger = logging.getLogger("access")


class ObservabilityMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        start = time.monotonic()
        response = await call_next(request)
        latency_ms = (time.monotonic() - start) * 1000

        logger.info(json.dumps({
            "path": request.url.path,
            "method": request.method,
            "status": response.status_code,
            "latency_ms": round(latency_ms, 1),
        }))
        return response


# Register in main.py
app.add_middleware(ObservabilityMiddleware)
Key Metrics to Track For Claude-backed APIs: p50/p95/p99 response latency (LLM calls are bimodal — most are fast, outliers are slow), error rate by type (rate limit vs. timeout vs. API error), and token usage per request (feeds directly into cost tracking). All three platforms export logs to your observability stack of choice.

Summary

  • Structure your app for production from day one: lifespan hooks, module-level async client, deep /health endpoint
  • Dockerfile: multi-stage, non-root user, baked-in HEALTHCHECK, --timeout-keep-alive 75 for long LLM responses
  • Never bake secrets into images — inject at runtime via platform secrets
  • Railway for fastest time-to-deploy, Fly.io for global edge, Cloud Run for GCP-native
  • min-instances=1 eliminates cold starts — critical for LLM APIs where latency perception starts before the model even generates a token
  • Structured JSON logs with latency, status, and path let any observability tool (Datadog, Grafana, Axiom) give you p95 latency without custom instrumentation
