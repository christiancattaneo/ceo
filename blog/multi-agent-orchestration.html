Multi-Agent Orchestration Patterns: Architectures for Production AI Systems

A single agent hits context limits, can't parallelize work, and makes every mistake once with full authority. Multi-agent systems fix these problems — but introduce new ones. The difference between a demo and a production system is choosing the right orchestration pattern for the task and knowing where each pattern breaks.

Why Multi-Agent at All

The case for multi-agent architectures isn't theoretical. There are three concrete problems that a single-agent design can't solve well:

Context limits. A single agent processing a 50,000-token codebase or a month of emails will hit the context window ceiling, hallucinate on the oldest material, or fail to synthesize across the full corpus. Multi-agent systems solve this by partitioning work — each agent handles a focused slice, and an orchestrator synthesizes the results.

Parallelism. Sequential agents are slow for tasks that have independent sub-tasks. Researching three topics, auditing five modules, or translating a document into six languages are all embarrassingly parallel. A single agent doing these one at a time is leaving latency and cost on the table.

Specialization and error correction. A generalist agent making every decision alone has no check on its reasoning. Separate agents with distinct roles — one to generate, one to critique, one to verify — implement a form of peer review that measurably reduces error rates on complex tasks.

The tradeoff is coordination complexity. Every agent boundary is a handoff, and handoffs fail. The patterns below are how experienced teams manage those handoffs without building an unmaintainable mess.

Pattern 1: Orchestrator / Worker

The most widely used multi-agent pattern. An orchestrator agent receives the top-level task, decomposes it into sub-tasks, dispatches those sub-tasks to worker agents, and synthesizes the results. Workers are stateless from the orchestrator's perspective — they receive a task, do it, return output.

class OrchestratorAgent:
    def __init__(self, client, workers: dict[str, WorkerAgent]):
        self.client = client
        self.workers = workers

    def run(self, task: str) -> str:
        # Step 1: decompose the task
        plan = self._plan(task)          # returns list of {worker, subtask}

        # Step 2: dispatch to workers
        results = {}
        for step in plan:
            worker = self.workers[step["worker"]]
            results[step["worker"]] = worker.run(step["subtask"])

        # Step 3: synthesize
        return self._synthesize(task, results)

    def _plan(self, task: str) -> list[dict]:
        response = self.client.messages.create(
            model="claude-opus-4-6",
            max_tokens=1024,
            system="""You are a task planner. Given a task, output a JSON list of
steps. Each step has 'worker' (one of: researcher, coder, reviewer) and 'subtask'.""",
            messages=[{"role": "user", "content": task}]
        )
        return json.loads(response.content[0].text)

The orchestrator's system prompt is critical here. It needs to understand what each worker can and cannot do, produce structured output the dispatcher can parse, and handle cases where a step fails or returns partial results.

Orchestrator scope creep is a common failure mode. If the orchestrator's context accumulates every worker's full output, it runs out of context just as fast as a single agent. Pass summaries or structured results to the orchestrator, not raw full-length worker outputs. Workers should compress their output to what the orchestrator needs for synthesis.

When to use it

Orchestrator/worker is the right default for tasks that naturally decompose into typed sub-tasks with different skill requirements. Code review (researcher reads docs, coder audits implementation, reviewer checks style), content pipelines (researcher gathers facts, writer drafts, editor polishes), and data analysis workflows all fit this shape.

Pattern 2: Parallel Fan-Out

A pure parallelism pattern: one coordinator dispatches the same class of task to N agents simultaneously, then aggregates results. Unlike orchestrator/worker, there's no sequential dependency between agents — they all start at once and the coordinator waits for all to complete.

import asyncio

async def parallel_research(topics: list[str], client) -> dict[str, str]:
    """Fan out research tasks across N agents simultaneously."""
    async def research_one(topic: str) -> tuple[str, str]:
        response = await client.messages.create_async(
            model="claude-haiku-4-5-20251001",  # cheaper model for worker tasks
            max_tokens=2048,
            system="Research the given topic and return a concise summary.",
            messages=[{"role": "user", "content": f"Research: {topic}"}]
        )
        return topic, response.content[0].text

    results = await asyncio.gather(*[research_one(t) for t in topics])
    return dict(results)


async def main():
    topics = ["transformer architecture", "RLHF", "constitutional AI", "RAG systems"]
    summaries = await parallel_research(topics, client)
    # All four run simultaneously — total latency = max(individual latencies)

The latency savings are real: four 10-second research calls take 10 seconds total, not 40. The cost is the same — you're paying for four calls — but throughput is 4x.

Model selection matters for fan-out. Worker agents in a fan-out pattern don't need your most capable model. Using claude-haiku-4-5 for research sub-tasks and reserving claude-opus-4-6 for the final synthesis step can cut per-request cost by 80% while maintaining output quality where it counts.

When to use it

Fan-out works best when tasks are homogeneous (all agents do the same kind of work on different inputs), independent (no agent needs another's output), and there's a clear aggregation step. Batch translation, parallel code auditing across multiple files, and simultaneous API calls to different data sources are ideal fits.

Pattern 3: Sequential Pipeline

A chain where each agent's output becomes the next agent's input. Unlike orchestrator/worker, there's no central coordinator — the pipeline is the architecture. Each stage transforms the artifact and passes it downstream.

from dataclasses import dataclass

@dataclass
class PipelineArtifact:
    content: str
    metadata: dict
    stage: str

class AgentPipeline:
    def __init__(self, stages: list[callable]):
        self.stages = stages

    def run(self, initial_input: str) -> PipelineArtifact:
        artifact = PipelineArtifact(
            content=initial_input,
            metadata={},
            stage="input"
        )
        for stage_fn in self.stages:
            artifact = stage_fn(artifact)
            # Each stage returns a new artifact with updated content + metadata
        return artifact

# Usage: research → draft → edit → format
pipeline = AgentPipeline([
    researcher_stage,   # adds facts to artifact.metadata
    writer_stage,       # uses metadata to produce draft
    editor_stage,       # polishes prose
    formatter_stage,    # applies output format
])
result = pipeline.run("Write a technical overview of vector databases")

Pipelines are simple to reason about and easy to debug — you can inspect the artifact at each stage. The downside is that errors accumulate: a bad output from stage 2 poisons every downstream stage. Build in validation between stages for anything consequential.

When to use it

Sequential pipelines are the right choice when transformations are ordered, each stage adds domain-specific value, and the output of one stage is meaningfully the input to the next. Content creation workflows, document processing pipelines, and code generation with downstream testing all fit this shape.

Pattern 4: Critic / Debate

A generator agent produces an answer; one or more critic agents evaluate it; the generator revises based on feedback. This can run for multiple rounds until a quality threshold is met or a maximum iteration count is reached.

def critic_loop(task: str, client, max_rounds: int = 3) -> str:
    # Round 0: initial generation
    draft = generate(task, client)

    for round_num in range(max_rounds):
        critique = criticize(draft, task, client)

        # Check if the critic is satisfied
        if critique.get("satisfied"):
            break

        # Revise based on critique
        draft = revise(draft, critique["feedback"], client)

    return draft


def criticize(draft: str, original_task: str, client) -> dict:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        system="""You are a rigorous critic. Review the draft against the original task.
Return JSON: {"satisfied": bool, "feedback": str, "issues": list[str]}
Be specific. If satisfied, set satisfied=true. Don't nitpick cosmetics.""",
        messages=[{
            "role": "user",
            "content": f"Task: {original_task}\n\nDraft:\n{draft}"
        }]
    )
    return json.loads(response.content[0].text)

The critic pattern measurably improves output quality on tasks where "right" has a clear definition — code correctness, factual accuracy, logical consistency. It's less useful for subjective creative tasks where the critic and generator may loop indefinitely without convergence.

Infinite loops are a real risk. A critic with no convergence condition and a generator that can't satisfy it will loop until you hit your token budget. Always set a hard max_rounds limit, and instrument which round the loop terminated on — patterns like "always runs to max_rounds" signal the critic's bar is unreachable.

Debate variant

Instead of generator/critic, spawn two agents with opposing system prompts — one to argue for an answer, one to argue against it — and a judge agent to rule on the debate. This is effective for decisions with real ambiguity: architecture tradeoffs, risk assessments, strategy questions. The judge sees both positions and produces a more calibrated synthesis than either agent alone.

Pattern 5: Supervisor / Swarm

A supervisor agent manages a pool of worker agents that can self-delegate — workers can spawn additional workers for sub-tasks or hand off to specialized agents. This creates a tree of agents rather than a flat list.

class SupervisorAgent:
    def __init__(self, client, agent_registry: dict):
        self.client = client
        self.registry = agent_registry  # name → agent factory
        self.active_agents: list = []

    def spawn(self, agent_type: str, task: str) -> str:
        """Spawn a worker agent and return its result."""
        agent = self.registry[agent_type](self.client)
        self.active_agents.append(agent)
        return agent.run(task)

    def run(self, task: str) -> str:
        # The supervisor has tool access to spawn agents
        tools = [
            {
                "name": "spawn_agent",
                "description": "Spawn a specialized agent for a sub-task",
                "input_schema": {
                    "type": "object",
                    "properties": {
                        "agent_type": {"type": "string", "enum": list(self.registry.keys())},
                        "task": {"type": "string"}
                    },
                    "required": ["agent_type", "task"]
                }
            }
        ]
        # Supervisor uses tool calls to spawn workers as needed
        return self._run_with_tools(task, tools)

Swarm architectures are powerful for open-ended tasks where the full task graph isn't known upfront. The supervisor adapts the plan based on what each agent discovers. The cost is observability — a tree of dynamically spawned agents is much harder to trace than a static pipeline.

When to use it

Supervisor/swarm fits complex research, autonomous software engineering tasks (where a coding agent may need to spawn a test agent, which may need to spawn a debugging agent), and any task where the decomposition itself requires intelligence. Don't reach for this pattern first — it's the most complex to debug.

Routing Between Patterns: The Dispatcher

In real systems, you often need different patterns for different tasks. A dispatcher layer sits in front of your agent pool and routes incoming tasks to the appropriate pattern:

class AgentDispatcher:
    def route(self, task: str, task_metadata: dict) -> str:
        complexity = task_metadata.get("complexity", "medium")
        has_parallel_subtasks = task_metadata.get("parallel_subtasks", False)
        requires_verification = task_metadata.get("requires_verification", False)

        if complexity == "low":
            return "single_agent"
        elif has_parallel_subtasks:
            return "fan_out"
        elif requires_verification:
            return "critic_loop"
        elif complexity == "high" and task_metadata.get("open_ended"):
            return "supervisor"
        else:
            return "orchestrator_worker"

In practice, task metadata comes from a lightweight classifier — a cheap model call or a rules-based system — rather than manual annotation.

Human-in-the-Loop Integration

Production multi-agent systems almost always need human checkpoints. Not for every step, but for decisions that are irreversible, expensive, or high-stakes. The clean integration point is as a first-class tool in the orchestrator:

HITL_TOOL = {
    "name": "request_human_approval",
    "description": """Request human approval before proceeding with an action.
Use this before: deleting data, making purchases, sending external messages,
or taking any action that cannot be undone.""",
    "input_schema": {
        "type": "object",
        "properties": {
            "action_description": {"type": "string"},
            "estimated_impact": {"type": "string"},
            "alternatives": {"type": "array", "items": {"type": "string"}}
        },
        "required": ["action_description", "estimated_impact"]
    }
}

def handle_approval_request(tool_input: dict) -> str:
    """Present the approval request to a human and wait for response."""
    # In production: push to a queue, send a notification, wait for webhook
    # In dev: prompt the terminal
    print(f"\n⚠️  APPROVAL REQUIRED")
    print(f"Action: {tool_input['action_description']}")
    print(f"Impact: {tool_input['estimated_impact']}")
    response = input("Approve? [y/n]: ")
    return "approved" if response.lower() == "y" else "rejected"

The key design choice: agents request approval; they don't wait for it synchronously in a way that blocks your entire system. In production, approval requests go into a task queue, the agent suspends its current execution context (using the checkpoint patterns from the state management article), and resumes when the human responds.

Common Failure Modes

Failure mode Cause Fix
Context overflow in orchestrator Orchestrator accumulates every worker's full output Workers return summaries; orchestrator gets structured results, not raw text
Infinite critic loops No convergence condition; critic bar is unreachable Hard max_rounds limit; log termination round; calibrate critic prompts against known-good outputs
Lost context across handoffs Each worker starts cold; orchestrator forgets to pass relevant background Structured context object passed through every handoff; essential context in worker system prompt
Inconsistent state across parallel agents Two agents modify shared state simultaneously Shared state is read-only during fan-out; writes happen only in the aggregation step
Unobservable agent trees Dynamically spawned agents have no trace IDs Propagate trace IDs through every spawn call; log agent type, task, and result at every node
Cascading failures One worker failure causes orchestrator to fail with no recovery Workers return structured results with success/error flag; orchestrator handles partial results gracefully

Framework Landscape

Several frameworks have emerged to implement these patterns without building the coordination infrastructure from scratch:

  • LangGraph: Graph-based orchestration with explicit state management and human-in-the-loop support. Best for systems where the agent graph needs to be reasoned about and the state transitions matter. Steeper learning curve than alternatives.
  • AutoGen (Microsoft): Conversation-based multi-agent framework where agents communicate via message passing. Good for debate/critic patterns. The "conversational" metaphor maps naturally to LLM interactions but can be noisy for production systems.
  • CrewAI: Role-based orchestration with an opinionated agent/task/crew abstraction. Easy to get started; less control over low-level coordination for complex production scenarios.
  • Claude Agents SDK (Anthropic): Native support for orchestrator/worker and tool-based agent patterns with built-in prompt caching and structured tool use. Minimal abstraction — you write the coordination logic, the SDK handles the API plumbing.

Start without a framework. The patterns in this article can be implemented directly against the Claude API in a few hundred lines. Frameworks add dependencies, opinionated abstractions, and upgrade friction. Once you know which pattern you need and have working code, adding a framework is optional — not the default. Most production teams that reach for a framework too early end up fighting it later.

Choosing the Right Pattern

The honest answer is that most tasks fit one of three patterns: single agent (simpler than you think), orchestrator/worker (the majority of complex tasks), or fan-out (anything embarrassingly parallel). Critic loops and supervisor/swarm are valuable but niche — reach for them when the simpler patterns demonstrably don't work, not as a default.

The throughline in all of these patterns is that the coordination layer — the orchestrator, the pipeline, the supervisor — must be more reliable than the agents it coordinates. It needs to handle partial failures gracefully, pass context deliberately, and have hard limits on loops and recursion. An orchestrator that crashes silently is worse than no orchestrator at all.
