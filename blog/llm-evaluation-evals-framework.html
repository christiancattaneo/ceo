LLM Evaluation: Building an Eval Framework for Production AI Agents

Unit tests can't tell you if your AI agent got worse. An LLM call that returns HTTP 200 with well-formed JSON can still produce an answer that's factually wrong, tonally off, or inconsistent with the context from three turns earlier. Catching that requires a different kind of testing infrastructure — one built around evals, not assertions.

Why Traditional Testing Breaks Down for LLMs

Software tests work because the output space is discrete and deterministic. You assert that add(2, 3) returns 5. LLM outputs are neither. The same prompt, same model, same temperature can produce meaningfully different responses on successive calls. And "meaningfully different" is itself a judgment that requires understanding the domain.

The failure modes that matter in production aren't exceptions — they're quality regressions. A prompt change that makes your agent 10% more verbose while slightly reducing factual accuracy won't throw any errors. Your CI pipeline will stay green. Your users will just start getting worse answers.

Evals are the testing infrastructure that addresses this. They measure quality against defined criteria, run at scale against representative inputs, and give you a score you can track across prompt versions, model upgrades, and feature changes.

The expensive mistake: Teams often wait until they have a human annotation team before building evals. Don't. Start with automated evals — even imperfect ones — from day one. Imperfect evals you run on every PR are more valuable than perfect evals you run once a quarter.

The Four Eval Strategies

1. Reference-Based Evals

The simplest form: compare the model output to a ground-truth answer. Works well for extraction tasks (pull the patient name from this form), classification (is this message toxic?), and structured outputs where correctness is binary or near-binary.

from dataclasses import dataclass
from typing import Optional
import re


@dataclass
class EvalCase:
    input: str
    expected: str
    metadata: dict = None


def exact_match(output: str, expected: str) -> float:
    return 1.0 if output.strip() == expected.strip() else 0.0


def token_f1(output: str, expected: str) -> float:
    """F1 over tokens — good for extraction tasks."""
    out_tokens = set(output.lower().split())
    exp_tokens = set(expected.lower().split())
    if not exp_tokens:
        return 1.0 if not out_tokens else 0.0
    precision = len(out_tokens & exp_tokens) / len(out_tokens) if out_tokens else 0.0
    recall = len(out_tokens & exp_tokens) / len(exp_tokens)
    if precision + recall == 0:
        return 0.0
    return 2 * precision * recall / (precision + recall)


def contains_expected(output: str, expected: str) -> float:
    return 1.0 if expected.lower() in output.lower() else 0.0

Reference-based evals break down when there are many correct phrasings for the same answer. Token F1 handles this better than exact match, but both fail on open-ended generation where no reference is available.

2. Rule-Based Evals

Regex, keyword presence, structural checks, and format validation. Fast, cheap, and 100% deterministic — run them first. They catch the obvious failures before you spend budget on LLM-as-judge.

@dataclass
class RuleEvalResult:
    passed: bool
    score: float
    reason: str


def eval_json_valid(output: str) -> RuleEvalResult:
    import json
    try:
        json.loads(output)
        return RuleEvalResult(True, 1.0, "valid JSON")
    except json.JSONDecodeError as e:
        return RuleEvalResult(False, 0.0, f"invalid JSON: {e}")


def eval_no_pii_leaked(output: str) -> RuleEvalResult:
    patterns = [
        (r'\b\d{3}-\d{2}-\d{4}\b', "SSN pattern"),
        (r'\b4[0-9]{12}(?:[0-9]{3})?\b', "Visa card pattern"),
        (r'\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b', "email address"),
    ]
    for pattern, label in patterns:
        if re.search(pattern, output):
            return RuleEvalResult(False, 0.0, f"PII detected: {label}")
    return RuleEvalResult(True, 1.0, "no PII patterns found")


def eval_length_range(output: str, min_words: int, max_words: int) -> RuleEvalResult:
    count = len(output.split())
    if count < min_words:
        return RuleEvalResult(False, count / min_words, f"too short: {count} words")
    if count > max_words:
        return RuleEvalResult(False, max_words / count, f"too long: {count} words")
    return RuleEvalResult(True, 1.0, f"{count} words — in range")

3. LLM-as-Judge

Use a capable model (typically a stronger or different model than the one being evaluated) to score outputs against a rubric. Expensive, but the only scalable approach for open-ended quality dimensions like helpfulness, accuracy, coherence, and tone.

import anthropic
from dataclasses import dataclass


@dataclass
class JudgeScore:
    score: float          # 0.0 – 1.0
    reasoning: str
    dimension: str


JUDGE_PROMPT = """You are an evaluation judge for an AI assistant. Score the following response.

USER QUERY: {query}

ASSISTANT RESPONSE: {response}

EVALUATION DIMENSION: {dimension}
RUBRIC: {rubric}

Respond with JSON only:
{{
  "score": ,
  "reasoning": ""
}}"""


def llm_judge(
    client: anthropic.Anthropic,
    query: str,
    response: str,
    dimension: str,
    rubric: str,
    judge_model: str = "claude-opus-4-6",
) -> JudgeScore:
    import json

    prompt = JUDGE_PROMPT.format(
        query=query,
        response=response,
        dimension=dimension,
        rubric=rubric,
    )
    result = client.messages.create(
        model=judge_model,
        max_tokens=256,
        messages=[{"role": "user", "content": prompt}],
    )
    try:
        data = json.loads(result.content[0].text)
        return JudgeScore(
            score=float(data["score"]),
            reasoning=data["reasoning"],
            dimension=dimension,
        )
    except (json.JSONDecodeError, KeyError, ValueError):
        return JudgeScore(score=0.5, reasoning="parse error", dimension=dimension)


DIMENSIONS = {
    "factual_accuracy": "1.0 = all factual claims are correct. 0.0 = contains confident factual errors.",
    "helpfulness": "1.0 = directly answers the question with actionable detail. 0.0 = vague, refuses unnecessarily, or misses the point.",
    "tone": "1.0 = professional and appropriate for the context. 0.0 = inappropriate tone for the audience.",
    "conciseness": "1.0 = no unnecessary padding or repetition. 0.0 = significantly longer than necessary.",
}

Judge bias: LLM judges prefer verbose, confident-sounding answers and systematically favor their own model family's style. Calibrate with human labels before treating judge scores as absolute. Use them for relative comparisons (prompt A vs. prompt B) more than for absolute thresholds.

4. Reference-Free Semantic Evals

Embedding-based similarity, entailment models, and task-specific classifiers. Faster than LLM-as-judge and more robust than string matching. Useful as a middle layer — run after rules, before the expensive judge pass.

import numpy as np


def cosine_similarity(a: list[float], b: list[float]) -> float:
    a_arr, b_arr = np.array(a), np.array(b)
    denom = np.linalg.norm(a_arr) * np.linalg.norm(b_arr)
    return float(np.dot(a_arr, b_arr) / denom) if denom > 0 else 0.0

Building a Regression Suite

A regression suite is a curated set of eval cases that runs on every prompt change or model upgrade. Its job isn't comprehensive coverage — it's catching regressions on the cases that have historically been tricky.

The most valuable eval cases come from production failures. When a user complains, or when you spot a bad output in your logs, that case goes into the regression suite. Over time you accumulate a suite that captures your actual failure distribution rather than hypothetical ones.

import json
import time
from pathlib import Path
from typing import Callable
from dataclasses import dataclass, field, asdict


@dataclass
class EvalResult:
    case_id: str
    input: str
    output: str
    scores: dict[str, float]
    passed: bool
    latency_ms: float
    model: str
    timestamp: str = field(default_factory=lambda: time.strftime("%Y-%m-%dT%H:%M:%SZ"))


class RegressionSuite:
    def __init__(self, suite_path: str, pass_threshold: float = 0.75):
        self.suite_path = Path(suite_path)
        self.pass_threshold = pass_threshold
        self.cases: list[dict] = json.loads(self.suite_path.read_text())

    def run(
        self,
        agent_fn: Callable[[str], str],
        eval_fns: list[Callable],
        model: str,
    ) -> dict:
        results = []
        for case in self.cases:
            t0 = time.time()
            output = agent_fn(case["input"])
            latency = (time.time() - t0) * 1000

            scores = {}
            for fn in eval_fns:
                result = fn(output=output, expected=case.get("expected", ""))
                scores[fn.__name__] = result.score if hasattr(result, "score") else result

            overall = sum(scores.values()) / len(scores) if scores else 0.0
            results.append(EvalResult(
                case_id=case["id"],
                input=case["input"],
                output=output,
                scores=scores,
                passed=overall >= self.pass_threshold,
                latency_ms=latency,
                model=model,
            ))

        total = len(results)
        passed = sum(1 for r in results if r.passed)
        avg_score = sum(
            sum(r.scores.values()) / len(r.scores) for r in results if r.scores
        ) / total if total else 0.0

        return {
            "total": total,
            "passed": passed,
            "failed": total - passed,
            "pass_rate": passed / total if total else 0.0,
            "avg_score": avg_score,
            "results": [asdict(r) for r in results],
        }

RAGAS for RAG Pipelines

If your agent uses retrieval, you need evals specifically for the retrieval-augmented generation loop. The RAGAS framework defines four metrics that cover each stage of the RAG pipeline independently.

Metric What It Measures Requires Reference? Failure Signal
Faithfulness Does the answer contain only claims that are grounded in the retrieved context? No Hallucination — model adds facts not in the retrieved chunks
Answer Relevance Does the answer actually address the question asked? No Off-topic answers, irrelevant retrieved context being parroted
Context Precision What fraction of the retrieved chunks are actually relevant? Yes (ground truth) Retrieval noise — pulling irrelevant chunks that dilute the context
Context Recall Did retrieval surface the chunks needed to answer the question? Yes (ground truth) Retrieval gaps — relevant chunks not making it to the context window 
@dataclass
class RAGEvalCase:
    question: str
    retrieved_contexts: list[str]
    generated_answer: str
    ground_truth: str | None = None


def eval_faithfulness(
    client: anthropic.Anthropic,
    case: RAGEvalCase,
    judge_model: str = "claude-sonnet-4-6",
) -> float:
    """
    Breaks the answer into claims, checks each against retrieved context.
    Returns fraction of claims that are grounded.
    """
    context_block = "\n---\n".join(case.retrieved_contexts)
    prompt = f"""Given these retrieved documents:
{context_block}

And this generated answer:
{case.generated_answer}

List each factual claim in the answer. For each claim, output:
{{"claim": "...", "grounded": true/false, "evidence": "quote from context or null"}}

Return a JSON array of these objects."""

    result = client.messages.create(
        model=judge_model,
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}],
    )
    import json
    try:
        claims = json.loads(result.content[0].text)
        if not claims:
            return 1.0
        grounded = sum(1 for c in claims if c.get("grounded", False))
        return grounded / len(claims)
    except (json.JSONDecodeError, TypeError):
        return 0.5  # uncertain


def eval_answer_relevance(
    client: anthropic.Anthropic,
    case: RAGEvalCase,
    n_questions: int = 3,
) -> float:
    """
    Generate questions the answer would be a good response to.
    Score = average semantic similarity to the original question.
    """
    prompt = f"""Given this answer: {case.generated_answer}

Generate {n_questions} questions that this answer would directly address.
Return as a JSON array of strings."""

    result = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=256,
        messages=[{"role": "user", "content": prompt}],
    )
    import json
    try:
        generated_qs = json.loads(result.content[0].text)
        # Compute embedding similarity to original question
        # (substitute your embedding call)
        similarities = [
            embedding_similarity(case.question, q) for q in generated_qs
        ]
        return sum(similarities) / len(similarities) if similarities else 0.0
    except (json.JSONDecodeError, TypeError):
        return 0.5

Integrating Evals into CI

Evals are only useful if they run automatically. The pattern that works: a small fast suite on every PR (rule-based + cheap semantic checks, <30 seconds), and a deeper judge-based suite that runs nightly or on main-branch merges.

#!/usr/bin/env python3
"""
eval_ci.py — lightweight eval gate for CI.
Fails with exit code 1 if pass rate drops below threshold.
"""
import sys
import json
import argparse
from pathlib import Path


def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--suite", required=True)
    parser.add_argument("--agent-module", required=True)
    parser.add_argument("--threshold", type=float, default=0.80)
    parser.add_argument("--baseline", help="path to baseline results JSON")
    parser.add_argument("--max-regression", type=float, default=0.05,
                        help="max allowed score drop vs baseline")
    args = parser.parse_args()

    # Import agent dynamically
    import importlib.util
    spec = importlib.util.spec_from_file_location("agent", args.agent_module)
    agent_module = importlib.util.module_from_spec(spec)
    spec.loader.exec_module(agent_module)

    suite = RegressionSuite(args.suite, pass_threshold=args.threshold)
    report = suite.run(
        agent_fn=agent_module.run,
        eval_fns=[eval_json_valid, eval_no_pii_leaked],
        model=agent_module.MODEL,
    )

    print(json.dumps({
        "pass_rate": report["pass_rate"],
        "avg_score": report["avg_score"],
        "passed": report["passed"],
        "total": report["total"],
    }, indent=2))

    # Absolute threshold check
    if report["pass_rate"] < args.threshold:
        print(f"FAIL: pass rate {report['pass_rate']:.2%} below threshold {args.threshold:.2%}")
        sys.exit(1)

    # Regression check against baseline
    if args.baseline and Path(args.baseline).exists():
        baseline = json.loads(Path(args.baseline).read_text())
        drop = baseline["avg_score"] - report["avg_score"]
        if drop > args.max_regression:
            print(f"FAIL: avg score dropped {drop:.3f} vs baseline (max allowed: {args.max_regression})")
            sys.exit(1)

    print("PASS")
    sys.exit(0)


if __name__ == "__main__":
    main()

The baseline pattern: Commit a eval_baseline.json alongside your code. On each CI run, compare the new results against the baseline. If avg_score drops more than 5%, the build fails. This catches silent regressions that don't break the absolute threshold.

What to Actually Measure

Teams overthink eval dimensions at first, then settle on a small set that covers the failure modes that actually matter for their use case. The useful starting set for most production agents:

Dimension Eval Method When It Matters
Correctness / Accuracy Reference match or LLM judge Any agent that returns facts, answers questions, or makes recommendations
Format compliance Rule-based (JSON schema, regex) Agents with structured output contracts — always run first
Safety / PII Rule-based + classifier Any agent handling user data or generating public-facing content
Faithfulness LLM judge (RAGAS-style) RAG pipelines — did the model stay grounded in retrieved context?
Helpfulness LLM judge Conversational agents — is the response actually useful?
Latency Timing (p50, p95) All agents — track alongside quality to catch the quality/speed tradeoff
Cost per call Token counting + pricing table All agents — prompt changes that improve quality but double cost need visibility

Managing the Eval Dataset

The eval dataset is a first-class engineering artifact. Version it in source control, review additions in pull requests, and treat contamination (eval cases leaking into training or few-shot examples) as a critical bug.

Practical dataset management principles:

  • Source production failures first — when something breaks in prod, the fix includes adding the failing case to the eval suite
  • Keep cases independent — each case should be evaluable standalone, no dependencies on case ordering or shared state
  • Tag cases by failure mode — add metadata like "category": "edge_case", "source": "user_complaint_2026-03" so you can slice results and understand where regressions are concentrated
  • Refresh periodically — remove cases that haven't failed in 12+ months; they're not earning their eval budget
  • Balance the suite — if 80% of your cases are one type, a regression in another type won't move the aggregate score enough to trigger an alert

Eval Tooling Landscape

The ecosystem has matured quickly. Most teams don't build the whole stack from scratch:

  • Braintrust — end-to-end eval platform with dataset management, LLM-as-judge, and CI integration; the closest to a complete solution for production teams
  • LangSmith — eval + tracing together if you're in the LangChain ecosystem; good for teams that want one tool for both
  • RAGAS — open-source library specifically for RAG eval metrics; easy to integrate into any pipeline
  • Promptfoo — CLI-first, good for prompt regression testing and red-teaming; low setup cost
  • Inspect AI — UK AI Safety Institute's open-source eval framework; rigorous, academic-leaning, good for capability evals
  • Custom + Anthropic SDK — for teams with specific rubrics or proprietary domains; worth it when off-the-shelf rubrics don't fit your use case well

Start cheap, scale up: Begin with rule-based evals and a small LLM judge suite using claude-haiku-4-5-20251001 for the judge. Once your suite is stable and your eval cases are representative, upgrade the judge model for higher-stakes evals. You can run a surprisingly effective eval pipeline for <$5/day at this scale.

The Eval Flywheel

The teams with the best AI quality aren't the ones who built the most sophisticated eval infrastructure on day one. They built a basic eval suite, ran it consistently, and used the results to guide every prompt change and model upgrade. That creates a compounding advantage: each production incident funds another eval case, which catches the next regression before it ships.

The flywheel: instrument → catch regressions → add cases → instrument better. The teams that skip evals break that loop and end up doing incident response instead of systematic improvement.
