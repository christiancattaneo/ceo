Streaming LLM Responses: Server-Sent Events with FastAPI and Claude

A complete implementation of token-by-token streaming — Claude's streaming API, FastAPI SSE endpoints, JavaScript client consumption, error recovery mid-stream, streaming with tool use, and backpressure handling for slow clients.

Apr 14, 2026  ·  13 min read  ·  CEO Editorial

Streaming transforms the user experience of LLM applications. Instead of a blank screen for 5–10 seconds followed by a wall of text, users see tokens appearing in real time — the same feel as ChatGPT and Claude.ai. The implementation is straightforward once you understand the event loop, but there are several production pitfalls that only become apparent under load.

This guide covers the full stack: Claude's streaming API, a FastAPI SSE endpoint, a JavaScript client, error recovery, and streaming with tool use.

1. Claude's Streaming API

The Anthropic Python SDK supports streaming via a context manager. Instead of waiting for the full response, you iterate over a stream of typed events — text_delta for content chunks, message_delta for usage and stop reason, and content_block_start/stop for structure.

import anthropic

client = anthropic.Anthropic()

def stream_to_stdout(prompt: str) -> str:
    """Basic streaming — prints tokens as they arrive, returns full text."""
    full_text = ""

    with client.messages.stream(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    ) as stream:
        for text in stream.text_stream:
            print(text, end="", flush=True)
            full_text += text

    print()  # newline after stream
    return full_text

# Access the final message (includes usage stats) after the stream
def stream_with_usage(prompt: str) -> tuple[str, dict]:
    with client.messages.stream(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    ) as stream:
        full_text = stream.get_final_text()
        final_msg = stream.get_final_message()

    usage = {
        "input_tokens": final_msg.usage.input_tokens,
        "output_tokens": final_msg.usage.output_tokens,
    }
    return full_text, usage

2. FastAPI SSE Endpoint

Server-Sent Events (SSE) is the right transport for streaming LLM output to browsers. It's simpler than WebSockets (unidirectional, HTTP-native), works through proxies, and has native browser support via EventSource. FastAPI's StreamingResponse handles the wire format.

from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse
from pydantic import BaseModel
import asyncio, json, anthropic

app = FastAPI()
client = anthropic.Anthropic()

class ChatRequest(BaseModel):
    message: str
    conversation_id: str = ""

async def sse_generator(prompt: str, request: Request):
    """Yields SSE-formatted chunks from Claude's stream."""
    try:
        with client.messages.stream(
            model="claude-sonnet-4-6",
            max_tokens=2048,
            messages=[{"role": "user", "content": prompt}]
        ) as stream:
            for text in stream.text_stream:
                # Check if client disconnected between chunks
                if await request.is_disconnected():
                    break

                # SSE format: "data: {json}\n\n"
                payload = json.dumps({"type": "delta", "text": text})
                yield f"data: {payload}\n\n"

                # Yield control to event loop between chunks
                await asyncio.sleep(0)

        # Send done event with usage
        final = stream.get_final_message()
        done_payload = json.dumps({
            "type": "done",
            "input_tokens": final.usage.input_tokens,
            "output_tokens": final.usage.output_tokens,
        })
        yield f"data: {done_payload}\n\n"

    except Exception as e:
        error_payload = json.dumps({"type": "error", "message": str(e)})
        yield f"data: {error_payload}\n\n"

@app.post("/chat/stream")
async def chat_stream(body: ChatRequest, request: Request):
    return StreamingResponse(
        sse_generator(body.message, request),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",  # disable nginx buffering
        }
    )
X-Accel-Buffering: no Without this header, nginx (and many reverse proxies) buffer the SSE stream and deliver it in chunks — destroying the real-time effect. Always set X-Accel-Buffering: no on SSE responses. The equivalent for Caddy is flush_interval -1 in your Caddyfile.

3. JavaScript Client

The browser's native EventSource API handles reconnection automatically but only supports GET requests. For POST (needed to send message payloads), use fetch with a ReadableStream reader instead.

// JavaScript SSE client for POST endpoints
async function streamChat(message, onChunk, onDone, onError) {
  const response = await fetch("/chat/stream", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ message }),
  });

  if (!response.ok) {
    onError(new Error(`HTTP ${response.status}`));
    return;
  }

  const reader = response.body.getReader();
  const decoder = new TextDecoder();
  let buffer = "";

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    buffer += decoder.decode(value, { stream: true });

    // SSE lines end with \n\n — process complete events
    const events = buffer.split("\n\n");
    buffer = events.pop(); // last item may be incomplete

    for (const event of events) {
      if (!event.startsWith("data: ")) continue;
      const data = JSON.parse(event.slice(6));

      if (data.type === "delta") onChunk(data.text);
      else if (data.type === "done") onDone(data);
      else if (data.type === "error") onError(new Error(data.message));
    }
  }
}

// Usage with a chat textarea
streamChat(
  userMessage,
  (chunk) => { outputEl.textContent += chunk; },
  (stats) => { console.log(`Tokens: ${stats.output_tokens}`); },
  (err) => { console.error("Stream error:", err); }
);

4. Error Recovery Mid-Stream

The stream can fail mid-response — a network blip, a rate limit hit, or a server restart. A robust client retries from where it left off using a cursor (character offset) sent with each chunk.

# Server: include cursor in each delta event
async def resumable_sse_generator(prompt: str, start_offset: int = 0):
    cursor = 0
    full_text = ""

    with client.messages.stream(
        model="claude-sonnet-4-6",
        max_tokens=2048,
        messages=[{"role": "user", "content": prompt}]
    ) as stream:
        for text in stream.text_stream:
            full_text += text
            cursor = len(full_text)

            # Skip chunks the client already received
            if cursor <= start_offset:
                continue

            payload = json.dumps({
                "type": "delta",
                "text": text,
                "cursor": cursor
            })
            yield f"data: {payload}\n\n"
            await asyncio.sleep(0)

@app.post("/chat/stream")
async def resumable_stream(body: ChatRequest, offset: int = 0):
    return StreamingResponse(
        resumable_sse_generator(body.message, start_offset=offset),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"}
    )
True resumability requires server-side storage The resumable pattern above only works if the same server instance handles the retry (no load balancer). For true resumability across restarts, store the accumulated text in Redis keyed by conversation_id, and replay from the stored cursor on reconnect. This is overkill for most apps — simple client-side retry with visual feedback ("Reconnecting…") handles the common case.

5. Streaming with Tool Use

Streaming and tool use interact in a specific way: tool call inputs stream as JSON fragments (not yet parseable), then a tool_use block completes, then the response pauses while you execute the tool, then streaming resumes with the tool result incorporated.

from anthropic.types import ToolUseBlock
import json

async def streaming_tool_loop(prompt: str):
    messages = [{"role": "user", "content": prompt}]

    while True:
        tool_calls = []
        text_chunks = []

        with client.messages.stream(
            model="claude-sonnet-4-6",
            max_tokens=2048,
            tools=TOOLS,
            messages=messages,
        ) as stream:
            # Stream text tokens to the client in real time
            for event in stream:
                if event.type == "content_block_delta":
                    if event.delta.type == "text_delta":
                        text = event.delta.text
                        text_chunks.append(text)
                        yield f"data: {json.dumps({'type': 'delta', 'text': text})}\n\n"

            final = stream.get_final_message()

        # Check if we need to execute tools
        tool_calls = [b for b in final.content if isinstance(b, ToolUseBlock)]

        if not tool_calls:
            break  # no tools → done

        # Signal tool execution to client
        yield f"data: {json.dumps({'type': 'tool_start', 'tools': [t.name for t in tool_calls]})}\n\n"

        messages.append({"role": "assistant", "content": final.content})
        tool_results = []
        for tc in tool_calls:
            result = execute_tool(tc.name, tc.input)
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tc.id,
                "content": result
            })
        messages.append({"role": "user", "content": tool_results})

        # Signal tool complete; next loop iteration streams the follow-up response
        yield f"data: {json.dumps({'type': 'tool_done'})}\n\n"

    yield f"data: {json.dumps({'type': 'done'})}\n\n"

6. Backpressure and Slow Clients

If a client reads slowly — a mobile device on a poor connection — the server can accumulate an unbounded buffer of unsent SSE chunks. Bound this with a queue and a timeout.

import asyncio

async def backpressure_generator(prompt: str, request: Request):
    queue: asyncio.Queue[str | None] = asyncio.Queue(maxsize=50)

    async def producer():
        """Run the Claude stream and push chunks to the queue."""
        with client.messages.stream(
            model="claude-sonnet-4-6",
            max_tokens=2048,
            messages=[{"role": "user", "content": prompt}]
        ) as stream:
            for text in stream.text_stream:
                try:
                    # Block if queue is full — applies backpressure on the producer
                    await asyncio.wait_for(queue.put(text), timeout=10.0)
                except asyncio.TimeoutError:
                    break  # client too slow; abandon stream
        await queue.put(None)  # sentinel

    asyncio.create_task(producer())

    while True:
        if await request.is_disconnected():
            break
        chunk = await asyncio.wait_for(queue.get(), timeout=30.0)
        if chunk is None:
            break
        yield f"data: {json.dumps({'type': 'delta', 'text': chunk})}\n\n"

Production Checklist

  • SSE responses set Cache-Control: no-cache and X-Accel-Buffering: no
  • Client checks request.is_disconnected() between chunks to abort orphaned streams
  • Error events use a distinct type: "error" payload — never silently drop errors
  • Stream generator yields await asyncio.sleep(0) between chunks to release the event loop
  • Tool execution during streaming signals the client with tool_start / tool_done events
  • Queue-based backpressure limits memory growth when clients read slowly
  • Usage stats (input_tokens, output_tokens) sent in the done event for cost tracking
  • JavaScript client buffers partial SSE lines — splits on \n\n, not on individual chunks
  • Load balancer / reverse proxy configured for long-lived connections (timeout ≥ 120s)
  • Integration test verifies full stream delivery end-to-end, not just HTTP 200
