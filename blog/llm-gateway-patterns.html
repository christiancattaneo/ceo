LLM Gateway Patterns: Rate Limiting, Caching, and Model Routing for Production AI

Every team building production AI eventually builds the same infrastructure layer: something that sits between their application and the model APIs, handling rate limits, caching repeated calls, routing to cheaper models when possible, and giving ops teams visibility into what's actually happening. That layer is the LLM gateway. Here's how to build it right.

Why a Gateway Layer?

When you start building with LLMs, you call the API directly from your application code. That works fine for prototypes. It breaks down in production for several reasons:

  • Rate limits hit at the worst time. Anthropic and OpenAI both impose per-minute and per-day rate limits. Without centralized rate limit management, a traffic spike silently degrades all users simultaneously instead of gracefully queuing.
  • Costs are invisible until the bill arrives. Direct API calls provide no real-time cost visibility. Teams routinely discover they're spending 10× their estimate only after seeing a monthly invoice.
  • Identical prompts get re-executed. Many agent workflows send the same or highly similar prompts repeatedly — the same system prompt, the same tool schema injections, the same knowledge base documents. Without a caching layer, you pay for each execution.
  • Model selection is hardcoded. When you need to switch models for cost or performance reasons, or when a model API is down, changing hardcoded model names across a codebase is slow and error-prone.
  • Observability is an afterthought. Debugging production issues without centralized logging is painful. What prompt triggered the bad response? How often does this tool get called? What's the p95 latency for this endpoint?

A gateway layer solves all of these. It's a service (or middleware layer) that wraps your LLM calls and handles the cross-cutting concerns your application code shouldn't have to manage.

Pattern 1: Rate Limiting and Backpressure

Rate limiting at the gateway level lets you manage API quota centrally rather than fighting it piecemeal in application code. The key design decisions:

Token bucket vs. sliding window

For LLM rate limiting, a token bucket approach tends to work better than a fixed-window counter. Token bucket allows short bursts above the average rate (absorbing spikes) while enforcing an average rate over time — which matches how LLM APIs actually work. A fixed window that resets every minute leads to thundering herd behavior at reset time.

import time
from threading import Lock

class TokenBucket:
    def __init__(self, capacity: int, refill_rate: float):
        """
        capacity: max tokens in bucket (burst allowance)
        refill_rate: tokens added per second
        """
        self.capacity = capacity
        self.tokens = capacity
        self.refill_rate = refill_rate
        self.last_refill = time.monotonic()
        self._lock = Lock()

    def consume(self, tokens: int = 1) -> bool:
        with self._lock:
            now = time.monotonic()
            elapsed = now - self.last_refill
            self.tokens = min(
                self.capacity,
                self.tokens + elapsed * self.refill_rate
            )
            self.last_refill = now

            if self.tokens >= tokens:
                self.tokens -= tokens
                return True
            return False  # rate limited

class LLMGateway:
    def __init__(self):
        # 100k TPM limit → ~1,667 tokens/second
        self.rate_limiter = TokenBucket(capacity=10_000, refill_rate=1_667)

    async def complete(self, messages, estimated_tokens: int = 1000, **kwargs):
        if not self.rate_limiter.consume(estimated_tokens):
            raise RateLimitError("Gateway rate limit exceeded — retry after backoff")
        return await self._call_api(messages, **kwargs)

Per-tenant rate limiting

If you're building a multi-tenant system, you need per-tenant limits in addition to global limits. A single tenant running a batch job shouldn't be able to starve interactive users. Maintain a separate bucket per tenant ID, with a hierarchy: per-tenant limits enforce fairness, global limits protect your API quota.

Pattern 2: Semantic and Exact Caching

Caching is typically the highest-ROI optimization available to an LLM gateway. There are two levels:

Exact cache (deterministic prompts)

When the exact same prompt is sent multiple times, serve the cached response without hitting the API. This is most valuable for:

  • Classification or extraction endpoints where many inputs map to the same prompt template
  • FAQ or knowledge base queries where the same questions recur
  • Any agent step where the prompt is deterministic given the same inputs
import hashlib
import json
from functools import lru_cache

class CachedGateway:
    def __init__(self, cache_client, ttl_seconds=3600):
        self.cache = cache_client  # Redis, Memcached, etc.
        self.ttl = ttl_seconds

    def _cache_key(self, messages: list, model: str, **kwargs) -> str:
        payload = {"messages": messages, "model": model, **kwargs}
        return "llm:" + hashlib.sha256(
            json.dumps(payload, sort_keys=True).encode()
        ).hexdigest()[:16]

    async def complete(self, messages, model="claude-sonnet-4-6", **kwargs):
        key = self._cache_key(messages, model, **kwargs)

        cached = await self.cache.get(key)
        if cached:
            return json.loads(cached)  # cache hit — no API call

        result = await self._call_api(messages, model=model, **kwargs)
        await self.cache.setex(key, self.ttl, json.dumps(result))
        return result

Semantic cache (similar prompts)

Exact caching requires identical inputs. Semantic caching uses embeddings to find responses to semantically similar queries. A question about "how do I reset my password" and "I forgot my login password" should return the same cached answer.

The implementation: embed the query, search a vector store for similar cached queries above a similarity threshold, return the cached response if found. This is more complex and introduces false positive risk (returning a cached response for a query that's similar but not actually the same). Set the similarity threshold conservatively (0.95+) to start.

Don't cache agent responses that involve real-time data (current stock prices, live customer records, today's news) or any response where freshness matters. Build cache key namespacing so you can selectively disable caching per endpoint type. The most common caching mistake is caching too aggressively and serving stale data for queries that needed a fresh answer.

Pattern 3: Model Routing

Model routing — sending different requests to different models based on cost, capability, or load requirements — is one of the most powerful things a gateway layer enables. The routing logic can be as simple or sophisticated as your use case demands.

Complexity-based routing

Most production workloads have a bimodal distribution: many simple requests (classification, extraction, short Q&A) and fewer complex ones (long-form generation, multi-step reasoning, code synthesis). Routing simple requests to a cheaper, faster model and complex ones to a more capable model can cut costs by 50-80% with no perceptible quality degradation.

class ModelRouter:
    FAST_MODEL = "claude-haiku-4-5"      # cheap, fast
    CAPABLE_MODEL = "claude-sonnet-4-6"  # more capable, moderate cost
    EXPERT_MODEL = "claude-opus-4-6"     # most capable, expensive

    def select_model(self, task_type: str, context_tokens: int,
                     requires_reasoning: bool) -> str:
        # Very long contexts benefit from larger models
        if context_tokens > 50_000:
            return self.CAPABLE_MODEL

        # Tasks requiring complex reasoning
        if requires_reasoning or task_type in {"code_synthesis", "strategic_analysis"}:
            return self.CAPABLE_MODEL

        # Simple classification, extraction, short answers
        if task_type in {"classify", "extract", "yes_no", "short_answer"}:
            return self.FAST_MODEL

        return self.CAPABLE_MODEL  # default

Fallback routing

When your primary model API is unavailable or rate-limited, a fallback route to a secondary provider keeps the system available. This requires standardizing on a common request/response format across providers — or using a library that abstracts provider differences.

async def complete_with_fallback(self, messages, **kwargs):
    providers = [
        ("anthropic", "claude-sonnet-4-6"),
        ("openai", "gpt-4o"),          # fallback
    ]
    last_error = None
    for provider, model in providers:
        try:
            return await self._call(provider, model, messages, **kwargs)
        except (RateLimitError, ServiceUnavailableError) as e:
            last_error = e
            continue
    raise last_error

A/B routing for model evaluation

A gateway is the natural place to implement A/B testing between models. Route X% of traffic to model A, the remainder to model B. Log both responses along with their latency and cost. Evaluate quality offline using LLM-as-judge or human review. This is far better than switching models wholesale and hoping quality holds.

Pattern 4: Cost Controls

Hard cost limits prevent runaway spend from infinite loops, malformed prompts that generate massive completions, or unexpected traffic spikes.

Per-request token budget

Always set max_tokens on your API calls. A missing max_tokens parameter on a generation endpoint can result in responses consuming far more tokens than intended. Set a sensible default in the gateway layer so individual callers can't accidentally omit it.

DEFAULT_MAX_TOKENS = {
    "classify": 50,
    "extract": 500,
    "summarize": 800,
    "generate": 2000,
    "code": 4000,
    "default": 1000,
}

def apply_token_budget(self, kwargs: dict, task_type: str) -> dict:
    if "max_tokens" not in kwargs:
        kwargs["max_tokens"] = DEFAULT_MAX_TOKENS.get(task_type, DEFAULT_MAX_TOKENS["default"])
    else:
        # Don't let callers set max_tokens higher than your policy allows
        kwargs["max_tokens"] = min(kwargs["max_tokens"], self.max_allowed_tokens)
    return kwargs

Budget alerts and hard stops

Track cumulative spend in real time and trigger alerts or hard stops at defined thresholds. Alert at 80% of daily budget; hard-stop non-critical endpoints at 100%. Keep a reserve for critical paths (customer-facing endpoints) that can run even when the budget is exhausted for background jobs.

Budget levelAction
50% of daily budgetLog warning, no action
80% of daily budgetAlert on-call via PagerDuty/Slack
90% of daily budgetRoute non-critical tasks to cheapest available model
100% of daily budgetDisable background/batch endpoints; serve degraded response for non-critical interactive endpoints
110% of daily budgetHard stop all non-essential LLM calls; return error with retry-after header

Pattern 5: Observability

The gateway is the right place to add comprehensive logging because every LLM call flows through it. Log at minimum:

  • Request ID (tie to application trace)
  • Model used and why (routing decision log)
  • Input tokens, output tokens, estimated cost
  • Latency (total, time-to-first-token for streaming)
  • Cache hit or miss
  • Error type and count for failures
  • Tenant/user ID (for multi-tenant systems)

This data feeds dashboards that answer the questions your team will ask in production: What's our cost per user? Which endpoints are the most expensive? What's causing latency spikes? Where are we hitting rate limits? Without the gateway logging it centrally, answering these questions requires digging through application logs across multiple services.

Don't log prompt content by default in production. Prompt content may contain PII, customer data, or sensitive business information. Log token counts and hashes, not raw prompts. If you need prompt logging for debugging, implement it as an opt-in trace mode that can be enabled per-request or per-tenant, not as the default behavior.

Build vs. Buy

Several open-source and commercial LLM gateway solutions exist: LiteLLM, Portkey, Helicone, OpenRouter, AWS Bedrock Gateway, and others. They handle the basic patterns described here and some add advanced features like semantic caching, automatic PII redaction, and prompt management.

The build-vs-buy calculus:

  • Use an existing gateway if you primarily need observability, multi-provider support, and basic rate limiting. The setup cost is low and the functionality is solid.
  • Build your own gateway if you have custom routing logic, strict data governance requirements that prevent external proxies, or unusual reliability requirements. Building the core patterns described here is 1-2 engineer-weeks and gives you full control.
  • Use the API's native features first. Anthropic and OpenAI's prompt caching, Anthropic's token counting API, and provider-level rate limit headers handle many needs before a gateway is warranted. Don't build infrastructure for problems you don't have yet.
