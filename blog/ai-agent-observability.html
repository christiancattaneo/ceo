AI Agent Observability: Tracing, Logging, and Monitoring Production Agents

Traditional application observability — request latency, error rate, throughput — tells you almost nothing useful about an AI agent. You need to know which step in a multi-turn conversation failed, how much that session cost, whether the model's output was actually good, and whether a prompt change last week quietly broke quality. That's a different instrumentation problem.

Why AI Observability Is Different

Standard APM tools treat an HTTP request as the unit of work. For an AI agent, a single user-visible action may involve a dozen LLM calls, multiple tool invocations, retrieval operations, and state reads — each with its own latency, cost, and failure mode. Aggregating all of that into a single request duration loses everything that matters.

The other difference is that correctness isn't binary. A web server either returns the right status code or it doesn't. An LLM call can succeed (200 OK, tokens returned) while producing output that is factually wrong, inconsistent with previous turns, or subtly off-brand. Traditional error monitoring doesn't surface these quality regressions at all.

AI observability requires three things standard monitoring doesn't provide:

  • Trace-level visibility into every LLM call and tool invocation within a session, not just the top-level request
  • LLM-specific metrics — token counts, cost, cache hit rate, model version — that have no analogue in traditional APM
  • Quality signals — output evaluation, user feedback, behavioral consistency — that require LLM-as-judge or human annotation to measure

Distributed Tracing for Agent Sessions

A trace for an AI agent session should capture every operation in the call graph, with parent-child relationships that let you reconstruct the full execution path. OpenTelemetry provides the standard data model; the key is emitting the right spans.

from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.http.trace_exporter import OTLPSpanExporter
import time

provider = TracerProvider()
provider.add_span_processor(BatchSpanProcessor(OTLPSpanExporter()))
trace.set_tracer_provider(provider)
tracer = trace.get_tracer("agent.tracer")


def traced_llm_call(
    client,
    model: str,
    messages: list[dict],
    system: str,
    session_id: str,
    step_name: str,
    **kwargs
):
    with tracer.start_as_current_span(f"llm.{step_name}") as span:
        input_tokens_est = sum(len(m["content"].split()) * 1.3
                               for m in messages)
        span.set_attributes({
            "llm.model": model,
            "llm.step": step_name,
            "session.id": session_id,
            "llm.input_tokens_estimated": int(input_tokens_est),
            "llm.message_count": len(messages),
        })

        start = time.time()
        try:
            response = client.messages.create(
                model=model,
                messages=messages,
                system=system,
                **kwargs
            )
            duration_ms = (time.time() - start) * 1000

            span.set_attributes({
                "llm.input_tokens": response.usage.input_tokens,
                "llm.output_tokens": response.usage.output_tokens,
                "llm.duration_ms": duration_ms,
                "llm.stop_reason": response.stop_reason,
                "llm.cache_read_tokens": getattr(
                    response.usage, "cache_read_input_tokens", 0),
                "llm.cache_write_tokens": getattr(
                    response.usage, "cache_creation_input_tokens", 0),
            })
            span.set_status(trace.StatusCode.OK)
            return response

        except Exception as e:
            span.set_status(trace.StatusCode.ERROR, str(e))
            span.record_exception(e)
            raise

Every LLM call in your agent wraps this function. The result is a trace tree where each node is a span with timing, token counts, and model metadata — making it possible to identify which step in a complex agent pipeline is slow or expensive.

Propagate trace context through every async boundary. If your agent uses asyncio.gather for parallel tool calls or fans out to sub-agents, the trace context must be explicitly propagated. OpenTelemetry's context propagation API handles this, but you need to call it — it doesn't happen automatically across thread pools or async task groups.

LLM-Specific Metrics

These metrics have no analogue in traditional services. Instrument them from day one — discovering cost overruns or cache miss rates after the bill arrives is too late.

Metric What it tells you Alert threshold
llm.input_tokens / llm.output_tokens Token consumption per call — drives cost and latency Alert if p95 input tokens spikes >2x baseline (context bloat)
llm.cost_usd Per-call and per-session cost; multiply tokens by model rates Alert if per-session cost exceeds budget tier; daily cost rate anomalies
llm.cache_hit_rate Fraction of input tokens served from prompt cache vs. computed fresh Alert if drops below 60% — suggests context ordering changed
llm.time_to_first_token_ms User-perceived latency for streaming responses Alert if p95 >3s (streaming) or p95 >10s (non-streaming)
llm.tool_call_rate Fraction of responses that include tool calls Alert if drops significantly — may indicate prompt regression
llm.stop_reason Why the model stopped: end_turn, max_tokens, tool_use Alert if max_tokens rate >5% — responses being cut off
llm.error_rate API errors (rate limits, timeouts, 5xx) per model per window Alert if >1% — investigate before it affects users 
from dataclasses import dataclass

MODEL_RATES = {
    "claude-opus-4-6":   {"input": 15.00, "output": 75.00,
                          "cache_write": 18.75, "cache_read": 1.50},
    "claude-sonnet-4-6": {"input":  3.00, "output": 15.00,
                          "cache_write":  3.75, "cache_read": 0.30},
    "claude-haiku-4-5-20251001": {"input": 0.80, "output": 4.00,
                          "cache_write":  1.00, "cache_read": 0.08},
}

@dataclass
class CallMetrics:
    model: str
    input_tokens: int
    output_tokens: int
    cache_read_tokens: int = 0
    cache_write_tokens: int = 0
    duration_ms: float = 0.0

    @property
    def cost_usd(self) -> float:
        rates = MODEL_RATES.get(self.model, MODEL_RATES["claude-sonnet-4-6"])
        return (
            self.input_tokens * rates["input"] / 1_000_000 +
            self.output_tokens * rates["output"] / 1_000_000 +
            self.cache_write_tokens * rates["cache_write"] / 1_000_000 +
            self.cache_read_tokens * rates["cache_read"] / 1_000_000
        )

    @property
    def cache_hit_rate(self) -> float:
        total = self.input_tokens + self.cache_read_tokens
        return self.cache_read_tokens / total if total > 0 else 0.0

Structured Logging Without PII Leaks

Logging conversation content is necessary for debugging but dangerous for privacy. A naive approach that logs full message payloads will inevitably capture PII — names, email addresses, health information, financial data — that violates GDPR, HIPAA, or your contractual obligations.

Log metadata and hashed identifiers, not content:

import hashlib, json, re
from datetime import datetime, timezone

PII_PATTERNS = [
    (r'\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Z|a-z]{2,}\b', '[EMAIL]'),
    (r'\b\d{3}[-.]?\d{3}[-.]?\d{4}\b', '[PHONE]'),
    (r'\b\d{3}-\d{2}-\d{4}\b', '[SSN]'),
]

def redact_pii(text: str) -> str:
    for pattern, replacement in PII_PATTERNS:
        text = re.sub(pattern, replacement, text)
    return text

def log_llm_call(
    session_id: str,
    step: str,
    metrics: CallMetrics,
    user_message_hash: str,     # SHA-256, not the message
    response_preview: str,      # first 100 chars, PII-redacted
    tool_calls: list[str],      # tool names only, not arguments
) -> dict:
    record = {
        "timestamp": datetime.now(timezone.utc).isoformat(),
        "session_id": session_id,
        "step": step,
        "model": metrics.model,
        "input_tokens": metrics.input_tokens,
        "output_tokens": metrics.output_tokens,
        "cache_hit_rate": round(metrics.cache_hit_rate, 3),
        "cost_usd": round(metrics.cost_usd, 6),
        "duration_ms": round(metrics.duration_ms, 1),
        "user_message_hash": user_message_hash,
        "response_preview": redact_pii(response_preview[:100]),
        "tool_calls": tool_calls,
    }
    print(json.dumps(record))
    return record

def hash_content(content: str) -> str:
    return hashlib.sha256(content.encode()).hexdigest()[:16]

Log aggregators may ingest raw lines before redaction runs. Validate that PII redaction happens before any I/O, not after. Use a pre-commit hook or CI check that flags raw PII patterns in logging code. Even a buffered log line in a pipeline can violate data residency rules.

Session-Level Observability

A single LLM call tells you about one step. A session tells you about the user's experience. Aggregate at the session level:

from dataclasses import dataclass, field
import time

@dataclass
class SessionMetrics:
    session_id: str
    user_id: str
    started_at: float = field(default_factory=time.time)
    calls: list = field(default_factory=list)
    tool_invocations: list[str] = field(default_factory=list)
    error_count: int = 0
    turns: int = 0

    def add_call(self, metrics: CallMetrics):
        self.calls.append(metrics)
        self.turns += 1

    @property
    def total_cost_usd(self) -> float:
        return sum(c.cost_usd for c in self.calls)

    @property
    def avg_cache_hit_rate(self) -> float:
        rates = [c.cache_hit_rate for c in self.calls]
        return sum(rates) / len(rates) if rates else 0.0

    def to_log_record(self) -> dict:
        return {
            "event": "session_complete",
            "session_id": self.session_id,
            "turns": self.turns,
            "total_cost_usd": round(self.total_cost_usd, 6),
            "total_tokens": sum(c.input_tokens + c.output_tokens
                                for c in self.calls),
            "avg_cache_hit_rate": round(self.avg_cache_hit_rate, 3),
            "tool_calls": self.tool_invocations,
            "error_count": self.error_count,
            "models_used": list({c.model for c in self.calls}),
        }

Quality Metrics

Operational metrics tell you if the system is working. Quality metrics tell you if it's working well.

Implicit quality signals

  • Session abandonment rate: Users who leave mid-session after the agent responds. Correlate with the preceding response.
  • Retry rate: Users who immediately rephrase a request ("no, what I meant was...") — implicit thumbs-down.
  • Tool call success rate: Tool calls returning errors indicate the model is generating invalid arguments — a prompt quality issue.
  • Turn count to task completion: Tasks that consistently need 10+ turns when 3 should suffice signal a capability or prompt problem.

LLM-as-judge evaluation

For sampled sessions, run a judge model that scores response quality:

def evaluate_response(
    user_message: str,
    agent_response: str,
    context: str,
    client
) -> dict:
    response = client.messages.create(
        model="claude-haiku-4-5-20251001",
        max_tokens=512,
        system="""Score the agent response on these dimensions (1-5 each):
- relevance: Does it address what the user asked?
- accuracy: Is it factually correct given the context?
- conciseness: Appropriately brief without missing key points?
- helpfulness: Would the user find this genuinely useful?

Return JSON: {"relevance":N,"accuracy":N,"conciseness":N,"helpfulness":N,"notes":"..."}""",
        messages=[{
            "role": "user",
            "content": f"Context:\n{context}\n\nUser: {user_message}\n\nAgent: {agent_response}"
        }]
    )
    scores = json.loads(response.content[0].text)
    scores["overall"] = sum(v for k, v in scores.items()
                            if k != "notes") / 4
    return scores

Run the judge on 5–10% of production sessions, plus any with implicit negative signals. Track quality scores over time — a drop after a prompt change is a quality regression even if operational metrics look fine.

The Production Dashboard

Panels that production AI teams actually look at, in priority order:

  1. Daily cost by model — split by model version; cost spikes are usually the first signal that something changed
  2. Error rate by model and step — rate limit errors, timeouts, malformed responses; grouped by pipeline step
  3. p50/p95 latency by step — which step is the bottleneck; model call vs. tool call vs. retrieval
  4. Cache hit rate over time — a drop increases cost and latency; usually caused by prompt restructuring that moved stable content after dynamic content
  5. Quality score trend — 7-day rolling average of LLM-judge scores; alert if drops >0.3 points
  6. Session abandonment rate — users leaving after an agent response without completing the task
  7. Tool call error rate — rising rate indicates prompt-tool schema drift

Instrument before you optimize. The most common mistake is optimizing based on intuition — switching models, rewriting prompts, changing chunk sizes — without baseline metrics to measure against. Set up observability first, establish a baseline, then change one variable at a time. A/B test prompt changes in production using session ID routing, with quality metrics as the deciding criterion.

Tooling Landscape

Several specialized tools handle common LLM observability boilerplate:

  • LangSmith: Deep LangChain/LangGraph integration; automatic tracing; human annotation workflow
  • Arize Phoenix: Open-source; span visualization, embedding drift detection, eval framework
  • Helicone: Proxy-based logging with no code changes; captures all Claude/OpenAI/Gemini calls
  • Braintrust: Eval pipelines with human review workflow and automatic regression detection

Whatever tooling you choose, the instrumentation primitives are the same: spans with token metadata, session-level aggregations, and quality signals from implicit behavior or explicit evaluation. The code above works regardless of which backend you ship traces to.
